## OOPs Fundamentals

---

### Q1. What is OOP? Explain the 4 pillars with real-life examples and code.

**OOP (Object-Oriented Programming)** is a programming paradigm that organizes software design around objects — instances of classes that bundle data (fields) and behavior (methods) together. The four pillars are:

**Encapsulation** — Wrapping data and methods into a single unit, restricting direct access. → Real life: A capsule hides medicine inside.

```java
class BankAccount {
    private double balance;
    public double getBalance() { return balance; }
    public void deposit(double amount) {
        if (amount > 0) balance += amount;
    }
}
```

**Abstraction** — Hiding internal complexity, showing only what's necessary. → Real life: You drive a car using steering and pedals without knowing engine internals.

```java
interface PaymentGateway {
    void processPayment(double amount);
}
class StripePayment implements PaymentGateway {
    public void processPayment(double amount) {
        connectToStripe(); validateMerchant(); chargeCard(amount);
    }
}
```

**Inheritance** — One class acquires properties and behavior of another. Promotes reusability. → Real life: A child inherits traits from parents but can have unique ones too.

```java
class Employee {
    String name; double salary;
    void work() { System.out.println("Working..."); }
}
class Manager extends Employee {
    int teamSize;
    void conductMeeting() { System.out.println("Meeting with " + teamSize + " people"); }
}
```

**Polymorphism** — Same method behaves differently based on context. → Real life: A person acts as employee at work, parent at home, customer at a shop.

```java
class Animal { void sound() { System.out.println("Some sound"); } }
class Dog extends Animal { @Override void sound() { System.out.println("Bark"); } }
class Cat extends Animal { @Override void sound() { System.out.println("Meow"); } }

Animal a1 = new Dog(); a1.sound(); // "Bark"
Animal a2 = new Cat(); a2.sound(); // "Meow"
```

### Q2. Difference between abstraction and encapsulation.

|Aspect|Abstraction|Encapsulation|
|---|---|---|
|Purpose|Hide complexity, show essentials|Hide data, protect internals|
|How|Abstract classes, interfaces|Private fields, getters/setters|

**Interview tip**: Abstraction = design-level (what to expose). Encapsulation = implementation-level (how to protect).

|---|---|---|

|Variables|Any type|Only public static final|
|Constructor|Yes|No|
|Inheritance|Single (extends)|Multiple (implements)|

**Punchline**: "Abstract class for IS-A with shared state. Interface for CAN-DO capability."

### Q4. Concrete Class vs Abstract Class vs Interface — what can each contain?

|Feature|Concrete Class|Abstract Class|Interface|
|---|---|---|---|
|Instantiate (`new`)|✅|❌|❌|
|Methods|Concrete, static, private|Concrete, abstract, static, private|Abstract (implicit), default (Java 8+), static (Java 8+, NOT inherited), private (Java 9+, helper only)|
|Variables|Instance + static (any access)|Instance + static (any access)|Only public static final (constants). No instance variables|
|Constructor|✅|✅ (via `super()`)|❌|
|Blocks|Instance `{ }` + static `static { }`|Instance `{ }` + static `static { }`|❌|
|Inheritance|Single (`extends`)|Single (`extends`)|Multiple (`implements`)|

**Key**: `default` keyword in interface = "method has a body", NOT "package-private access". All interface members are **public**.

```java
interface MyInterface {
    int MAX = 100;                    // implicitly public static final
    void doWork();                    // implicitly public abstract
    default void log() { helper(); }  // public, has body (Java 8+)
    static void info() { }           // public, NOT inherited (Java 8+)
    private void helper() { }        // helper for default methods (Java 9+)
}
```

**Punchline**: "Interface = contract (no state, no blocks, no constructors). Abstract class = partial implementation. Concrete class = complete, ready to instantiate."

### Q5. Can private methods be overridden? Why not?

**No.** Private methods are not visible to child classes. The child class doesn't know they exist.

---

### Q6. Can static methods be overridden? Why not? (Method hiding)

**No.** Static methods are resolved at **compile time based on reference type, not object type**. If a child defines the same static method, it's **method hiding**, not overriding.

```java
Parent p = new Child();
p.greet(); // prints "Parent" — resolved by reference type
```

---

### Q7. Can we overload private and static methods?

**Yes to both.** Overloading is resolved at compile time based on method signature. Doesn't depend on inheritance.

---

### Q8. Difference between method overloading and method overriding.

|Aspect|Overloading|Overriding|
|---|---|---|
|Where|Same class|Parent-child classes|
|Parameters|Must differ|Must be same|
|Return type|Can differ|Must be same or covariant|
|Binding|Compile-time (static)|Runtime (dynamic)|
|Access modifier|No restriction|Cannot be more restrictive|

### Q9. What is the diamond problem? How does Java solve it?

Two interfaces with the same default method — ambiguity for the implementing class. Java forces you to **override and explicitly choose**.

```java
interface A { default void show() { System.out.println("A"); } }
interface B { default void show() { System.out.println("B"); } }

class C implements A, B {
    @Override
    public void show() {
        A.super.show();  // calls A's version
        B.super.show();  // calls B's version
    }
}
```

**How to call default methods?**

- `A.super.show()` → calls A's default method (still runs on `this`, just picks which version)
- `A.show()` → ❌ won't compile — this syntax is only for **static** methods

**Quick rule**: `InterfaceName.method()` = static. `InterfaceName.super.method()` = default.

### Q11. What are SOLID principles?

**S** — Single Responsibility: One class, one reason to change. **O** — Open/Closed: Open for extension, closed for modification. **L** — Liskov Substitution: Subclass should be replaceable for parent without breaking behavior. **I** — Interface Segregation: Don't force classes to implement methods they don't use. **D** — Dependency Inversion: Depend on abstractions, not concrete implementations.

## equals(), hashCode() & Immutability

---

### Q12. What is the contract between equals() and hashCode()?

1. If `a.equals(b)` is true → `a.hashCode() == b.hashCode()` **must** be true.
2. Same hashCode does NOT mean equals is true (collision).

**Follow-up**: Same hashCode but equals false? → **Hash collision**. Both land in same bucket, equals() differentiates them.

---

### Q13. What happens if you override only equals() but not hashCode()?

HashMap breaks. Logically equal objects get different hashCodes → placed in different buckets → `map.get()` returns null.

**Punchline**: "hashCode decides bucket, equals decides uniqueness."

---

### Q14. Why are equals() and hashCode() important in HashMap?

1. **hashCode()** → which bucket
2. **equals()** → which key within that bucket

---

### Q15. How does HashMap use equals() internally during get/put?

**put**: hashCode → bucket index → if empty, insert. If occupied, iterate and check equals(). Match → overwrite. No match → append. **get**: hashCode → bucket → iterate with equals() → return value or null.

---

### Q16. How do you make a custom class immutable?

1. Class is `final`. 2. Fields are `private final`. 3. No setters. 4. Constructor initializes all. 5. Defensive copies for mutable fields.

```java
public final class Employee {
    private final String name;
    private final List<String> skills;
    public Employee(String name, List<String> skills) {
        this.name = name;
        this.skills = new ArrayList<>(skills); // defensive copy in
    }
    public List<String> getSkills() { return new ArrayList<>(skills); } // defensive copy out
}
```

---

### Q17. Why are wrapper classes (Integer, String) immutable?

Thread safety, caching (String Pool, Integer Cache), security, HashMap key safety.

### Q32. What is try-with-resources?

Auto-closes resources implementing AutoCloseable. Cleaner than manual finally.

### Q33. How can you exit without executing the finally block?

```java

    System.out.println("Never prints");
}
```

Other ways: JVM crash, infinite loop in try, Thread.stop() (deprecated).

## Collections & HashMap Deep Internals

---

### Q36. HashMap internal working — put() and get().

Array of Nodes (buckets). Default capacity 16, load factor 0.75.

**put**: hashCode → spread (hash ^ hash>>>16) → bucket index (hash & capacity-1) → empty? insert. Collision? linked list → treeify at 8 nodes (if capacity ≥ 64). Size > threshold → rehash.

**Null key**: hash=0, always bucket 0. **NOT thread-safe.** Average O(1), worst O(log n) after treeification.

**Real usage**: "Used HashMap for caching config values and DTO mapping."

---

### Q37. What is load factor and threshold?

Load factor = 0.75 (balance between space and time). Threshold = capacity × load factor. When exceeded → resize.

---

### Q38. What happens during rehashing?

New array with double capacity. Recalculate index for every entry. O(n) operation. Set initial capacity to avoid repeated rehashing.

---

### Q39. What is a bucket? Collision? Resolution?

Bucket = each array index. Collision = two keys same bucket index. Resolution = separate chaining (linked list → Red-Black Tree in Java 8).

---

### Q40. What is treeification in Java 8?

List > 8 entries AND capacity ≥ 64 → converts to Red-Black Tree (O(log n)). If capacity < 64 → resizes instead. Untreeifies below 6.

---

### Q41. Why is HashMap capacity always a power of 2?

`hash & (capacity-1)` distributes evenly when capacity is power of 2 (all 1s in binary).

---

### Q42. Why hash spreading — hash ^ (hash >>> 16)?

Mixes higher bits into lower bits. HashMap only uses lower bits for indexing, so spreading reduces collisions from poor hashCode implementations.

---

### Q43. HashMap vs ConcurrentHashMap.

|HashMap|ConcurrentHashMap|
|---|---|
|Not thread-safe|Thread-safe (CAS + bucket-level locking)|
|1 null key|No nulls|
|Fail-fast|Weakly consistent|
|Fast (single thread)|Reads almost as fast, writes slightly slower|

---

### Q44. How does ConcurrentHashMap achieve thread safety?

Java 7: 16 segments, each independently lockable. Java 8+: CAS for empty buckets, synchronized on bucket head node for collisions. Reads are lock-free.

---

### Q45. ArrayList vs LinkedList.

||ArrayList|LinkedList|
|---|---|---|
|get(index)|O(1)|O(n)|
|add(end)|O(1) amortized|O(1)|
|add(middle)|O(n) shift|O(1) after finding|
|Memory|Less|More (prev+next pointers)|

"ArrayList wins in practice due to CPU cache friendliness."

---

### Q46. HashSet internal working.

Backed by HashMap. Element = key, value = dummy constant `PRESENT`. Duplicates rejected because HashMap keys are unique.

### Q55. How do streams work internally? Lazy evaluation and pipeline fusion.

Intermediate ops build a pipeline. Terminal op triggers execution. Each element flows through the entire pipeline before the next (**pipeline fusion / vertical processing**). Short-circuit ops like `findFirst()` stop early.

**Punchline**: "Streams are lazy, single-use pipelines that fuse operations vertically."

### Q56. Why are streams not reusable?

One-pass pipeline. After terminal op, stream is closed. Re-create from source.

**map** → one-to-one. **flatMap** → one-to-many + flatten.

```java
    .flatMap(Collection::stream)   // Stream<Order> — flattened
```

### Q59. Second highest number.

### Q61. What is Optional?

Container that may or may not have a value. Avoids NullPointerException.

```java

```

**Don't use for**: fields, method parameters. Only as return type.

---

### Q62. orElse() vs orElseGet().

`orElse()` → ALWAYS evaluates default (even if value exists). `orElseGet()` → evaluates ONLY when empty (lazy, preferred for expensive operations).

---

### Q63. forEach() vs stream().forEach().

`list.forEach()` → direct, simpler. `stream().forEach()` → only when chaining stream ops. Parallel streams: use `forEachOrdered()` for order.

---

### Q64. Parallel streams — when to use, when NOT.

Use: CPU-intensive, large datasets, order doesn't matter. NOT: small data, I/O ops, shared mutable state, order matters.

Uses `ForkJoinPool.commonPool()` — shared across app. Can starve other operations.

## Multithreading & Concurrency

---

### Q71. Thread vs Runnable.

|Thread|Runnable|
|---|---|
|extends (can't extend other class)|implements (can extend another)|
|Heavier|Lightweight, just a task|

**Always prefer Runnable.** Separates task from thread mechanism.

```java
Runnable task = () -> System.out.println("Running");
new Thread(task).start();
```

---

### Q72. start() vs run().

`start()` → creates **new thread**, calls run() on it. `run()` → normal method call on **current thread**, no new thread.

```java
Thread t = new Thread(() -> System.out.println(Thread.currentThread().getName()));
t.start(); // "Thread-0" (new thread)
t.run();   // "main" (same thread)
```

---

### Q73. Can you call start() twice?

**No.** `IllegalThreadStateException`. Once TERMINATED, can't restart. Use new Thread with same Runnable.

```java
Runnable task = () -> System.out.println("Running");
new Thread(task).start();  // first time
new Thread(task).start();  // again — new Thread object
```

---

### Q74. What does thread.join() do?

Calling thread waits until target thread completes. `t.join()` → main waits for t to finish.

---

### Q75. Synchronization — method vs block.

**Method** → locks entire method on `this`. **Block** → locks specific section on specific object. Prefer block — minimizes lock scope.

---

### Q76. Deadlock.

Two threads waiting for each other's locks forever. Prevention: lock ordering, tryLock with timeout, minimize lock scope, avoid nested locks.

---

### Q77. Thread lifecycle.

NEW → start() → RUNNABLE → RUNNING → sleep/wait → WAITING → notify/timeout → RUNNABLE → run() finishes → TERMINATED

---

### Q78. wait() vs sleep().

|wait()|sleep()|
|---|---|
|Releases lock|Keeps lock|
|Object class|Thread class|
|Inside synchronized only|Anywhere|
|notify() wakes|Timeout only|

**Punchline**: "wait() releases lock and waits for signal. sleep() holds lock and pauses."

---

### Q79. Volatile keyword.

Ensures reads/writes go to main memory, not CPU cache. **Visibility yes, atomicity no.**

```java
volatile int count = 0;
count++; // NOT safe — 3 operations (read, modify, write)
```

**Punchline**: "volatile gives visibility, not atomicity."

---

### Q80. Atomic classes.

`AtomicInteger`, `AtomicLong`, `AtomicBoolean` — thread-safe without synchronized. Uses CAS internally.

```java
AtomicInteger count = new AtomicInteger(0);
count.incrementAndGet();       // atomic
count.compareAndSet(5, 10);    // if 5, set to 10
```

**When to use what:** volatile → boolean flags. AtomicInteger → counters. synchronized → complex operations.

---

### Q81. ExecutorService.

Manages a thread pool. Reuses threads instead of creating new ones.

```java
ExecutorService executor = Executors.newFixedThreadPool(5);
executor.submit(() -> System.out.println("Task"));
executor.shutdown();
```

**Why not new Thread()?** Expensive, uncontrolled, no queuing, no reuse.

---

### Q82. Thread pool types.

|Type|What it does|
|---|---|
|`newFixedThreadPool(5)`|Fixed 5 threads, extras wait in queue|
|`newCachedThreadPool()`|Creates as needed, reuses idle ones|
|`newSingleThreadExecutor()`|1 thread, tasks run sequentially|
|`newScheduledThreadPool(3)`|Delayed/repeated tasks|

**Real usage**: "Fixed thread pool for sending notification emails — 5 threads handled thousands from a queue."

### Q99. Thread start/join ordering

Calling `start()` on a thread does NOT guarantee immediate execution — it just marks the thread runnable. `join()` makes the caller wait for completion. Without `join()`, main thread may finish before children.

### Q100. Integer caching (-128 to 127)

*(See Night Before Interview, Section 5.8 — `Integer a = 127; Integer b = 127;` → `a == b` is `true` due to caching; at 128 it's `false`.)*

### Q101. Method overloading with null argument

Passing `null` to overloaded methods causes ambiguity if multiple versions accept reference types. Compiler picks the **most specific** type; if two candidates are equally specific, compile error.

### Q102. Static block vs instance block execution order

Static block runs **once per class load** (before any object). Instance block runs **every object creation** (before constructor body). Order: static → instance block → constructor.

### Q103. Autoboxing and unboxing edge cases

Unboxing `null` Integer → NullPointerException. Comparing `Integer == Integer` uses reference equality (not content) — use `.equals()` or `.intValue()`. Autoboxing in collections can cause unexpected memory overhead.

## Rapid-Fire Follow-ups

**Streams**: `reduce()` = single value. `collect()` = mutable container. Exception mid-stream → terminates, propagates. **Concurrency**: Race condition = threads access shared data, result depends on timing. CAS = "if value is X, set to Y, else retry." ReentrantLock = tryLock with timeout, more flexible than synchronized. **Singleton**: Eager = instance at class load. Lazy = on first access. Double-checked locking needs volatile to prevent partial construction.
