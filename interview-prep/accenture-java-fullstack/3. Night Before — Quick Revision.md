## 1. JAVA CORE — TOP HITS

### OOP — 4 Pillars (1 line each)

- **Encapsulation** → Private fields + public getters/setters. Hide data, control access.
- **Abstraction** → Hide complexity, show only what's needed. Interfaces/abstract classes.
- **Inheritance** → Child class reuses parent's code. `extends` keyword.
- **Polymorphism** → Same method, different behavior. Overloading (compile-time) + Overriding (runtime).

### Abstract Class vs Interface

|Aspect|Abstract Class|Interface|
|---|---|---|
|Methods|Abstract + concrete|Abstract, default (8+), static (8+), private (9+)|
|Variables|Any|Only public static final|
|Constructor|Yes|No|
|Inheritance|Single|Multiple|

**Punchline**: "Abstract class for IS-A with shared state. Interface for CAN-DO capability."

### String Pool + Immutability

```java
String a = "hello";           // pool
String b = "hello";           // same pool reference
String c = new String("hello"); // heap

a == b       // true (same pool ref)
a == c       // false (different objects)
a.equals(c)  // true (same content)
c.intern() == a  // true (intern returns pool ref)
```

**Why immutable?** → Pool safety, thread safety, security, hashCode caching.

### String vs StringBuilder vs StringBuffer

||String|StringBuilder|StringBuffer|
|---|---|---|---|
|Mutable?|No (creates new object)|Yes|Yes|
|Thread-safe?|Yes (immutable)|No|Yes (synchronized)|
|Speed|Slow for concatenation|Fastest|Slower than StringBuilder|

```java
// String — every + creates a new object
String s = "hello";
s = s + " world";  // old "hello" is wasted, new object created

// StringBuilder — modifies same object, fast
StringBuilder sb = new StringBuilder("hello");
sb.append(" world");  // same object modified

// StringBuffer — same as StringBuilder but thread-safe (rarely needed)
```

**When to use:** StringBuilder 99% of the time. StringBuffer only if multiple threads are building the same string (almost never happens).

### equals() & hashCode()

**Contract**: Equal objects MUST have equal hashCodes. Equal hashCodes DON'T mean equal objects. **Break only equals?** → HashMap breaks. Same logical key lands in different buckets. **Punchline**: "hashCode decides bucket, equals decides uniqueness."

### HashMap Internals (Short Version)

- Array of Nodes (buckets). Default capacity 16, load factor 0.75.
- `put()`: hashCode → bucket index → if empty insert, if collision → linked list → treeify at 8 nodes.
- Average O(1), worst O(log n) after Java 8 treeification.
- NOT thread-safe. Use ConcurrentHashMap for multi-threaded.
- Allows 1 null key (goes to bucket 0).

### HashMap vs ConcurrentHashMap

|HashMap|ConcurrentHashMap|
|---|---|
|Not thread-safe|Thread-safe (CAS + bucket-level locking)|
|1 null key allowed|No nulls|
|Fail-fast iterator|Weakly consistent|

### Immutable Class — 5 Steps

1. Class is `final`
2. Fields are `private final`
3. No setters
4. Initialize via constructor
5. Defensive copies for mutable fields (List, Date)

### Exception Handling

- **Checked** (compile-time): IOException, SQLException → must handle
- **Unchecked** (runtime): NullPointerException → programming errors
- **throw** = actually throw. **throws** = declare in signature
- **Custom exception**: `class MyException extends Exception { super(msg); }`

### final vs finally vs finalize

- **final** → can't change (variable/method/class)
- **finally** → always executes after try/catch
- **finalize** → deprecated GC method, don't use

### Stack vs Heap

- **Stack** → local variables, references, method calls. Per-thread, fast.
- **Heap** → objects, instance variables. Shared, GC-managed.
- **Punchline**: "References on stack, objects on heap."

## 2. JAVA 8 — TOP HITS

### Functional Interfaces ⚠️ MISSED CONSUMER IN INTERVIEW

|Interface|Method|What it does|Real usage|
|---|---|---|---|
|Predicate|test()|Takes input, returns true/false|`filter(n -> n > 5)` — filtering in streams|
|Function|apply()|Takes input, returns something different|`map(s -> s.length())` — transforming in streams|
|Consumer|accept()|Takes input, returns nothing|`forEach(s -> System.out.println(s))` — processing each item|
|Supplier|get()|Takes nothing, returns something|`orElseGet(() -> new User())` — lazy object creation|

```java
// Predicate — used in filter()
list.stream().filter(user -> user.getAge() > 18).collect(Collectors.toList());

// Function — used in map()
list.stream().map(user -> user.getName()).collect(Collectors.toList());

// Consumer — used in forEach(), ifPresent()
list.forEach(user -> sendEmail(user));
optional.ifPresent(user -> save(user));

// Supplier — used in orElseGet(), lazy creation
optional.orElseGet(() -> fetchFromDB());
```

### Streams — Key Concepts

- **Lazy** → intermediate ops build pipeline, terminal op triggers execution
- **Not reusable** → one terminal op = stream closed
- **Intermediate**: filter, map, sorted, distinct, flatMap
- **Terminal**: collect, forEach, count, reduce, findFirst

```java
// Filter even numbers
list.stream().filter(n -> n % 2 == 0).collect(Collectors.toList());

// Second highest
list.stream().distinct().sorted(Comparator.reverseOrder()).skip(1).findFirst();

// Group by department
employees.stream().collect(Collectors.groupingBy(Employee::getDepartment));
```

### map() vs flatMap()

- `map()` → one-to-one: each element → one result
- `flatMap()` → one-to-many + flatten: nested lists → single list

### Optional

A box that may or may not have a value inside. Instead of returning null and risking NullPointerException, you return Optional.

```java
// Without Optional — risky
User user = repo.findById(1);  // might return null
user.getName();                 // NullPointerException if null!

// With Optional — safe
Optional<User> user = repo.findById(1);
user.map(User::getName).orElse("Unknown");  // no NPE ever
```

**Key methods:**

```java
Optional.of(value)          // wraps value (throws if null)
Optional.ofNullable(value)  // wraps value (empty if null)
optional.isPresent()        // true if value exists
optional.ifPresent(u -> save(u))  // do something only if present
optional.orElse("default")       // return value or default
optional.orElseGet(() -> fetchDB()) // lazy default (preferred)
optional.orElseThrow()            // throw exception if empty
optional.map(User::getName)       // transform if present
```

**orElse vs orElseGet:**

- `orElse(fetchDB())` → fetchDB() runs ALWAYS, even if value exists — wasteful
- `orElseGet(() -> fetchDB())` → runs ONLY when empty — use this

**Don't use Optional for:** fields, method parameters, collections. Only use as return type.

### Parallel Streams — When NOT to use

- Small datasets, I/O operations, shared mutable state, order-dependent operations

## 3. MULTITHREADING — QUICK HITS

### Thread vs Runnable

- **Thread** = extends class (can't extend another). **Runnable** = implements interface (preferred).

### wait() vs sleep()

|wait()|sleep()|
|---|---|
|Releases lock|Keeps lock|
|Object class|Thread class|
|Inside synchronized only|Anywhere|

### volatile vs synchronized ⚠️ MISSED IN INTERVIEW

**Problem volatile solves**: Each thread has its own CPU cache. If Thread A changes a variable, Thread B might still see the old value from its cache.

`volatile` says: "Don't cache this variable. Always read/write from main memory."

```java
// Without volatile — Thread B might never see the change
boolean running = true;

// Thread A
while (running) { /* loops forever maybe */ }

// Thread B
running = false;  // Thread A might not see this!

// With volatile — Thread B's change is immediately visible
volatile boolean running = true;  // now Thread A sees the change instantly
```

**But volatile does NOT make operations atomic:**

```java
volatile int count = 0;
count++;  // NOT safe — this is read + modify + write (3 steps)
          // Two threads can read same value before either writes

// Use AtomicInteger for thread-safe counting:
AtomicInteger count = new AtomicInteger(0);
count.incrementAndGet();  // truly atomic, one step
```

||volatile|synchronized|
|---|---|---|
|Visibility|✅|✅|
|Atomicity|❌|✅|
|Blocking|No|Yes|
|Use for|boolean flags|count++, complex operations|

**Punchline**: "volatile gives visibility, not atomicity."

### Deadlock Prevention

- Lock ordering (always same order)
- Timeout with tryLock()
- Minimize lock scope

### Thread Pool & ExecutorService ⚠️ MISSED IN INTERVIEW

**Problem**: Creating a new thread every time is expensive — each thread uses OS resources. If you create 10,000 threads, your app crashes.

**Solution**: Thread pool — a fixed set of reusable threads. Tasks wait in a queue. When a thread finishes one task, it picks up the next.

```java
// Create pool of 5 threads
ExecutorService executor = Executors.newFixedThreadPool(5);

// Submit 100 tasks — only 5 run at a time, rest wait in queue
for (int i = 0; i < 100; i++) {
    executor.submit(() -> {
        System.out.println(Thread.currentThread().getName() + " running");
    });
}

executor.shutdown();  // no new tasks, finish existing ones
```

**Types of thread pools:**

|Type|What it does|
|---|---|
|`newFixedThreadPool(5)`|Fixed 5 threads. Extra tasks wait in queue|
|`newCachedThreadPool()`|Creates threads as needed, reuses idle ones. Good for short tasks|
|`newSingleThreadExecutor()`|Just 1 thread. Tasks run one by one in order|
|`newScheduledThreadPool(3)`|For delayed/repeated tasks (like run every 5 seconds)|

**Real usage**: "We used a fixed thread pool for sending notification emails — instead of creating a new thread per email, 5 threads handled thousands of emails from a queue."

**Why not just `new Thread()` every time?**

- Thread creation is expensive (OS resource)
- No control over how many threads run
- No task queuing
- Thread pool reuses threads = much faster

### Atomic Classes ⚠️ MISSED IN INTERVIEW

- `AtomicInteger`, `AtomicLong`, `AtomicBoolean` — thread-safe without synchronized.
- Uses CAS (Compare-And-Swap) internally — lock-free, better performance.

```java
AtomicInteger count = new AtomicInteger(0);
count.incrementAndGet();       // thread-safe increment
count.get();                   // read current value
count.compareAndSet(5, 10);    // if value is 5, set to 10
```

**When to use what:**

- Simple boolean flag across threads → `volatile`
- Counter / read-modify-write → `AtomicInteger`
- Complex multi-step operations → `synchronized`

### Can you start() the same thread twice?

**No.** Throws `IllegalThreadStateException`. Once a thread is TERMINATED, it's dead — can't restart.

```java
Thread t = new Thread(() -> System.out.println("Running"));
t.start();  // ✅ works
t.start();  // ❌ IllegalThreadStateException

// Need to run same task again? New thread, same Runnable:
Runnable task = () -> System.out.println("Running");
new Thread(task).start();  // first time
new Thread(task).start();  // again — new Thread object, same task
```

## 5. CODING PROBLEMS

### Palindrome String

```java
String s = "madam";
String rev = new StringBuilder(s).reverse().toString();
System.out.println(s.equals(rev)); // true
```

### Second Largest Element in Array

```java
int[] arr = {5, 3, 9, 1, 7};
int result = Arrays.stream(arr)
    .distinct().boxed()
    .sorted(Comparator.reverseOrder())
    .skip(1).findFirst().orElse(-1); // 7 (returns -1 if less than 2 distinct elements)
```

### Reverse String Without Built-in Methods

```java
String s = "hello";
char[] c = s.toCharArray();
int left = 0, right = c.length - 1;
while (left < right) {
    char temp = c[left];
    c[left] = c[right];
    c[right] = temp;
    left++; right--;
}
System.out.println(new String(c)); // "olleh"
```

### Find Duplicates in Array

```java
int[] arr = {1, 2, 3, 2, 4, 1, 5};
Set<Integer> seen = new HashSet<>();
List<Integer> dups = Arrays.stream(arr).boxed()
    .filter(n -> !seen.add(n))
    .collect(Collectors.toList()); // [2, 1]
```

### Check Anagram

```java
String a = "listen", b = "silent";
char[] c1 = a.toLowerCase().toCharArray();
char[] c2 = b.toLowerCase().toCharArray();
Arrays.sort(c1);
Arrays.sort(c2);
System.out.println(Arrays.equals(c1, c2)); // true
```

### String Output Prediction

```java
String a = "hello";
String b = new String("hello");
System.out.println(a == b);        // false
System.out.println(a.equals(b));   // true
String c = b.intern();
System.out.println(a == c);        // true
```

### try-finally Output — ⚠️ MISSED FOLLOW-UP: how to skip finally

```java
public static int test() {
    try { return 1; }
    finally { System.out.println("Finally"); }
}
// Output: "Finally", returns 1
// finally ALWAYS executes before return
```

**How to skip finally block?**

- `System.exit(0)` — kills the JVM immediately, finally won't run
- JVM crash or infinite loop in try block
- Thread killed by `Thread.stop()` (deprecated)

```java
try {
    System.out.println("Try");
    System.exit(0);  // JVM shuts down here
} finally {
    System.out.println("Finally");  // NEVER prints
}
```

### Integer Caching

```java
Integer a = 127, b = 127;
System.out.println(a == b);  // true (cached -128 to 127)

Integer c = 128, d = 128;
System.out.println(c == d);  // false (outside cache range)
```

### Taking Runtime Input (Scanner) ⚠️ MISSED IN INTERVIEW

```java
import java.util.Scanner;

Scanner sc = new Scanner(System.in);

int n = sc.nextInt();           // read integer
String s = sc.next();           // read single word
String line = sc.nextLine();    // read full line
double d = sc.nextDouble();     // read double

sc.close();
```

**Common trap**: After `nextInt()`, calling `nextLine()` reads leftover newline. Fix: add an extra `sc.nextLine()` after `nextInt()`.

```java
int num = sc.nextInt();
sc.nextLine();               // consume leftover newline
String name = sc.nextLine(); // now reads correctly
```

## 6. SPRING BOOT

### @SpringBootApplication

= @Configuration + @EnableAutoConfiguration + @ComponentScan. Remove it → app won't auto-configure or scan beans.

### Stereotype Annotations

|Annotation|Layer|Special Behavior|
|---|---|---|
|@Component|Generic|Bean registration|

### @PathVariable vs @RequestParam

```java
// @PathVariable → /users/123
@GetMapping("/users/{id}")
public User getUser(@PathVariable Long id) { }

// @RequestParam → /users?name=John
@GetMapping("/users")
public List<User> search(@RequestParam String name) { }
```

### Request Flow

HTTP Request → **DispatcherServlet** → HandlerMapping → **Controller** → **Service** → **Repository** → DB → Response

### Dependency Injection

Spring creates and injects objects. You don't do `new`. **Constructor injection preferred** (immutable, testable, no partial construction).

```java

        this.repo = repo;
```

Instantiate → Set properties → @PostConstruct → Bean ready → @PreDestroy → Destroy

### Bean Scopes

- **Singleton** (default) → one instance per container
- **Prototype** → new instance every time requested
- **Request/Session** → web-scoped

### Exception Handling

```java
@RestControllerAdvice
public class GlobalExceptionHandler {
    @ExceptionHandler(UserNotFoundException.class)
    public ResponseEntity<String> handleNotFound(UserNotFoundException ex) {
        return ResponseEntity.status(404).body(ex.getMessage());
    }
}
```

### @Transactional

- Proxy-based AOP. Spring wraps method in begin/commit/rollback.
- **Propagation**: REQUIRED (default, join existing or create new), REQUIRES_NEW (always new).
- Self-invocation within same class won't trigger proxy → @Transactional ignored.

### CORS


### Spring Profiles

### Spring Boot Actuator ⚠️ MISSED IN INTERVIEW

Gives you ready-made endpoints to monitor and manage your app in production. Just add `spring-boot-starter-actuator` dependency.

**Key endpoints:**

|Endpoint|What it does|
|---|---|
|/actuator/health|Is the app running? UP or DOWN|
|/actuator/info|App info (version, description)|
|/actuator/metrics|Memory usage, CPU, request counts|
|/actuator/beans|All beans loaded in the app|
|/actuator/env|Environment properties|

**By default** only /health and /info are exposed. To expose all:


**Security**: In production, these endpoints should be secured — don't expose /env or /beans publicly. Use Spring Security or restrict to internal network.

**Real usage**: "We used /health for load balancer health checks — if the app is DOWN, the load balancer stops sending traffic to that instance. We used /metrics to track API response times and error rates."

### JWT Authentication Flow

1. User sends credentials → Server validates → Returns JWT token
2. Client sends JWT in `Authorization: Bearer <token>` header
3. Server validates token on each request → grants/denies access

**Authentication** = who are you. **Authorization** = what can you do.

## 7. JPA / HIBERNATE

### JPA vs Hibernate

- **JPA** = specification (rules/interfaces). **Hibernate** = implementation.

### Entity States

- **Transient** → new object, not in DB
- **Persistent** → attached to session, synced with DB
- **Detached** → was persistent, session closed
- **Removed** → marked for deletion

### Lazy vs Eager

- **Lazy** (default for collections @OneToMany) → loads when accessed. Can cause N+1 if you access collections in a loop.
- **Eager** (default for single entity @ManyToOne) → loads everything immediately. Wasteful if you don't need the related data.
- **Prefer Lazy** + use JOIN FETCH when you actually need related data.

### N+1 Problem

1 query for parent + N queries for each child (triggered when Lazy-loaded collections are accessed in a loop). Fix: `JOIN FETCH`, `@EntityGraph`, `@BatchSize`.

```java
// BAD: N+1
List<Department> depts = repo.findAll(); // 1 query
depts.forEach(d -> d.getEmployees()); // N queries

// FIX: JOIN FETCH
@Query("SELECT d FROM Department d JOIN FETCH d.employees")
List<Department> findAllWithEmployees();
```

### Cascade Types

- **PERSIST** → save parent → saves children
- **MERGE** → update parent → updates children
- **REMOVE** → delete parent → deletes children
- **ALL** → all of the above

### save() vs saveAndFlush()

- `save()` → queues the operation (flushed at transaction end)
- `saveAndFlush()` → immediately writes to DB

### Unidirectional vs Bidirectional Mapping ⚠️ MISSED IN INTERVIEW

**Unidirectional** — only one side knows about the relationship. **Bidirectional** — both sides know about each other.

```java
// Unidirectional — Employee knows Department, but Department doesn't know its Employees
@Entity
class Employee {
    @ManyToOne
    private Department department;
}

@Entity
class Department {
    // no reference to Employee
}

// Bidirectional — both sides know each other
@Entity
class Employee {
    @ManyToOne
    private Department department;
}

@Entity
class Department {
    @OneToMany(mappedBy = "department")  // mappedBy = "I'm not the owner, Employee is"
    private List<Employee> employees;
}
```

**When to use:**

- **Unidirectional** → simpler, when you only need one direction (get employee's department)
- **Bidirectional** → when you need both directions (also get all employees in a department)

**Key point**: In bidirectional, `mappedBy` goes on the side that does NOT own the foreign key. The `@ManyToOne` side (Employee) owns the relationship because the foreign key (`dept_id`) lives in the employee table.

## 8. MICROSERVICES

### Monolith vs Microservices

- **Monolith** → single deployable unit. Simple, good for small teams.
- **Microservices** → independent services. Scalable, complex, good for large teams.

### Communication

- **Synchronous**: REST (RestTemplate/WebClient/Feign) — simple, blocking
- **Asynchronous**: Kafka/RabbitMQ — decoupled, non-blocking, better resilience

### Key Components

- **API Gateway** → single entry point, routing, auth, rate limiting
- **Service Discovery (Eureka)** → services register/discover each other dynamically
- **Circuit Breaker (Resilience4j)** → prevents cascading failures. States: Closed → Open → Half-Open
- **Config Server** → centralized configuration for all services

### Idempotent API

Same request multiple times → same result. Important because network retries happen. GET/PUT/DELETE are idempotent. POST is not.

## 9. SQL

### Second Highest Salary

```sql
-- Method 1
SELECT MAX(salary) FROM employee WHERE salary < (SELECT MAX(salary) FROM employee);

-- Method 2
SELECT salary FROM employee ORDER BY salary DESC LIMIT 1 OFFSET 1;
```

### JOINs

```sql
-- INNER JOIN: only matching rows from both tables
SELECT e.name, d.dept_name FROM employee e INNER JOIN department d ON e.dept_id = d.id;

-- LEFT JOIN: all from left + matching from right (NULL if no match)
SELECT e.name, d.dept_name FROM employee e LEFT JOIN department d ON e.dept_id = d.id;
```

### WHERE vs HAVING

```sql
-- WHERE: filters rows BEFORE grouping
SELECT * FROM employee WHERE salary > 50000;

-- HAVING: filters AFTER GROUP BY
SELECT dept, COUNT(*) FROM employee GROUP BY dept HAVING COUNT(*) > 5;
```

### GROUP BY vs ORDER BY

- **GROUP BY** → groups rows for aggregation (COUNT, SUM, AVG)
- **ORDER BY** → sorts result set (ASC/DESC)

### Index

- B-Tree data structure that speeds up SELECT but slows INSERT/UPDATE/DELETE.
- Use on: frequently queried columns, WHERE/JOIN/ORDER BY columns.
- Don't use on: small tables, columns with few unique values, frequently updated columns.

### ACID Properties

- **Atomicity** → all or nothing
- **Consistency** → valid state before and after
- **Isolation** → concurrent transactions don't interfere
- **Durability** → committed data survives crash

### Stored Procedure ⚠️ MISSED IN INTERVIEW

A pre-compiled block of SQL stored in the database. You write it once, call it by name. Runs on the DB server, not your app.

**Why use?** Faster (pre-compiled), reusable, less network calls, keeps logic in DB.

```sql
-- Create
DELIMITER //
CREATE PROCEDURE GetEmployeesByDept(IN deptName VARCHAR(50))
BEGIN
    SELECT * FROM employee WHERE department = deptName;
END //
DELIMITER ;

-- Call
CALL GetEmployeesByDept('IT');
```

**IN / OUT / INOUT:**

```sql
CREATE PROCEDURE GetSalaryStats(
    IN deptName VARCHAR(50),    -- input parameter
    OUT avgSalary DECIMAL,      -- output parameter
    OUT empCount INT             -- output parameter
)
BEGIN
    SELECT AVG(salary), COUNT(*) INTO avgSalary, empCount
    FROM employee WHERE department = deptName;
END;

-- Call
CALL GetSalaryStats('IT', @avg, @count);
SELECT @avg, @count;
```

**Call from Spring Boot:**

```java
@Repository
public interface EmployeeRepository extends JpaRepository<Employee, Long> {

    @Query(value = "CALL GetEmployeesByDept(:dept)", nativeQuery = true)
    List<Employee> getByDept(@Param("dept") String dept);
}
```

### DELETE vs TRUNCATE vs DROP

||DELETE|TRUNCATE|DROP|
|---|---|---|---|
|What|Removes rows|Removes all rows|Removes entire table|
|WHERE|Yes|No|No|
|Rollback|Yes|No (DDL, MySQL) / Yes (PostgreSQL)|No (DDL)|
|Speed|Slow (row by row)|Fast|Fast|

### Composite Key ⚠️ MISSED IN INTERVIEW

Two or more columns together acting as the primary key. Neither column is unique alone, but together they uniquely identify a row.

```sql
CREATE TABLE order_item (
    order_id INT,
    product_id INT,
    quantity INT,
    PRIMARY KEY (order_id, product_id)  -- composite key
);
-- Same order can have multiple products
-- Same product can be in multiple orders
-- But same order + same product = unique row
```

**In JPA:**

```java
@Embeddable
public class OrderItemId implements Serializable {
    private Long orderId;
    private Long productId;
}

@Entity
public class OrderItem {
    @EmbeddedId
    private OrderItemId id;
    private int quantity;
}
```

- **Short answer first** → expand only if asked deeper
- **Say "In my project..."** at least 2-3 times
- **Don't bluff** → "I haven't used that directly, but my understanding is..."
- **Stay calm** during coding → talk through your approach before writing
- **Ask clarifying questions** if problem is ambiguous — interviewers love this
- Interview is 40-60 mins. First 10-15 is project. Rest is technical.
