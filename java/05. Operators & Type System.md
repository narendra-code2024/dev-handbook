## Operators

### Arithmetic Operators

```java
int a = 10, b = 3;
a + b    // 13   (addition)
a - b    // 7    (subtraction)
a * b    // 30   (multiplication)
a / b    // 3    (integer division — truncates decimal)
a % b    // 1    (modulus — remainder)

// Division gotcha
10 / 3     // 3 (int / int = int — decimal lost)
10.0 / 3   // 3.333... (double / int = double)
10 / 0     // ArithmeticException
10.0 / 0   // Infinity (not exception — double division by zero)
```

### Unary Operators

```java
int a = 5;
+a     // 5   (unary plus — rarely used)
-a     // -5  (unary minus — negation)
!true  // false (logical NOT)
~a     // -6  (bitwise complement — flips all bits)

// Increment / Decrement
a++    // post-increment: use value first, then increment
++a    // pre-increment: increment first, then use value
a--    // post-decrement
--a    // pre-decrement

// Classic gotcha
int x = 5;
int y = x++;    // y = 5, x = 6 (used then incremented)
int z = ++x;    // z = 7, x = 7 (incremented then used)
```

### Assignment Operators

```java
int a = 10;
a += 5;    // a = 15  (a = a + 5)
a -= 3;    // a = 12
a *= 2;    // a = 24
a /= 4;    // a = 6
a %= 4;    // a = 2
```

See [Compound assignment (implicit narrowing cast)](<#Compound assignment (implicit narrowing cast)>) under Typecasting for the `byte b += 1` gotcha.

### Relational (Comparison) Operators

```java
a == b    // equal to
a != b    // not equal to
a > b     // greater than
a < b     // less than
a >= b    // greater than or equal
a <= b    // less than or equal

// == for primitives → compares values
// == for objects → compares references (use .equals() for content)
```

### Logical Operators

```java
true && false   // false (AND — both must be true)
true || false   // true  (OR — at least one true)
!true           // false (NOT)

// Short-circuit evaluation
// && → if left is false, right is NOT evaluated
// || → if left is true, right is NOT evaluated

int x = 0;
if (x != 0 && 10 / x > 2) { }   // safe — 10/x never executes when x=0
```

### Bitwise Operators

```java
int a = 5, b = 3;   // binary (low 4 bits): a = 0101, b = 0011
a & b    // 1  (0001)  — AND: 1 where both bits are 1
a | b    // 7  (0111)  — OR:  1 where either bit is 1
a ^ b    // 6  (0110)  — XOR: 1 where bits differ
~a       // -6         — NOT: flips every bit (see note below)

// Common uses
n & 1          // check odd/even (0 = even, 1 = odd)
n ^ n          // 0 — XORing anything with itself
flags & MASK   // test if a bit is set

// &, |, ^ work as non-short-circuiting logical ops on booleans (both sides always evaluated):
true & false   // false
true | false   // true
// Prefer &&, || unless you specifically want both sides to run.
```

> **Why `~5 = -6` and not `10`?** `int` is 32 bits, so `~` flips all 32 — including the 28 leading zeros, which become ones. That sets the sign bit, making the result negative. Shortcut: **`~x = -x - 1`**. See [Two's Complement — How Negative Integers Actually Work](<#Two's Complement — How Negative Integers Actually Work>) in the Appendix for the full story.

### Shift Operators

```java
int a = 8;     // 1000 in binary
a << 1    // 16 (10000) — left shift: multiply by 2
a >> 1    // 4  (0100)  — right shift: divide by 2 (preserves sign)
a >>> 1   // 4  (0100)  — unsigned right shift: fills with 0 (ignores sign)

// Negative number difference
int neg = -8;
neg >> 1    // -4  (sign preserved)
neg >>> 1   // large positive number (sign bit filled with 0)

// Shift distance is masked: mod 32 for int, mod 64 for long
int x = 1;
x << 32    // 1, NOT 0! (32 % 32 = 0, so effectively x << 0)

// Quick multiply/divide by powers of 2 (positive n only; negatives differ — see below)
n << 2    // n * 4   (exact, assuming no overflow)
n >> 3    // n / 8   (exact for positive n; for negatives >> floors, / truncates)
```

### Ternary Operator

```java
int max = (a > b) ? a : b;    // if true → a, else → b

// Nested (avoid — hard to read)
String grade = (score >= 90) ? "A"
             : (score >= 80) ? "B"
             : (score >= 70) ? "C"
             : "F";
```

### `instanceof` Operator

```java
// Checks if object is an instance of a class/interface
String s = "hello";
s instanceof String     // true
s instanceof Object     // true (all objects extend Object)
s instanceof Integer    // ❌ COMPILE ERROR — String and Integer are unrelated types

null instanceof String  // always false — null is not an instance of anything

// Pattern matching (Java 16+)
Object obj = "hello";
if (obj instanceof String str) {
    System.out.println(str.length());   // str already cast — no explicit cast needed
}
```

## Operator Precedence

From highest to lowest:

```
1.  Postfix:        a++, a--
2.  Unary:          ++a, --a, +a, -a, ~, !
3.  Multiplicative: *, /, %
4.  Additive:       +, -
5.  Shift:          <<, >>, >>>
6.  Relational:     <, >, <=, >=, instanceof
7.  Equality:       ==, !=
8.  Bitwise AND:    &
9.  Bitwise XOR:    ^
10. Bitwise OR:     |
11. Logical AND:    &&
12. Logical OR:     ||
13. Ternary:        ? :
14. Assignment:     =, +=, -=, etc.
```

**When in doubt, use parentheses** — readability > memorizing precedence.

## Typecasting

### Widening (Implicit) — small → large, no data loss

```java
byte → short → int → long → float → double
char → int

int i = 100;
long l = i;       // ✅ automatic — int fits in long
double d = i;     // ✅ automatic — int fits in double (100.0)
```

### Narrowing (Explicit) — large → small, possible data loss

A narrowing cast does **not** validate — it silently keeps only the **low bits that fit in the target type** and discards the rest. No exception is thrown on overflow; the contract of `(byte)`, `(int)`, etc. is "I know this may not fit, truncate anyway".

```java
double d = 9.99;
int i = (int) d;          // 9 — decimal truncated toward zero (not rounded)

long l = 130;
byte b = (byte) l;        // -126
//
//   130 as long (low bits shown):  0000 0000  1000 0010
//                                  \_________/ \_________/
//                                  thrown away  low 8 bits kept
//
//   Low 8 bits = 1000 0010.  High bit = 1 → negative (two's complement) = -126
//   (130 is 2 past byte's max of 127, so it "wraps around" — hence "overflow")

int big = 300;
byte small = (byte) big;  // 44 — same mechanic; only the low 8 bits (0010 1100) survive
```

Want overflow detection? Check explicitly — or use `Math.toIntExact(long)` / `Math.addExact` etc. for int-level overflow:

```java
long l = 130;
if (l < Byte.MIN_VALUE || l > Byte.MAX_VALUE) {
    throw new ArithmeticException("won't fit in byte");
}
byte b = (byte) l;

int i = Math.toIntExact(someLong);   // throws ArithmeticException if it doesn't fit
```

> Full coverage of overflow-checked arithmetic (`addExact`, `multiplyExact`, `toIntExact`) and related `java.lang.Math` utilities: see [Numbers — java.lang.Math utilities](<10. Numbers, Math & Random.md#`java.lang.Math` — core utilities>).

### char ↔ int

```java
char c = 'A';
int i = c;           // 65 (ASCII value — widening)
char c2 = (char) 66; // 'B' (narrowing)
```

### Compound assignment (implicit narrowing cast)

Compound assignment operators (`+=`, `-=`, `*=`, etc.) silently insert a narrowing cast — that's why `b += 1` works on a `byte` but `b = b + 1` doesn't.

```java
byte b = 10;
b = b + 1;           // ❌ COMPILE ERROR — b + 1 promotes to int, won't fit in byte
b += 1;              // ✅ works — compiles as b = (byte)(b + 1)
b = (byte)(b + 1);   // ✅ equivalent — explicit cast
```

The hidden cast means compound assignment can silently lose data too — `b += 200` on a `byte` compiles and overflows without warning.

## Type Promotion

In expressions, smaller types are **automatically promoted** to larger types before operations.

```java
byte a = 10;
byte b = 20;
// byte c = a + b;    // ❌ COMPILE ERROR — a + b promotes to int
int c = a + b;        // ✅ result is int

// Promotion ladder (narrow → wide):
//   byte / short / char  →  int  →  long  →  float  →  double
//   (char is UNSIGNED, 0..65,535 — byte/short are signed)
//
// The widest type in the expression wins (applied pairwise, left-to-right).
// Floor is int: even byte+byte → int, because the JVM has no byte/short arithmetic.

byte b = 10;
short s = 20;
int r1 = b + s;          // both promoted to int

long l = 100L;
long r2 = b + l;         // b promoted to long

float f = 1.5f;
float r3 = b + f;        // b promoted to float

double d = 2.5;
double r4 = f + d;       // f promoted to double
```

### Precision loss on `long → float` / `long → double`

`long → float` and `long → double` are both legal widenings — but `float` has only ~7 significant digits and `double` ~15. A large `long` silently rounds when widened.

```java
long big = 123_456_789L;
float f = big;               // legal — but f prints as 1.23456792E8 (rounded!)
```

## Boxing, Unboxing, and Auto-Boxing

Java has two parallel families of numeric types:

- **Primitives** (`int`, `long`, `double`, ...) — raw values, live on the stack, can't be `null`, can't be used where an `Object` is expected.
- **Wrapper classes** (`Integer`, `Long`, `Double`, ...) — full-fledged objects that *wrap* a single primitive value. Live on the heap, can be `null`, can be stored in Collections.

### The four terms

- **Boxing** — wrapping a primitive inside its wrapper object (`int → Integer`)
- **Unboxing** — extracting the primitive from a wrapper object (`Integer → int`)
- **Auto-boxing** (Java 5+) — the compiler performs boxing implicitly (no explicit `valueOf()` call)
- **Auto-unboxing** (Java 5+) — the compiler performs unboxing implicitly (no explicit `.intValue()` call)

```java
// Explicit (pre-Java 5 style)
Integer obj = Integer.valueOf(5);    // boxing
int val = obj.intValue();            // unboxing

// Implicit (Java 5+)
Integer obj = 5;                     // auto-boxing — compiler emits Integer.valueOf(5)
int val = obj;                       // auto-unboxing — compiler emits obj.intValue()
```

### Why auto-boxing was added (the pre-Java 5 pain)

- **Primitives and objects lived in separate worlds.** `int` was not an `Object`, so primitives couldn't be stored in any Collection (`List`, `Map`, `Set` all held `Object`).
- **Every conversion was manual boilerplate:**

  ```java
  list.add(new Integer(5));                          // primitive → wrapper
  int x = ((Integer) list.get(0)).intValue();        // wrapper → primitive
  ```

- **Generics made the pain louder.** When `List<Integer>` arrived in Java 5, writing `list.add(new Integer(5))` every time was friction against the very ergonomics generics were meant to provide.
- **Workarounds were ugly.** Code wrapped every primitive defensively, or used parallel libraries like `IntArrayList`. The Java stdlib had no primitive collections until `IntStream` in Java 8.

### How the compiler handles it

Implicit conversion applies anywhere primitives meet objects — Collections, method arguments, expressions, assignments, ternary branches:

```java
// Collections
List<Integer> list = new ArrayList<>();
list.add(5);                   // compiler emits list.add(Integer.valueOf(5))
int val = list.get(0);         // compiler emits list.get(0).intValue()

// Method calls with wrapper-typed parameters
void process(Integer n) { }
process(5);                    // auto-boxed

// Expressions — wrappers unbox, arithmetic happens in primitives
Integer a = 10, b = 20;
int sum = a + b;               // unboxes both, then adds
```

### Integer Cache (-128 to 127)

`Integer.valueOf()` caches values from -128 to 127, returning the same object for cached values. Small auto-boxed ints reuse cached instances; larger values allocate new objects.

```java
Integer a = 127, b = 127;
a == b;              // true  — same cached object

Integer c = 128, d = 128;
c == d;              // false — outside cache, different objects
c.equals(d);         // true  — always use .equals() for wrapper comparison
```

Cache ranges by wrapper: `Byte` (all 256 values), `Short` / `Long` / `Integer` (-128 to 127), `Character` (0 to 127), `Boolean` (`TRUE`/`FALSE`). `Float` / `Double` are not cached.

## Commonly Asked Gotchas

### Numeric / Arithmetic

1. **Division by zero — integer types throw, floating-point types don't:**

```java
10 / 0        // ❌ ArithmeticException
10.0 / 0      // Infinity       (double division by zero)
-10.0 / 0     // -Infinity
0.0 / 0.0     // NaN            (undefined result)

Double.NaN == Double.NaN     // false — NaN is not equal to anything, including itself
Double.isNaN(result)         // ✅ the only correct way to check for NaN
```

All integer types (`byte`, `short`, `int`, `long`, `char`) follow `int`'s rule (throw on `/0` or `%0`). `float` and `double` follow IEEE 754 (return ±Infinity or NaN, never throw). `BigInteger` / `BigDecimal` also throw.

2. **`float` precision loss** — use `BigDecimal` for money/exact math. See [Numbers — BigDecimal](<10. Numbers, Math & Random.md#BigDecimal — exact decimal math (money, finance)>) for `RoundingMode`, scale, and the string-constructor gotcha.

```java
float f = 0.1f + 0.2f;
// f = 0.30000001 (not 0.3) — binary can't represent 0.1 or 0.2 exactly
```

### Operators

3. **Compound assignment auto-casts** — `b += 1` works for byte, `b = b + 1` doesn't (RHS promotes to int).

4. **Short-circuit vs non-short-circuit:**

```java
// && / || skip the right side if result is already decided
if (x != 0 && 10 / x > 2) { }   // ✅ safe — 10/x skipped when x == 0

// & / | always evaluate both sides
if (x != 0 & 10 / x > 2) { }    // ❌ ArithmeticException when x == 0
```

5. **Ternary forces a common type:**

```java
Object obj = true ? Integer.valueOf(1) : Double.valueOf(2.0);
// Both branches unbox, numeric promotion → double
// Result: Double(1.0) — the Integer branch got promoted, not preserved as Integer
```

### Wrappers / Autoboxing

6. **`==` on wrapper objects compares references, not values.** Use `.equals()` (or `Objects.equals(x, y)` for null-safe comparison). Integer cache (-128..127) can make `==` *accidentally* work — don't rely on it.

7. **Unboxing `null` throws NPE:**

```java
Integer x = null;
int y = x;           // ❌ NPE — null.intValue()
if (x > 0) { }       // ❌ NPE — unboxing for comparison
```

## Appendix

### Two's Complement — How Negative Integers Actually Work

> Read this once, not every day. Helps demystify `~5 = -6`, overflow behavior, and bit-level weirdness.

**The same 32 bits mean different things depending on the type reading them.** There isn't one "correct" decimal value for a bit pattern — the type is the lens.

Take `11111111 11111111 11111111 11111010`:

```
As 4 unsigned bytes (like an IP):      255.255.255.250
As one UNSIGNED 32-bit integer:        4,294,967,290
As one SIGNED   32-bit integer (int):  -6
```

All three are mathematically valid — just different interpretations of the same bits.

### Java's `int` uses the signed lens (two's complement)

The top bit is treated as a sign flag:

```
0xxxxxxx ...   top bit 0 → positive, read the rest as plain binary
1xxxxxxx ...   top bit 1 → negative, two's-complement interpretation
```

### Decoding a negative by hand — invert-and-add-one

```
Step 1: flip every bit     11111111 11111111 11111111 11111010
                       →   00000000 00000000 00000000 00000101    = 5
Step 2: add 1              00000000 00000000 00000000 00000110    = 6
Step 3: negate                                                      = -6
```

Shortcut for `~`: **`~x == -x - 1`** always. (`~5 = -6`, `~0 = -1`, `~-1 = 0`.)

### Why two's complement?

- **Addition just works across signs.** `5 + (-6)` in two's complement gives the right bits without special-casing negatives. Hardware loves this.
- **Exactly one zero.** In sign-bit-plus-magnitude, you'd have `+0` and `-0` as different bit patterns. Two's complement has only `00000000...0000`.
- **Range is slightly asymmetric.** `int` spans `-2,147,483,648` to `+2,147,483,647` — one more negative. That extra negative is `10000000...0000`, which has no positive counterpart (and `Math.abs(Integer.MIN_VALUE)` returns itself, a famous gotcha).

### Quick reference

| Bit pattern (32-bit) | Signed int | Unsigned reading |
|---|---|---|
| `00000000 ... 00000000` | 0 | 0 |
| `00000000 ... 00000001` | 1 | 1 |
| `01111111 ... 11111111` | 2,147,483,647 (`Integer.MAX_VALUE`) | 2,147,483,647 |
| `10000000 ... 00000000` | -2,147,483,648 (`Integer.MIN_VALUE`) | 2,147,483,648 |
| `11111111 ... 11111110` | -2 | 4,294,967,294 |
| `11111111 ... 11111111` | -1 | 4,294,967,295 |
