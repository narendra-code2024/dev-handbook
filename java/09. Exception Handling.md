## 1. What & Why

An **exception** is an unexpected event that occurs during the execution of a program and disrupts its normal flow of instructions. Technically, it is an **object** that encapsulates information about an error — such as its **type** and the **state of the program** when it occurred.

Exception handling lets you catch these events gracefully instead of crashing.

```java
// Without handling — program crashes
int result = 10 / 0;   // ArithmeticException → program terminates

// With handling — program continues
try {
    int result = 10 / 0;
} catch (ArithmeticException e) {
    System.out.println("Can't divide by zero");
}
System.out.println("Program continues");  // still runs
```

## 2. Exception Hierarchy

**Quick rule:** `Exception` = your problem (app-level, recoverable). `Error` = JVM's problem (system-level, don't catch).

```
Throwable (root class — all exceptions and errors extend this. Catching Throwable is discouraged)
├── Error (DON'T catch — JVM/system level, unrecoverable)
│   ├── OutOfMemoryError
│   ├── StackOverflowError
│   └── VirtualMachineError
│
└── Exception (application-level conditions, often recoverable)
    │
    │   ⚠️ NOTE: "Checked Exception" is a CATEGORY, not a class.
    │   Any direct (or indirect) subclass of Exception that is NOT
    │   RuntimeException is considered "checked" by the compiler.
    │
    ├── IOException           ──┐
    ├── SQLException            │  ← Checked exceptions
    ├── FileNotFoundException   │    (compile-time — must handle or declare)
    ├── ClassNotFoundException  │
    ├── InterruptedException  ──┘
    │
    └── RuntimeException (unchecked — optional to handle)
        ├── NullPointerException
        ├── ArrayIndexOutOfBoundsException
        ├── ClassCastException
        ├── IllegalArgumentException
        ├── ArithmeticException
        ├── NumberFormatException
        ├── ConcurrentModificationException
        ├── UnsupportedOperationException
        └── IllegalStateException
```

## 3. Checked vs Unchecked Exceptions

|Feature|Checked Exception|Unchecked Exception|
|---|---|---|
|**Compiler Check**|Verified at **compile-time**. The compiler ensures they are handled.|Ignored at compile-time. Verified only at **runtime**.|
|**Mandatory Handling**|Yes. You **must** use a try-catch block or declare them with `throws`.|No. Handling is optional; the program will compile without explicit `catch`/`throws`.|
|**Class Hierarchy**|Extends `Exception` (but **not** `RuntimeException`).|Extends `RuntimeException` or `Error`.|
|**Typical Cause**|External factors outside the program's control (e.g., missing file, network failure).|Programming logic errors (e.g., null pointers, dividing by zero, array index out of bounds).|
|**Recovery**|Generally considered **recoverable**.|Generally considered **unrecoverable** programming bugs.|
|**Examples**|`IOException`, `SQLException`, `FileNotFoundException`|`NullPointerException`, `ClassCastException`, `ArithmeticException`|

```java
// Checked — compiler forces you to handle
public void readFile() throws IOException {     // must declare
    FileReader fr = new FileReader("file.txt"); // may throw FileNotFoundException
}

// Unchecked — compiler doesn't care
public void process(String s) {
    s.length();   // may throw NullPointerException — no handling required
}
```

**Simple rule:** If it's a programming bug (can be fixed by writing better code), it's unchecked. If it's an external failure (can't control), it's checked.

## 4. try-catch-finally

### Basic structure

```java
try {
    // risky code
    int result = 10 / 0;
} catch (ArithmeticException e) {
    // handle specific exception
    System.out.println("Error: " + e.getMessage());
} finally {
    // ALWAYS runs — whether exception or not
    // Used for cleanup (close files, connections, etc.)
    System.out.println("Cleanup done");
}
```

### Multiple catch blocks

```java
try {
    // risky code
} catch (FileNotFoundException e) {
    // most specific first
} catch (IOException e) {
    // more general second
} catch (Exception e) {
    // most general last
}
// ❌ Putting general before specific → compile error (unreachable code)
```

### Multi-catch (Java 7+)

```java
// Multiple exceptions in one catch — separated by |
try {
    // risky code
} catch (IOException | SQLException e) {
    // handle both the same way
    System.out.println("Error: " + e.getMessage());
}
// ⚠️ Exceptions in multi-catch can't be parent-child of each other
// ❌ catch (FileNotFoundException | IOException e) — FNF is child of IO
```

### finally guarantees

```java
// finally runs even if:
// - try succeeds (no exception)
// - catch handles exception
// - try or catch has return statement
// - uncaught exception occurs (finally runs, then exception propagates)

// Only skipped if: System.exit() called or JVM crashes
```

## 5. try-with-resources (Java 7+)

Automatically closes resources that implement `AutoCloseable` or `Closeable`. No need for explicit `finally` to close.

### Why try-with-resources was added (the pre-Java 7 pain)

- **Manual `finally { resource.close() }` was error-prone** — every file, socket, DB connection needed a 7-line try/finally. Developers forgot, especially for multiple resources. Forgotten close = file handle leak, connection pool exhaustion, server crash.
- **`close()` can itself throw** — forced an inner try/catch inside the `finally` block, creating ugly nested `try-finally-try-catch`. Worse: if the `try` body threw and `close()` also threw, the `close()` exception would overwrite the original (losing the real cause).
- **Multiple resources compounded the mess** — two files open? You needed nested try/finally blocks or careful null-checking. Forgetting the reverse-order close could leave dependent resources in a bad state.
- **Workaround was boilerplate + discipline** — teams wrote utility wrappers (`IOUtils.closeQuietly`) but these silenced exceptions and still required remembering to call them.

### How try-with-resources solves it

- **Automatic close on exit** — `try (Resource r = new Resource()) {...}` guarantees `r.close()` runs whether the block completes normally or throws. No finally needed.
- **Multiple resources cleanly handled** — declare several in the `try(...)` clause, each closed in **reverse order of declaration** (dependency-safe).
- **Suppressed exceptions preserve both** — if the `try` body and `close()` both throw, the body's exception is thrown and `close()`'s exception is attached via `getSuppressed()` — no more lost exceptions.
- **Type-checked via `AutoCloseable`** — compiler only allows resources whose type implements `AutoCloseable` (or older `Closeable`). Impossible to forget `close()` on a proper resource.
- **Java 9+ improvement** — can use an already-declared effectively-final variable directly: `try (br) { ... }`.

```java
// Before Java 7 — manual close in finally
BufferedReader br = null;
try {
    br = new BufferedReader(new FileReader("file.txt"));
    String line = br.readLine();
} catch (IOException e) {
    e.printStackTrace();
} finally {
    if (br != null) {
        try { br.close(); } catch (IOException e) { }  // ugly nested try
    }
}

// Java 7+ — try-with-resources (clean)
try (BufferedReader br = new BufferedReader(new FileReader("file.txt"))) {
    String line = br.readLine();
} catch (IOException e) {
    e.printStackTrace();
}
// br.close() called automatically — even if exception occurs
// If both try and close() throw, try's exception is thrown and close() exception is suppressed

// Multiple resources — closed in reverse order
try (FileInputStream fis = new FileInputStream("in.txt");
     FileOutputStream fos = new FileOutputStream("out.txt")) {
    // fos closed first, then fis
}

// In try-with-resources: close() executes BEFORE the finally block
try (MyResource r = new MyResource()) {
    // work
} finally {
    // close() already called before this runs
}

// Java 9+ — can use pre-declared effectively final variable
BufferedReader br = new BufferedReader(new FileReader("file.txt"));
try (br) {                // no need to re-declare inside try
    String line = br.readLine();
}
// ⚠️ Gotcha: if `new BufferedReader(...)` throws BEFORE the try block,
// auto-close doesn't apply — you're outside try-with-resources.
// Safer: declare resource INSIDE try(...) whenever possible.
```

### AutoCloseable vs Closeable

The resource class **must** implement one of these interfaces for try-with-resources to work.

|Interface|Package|`close()` throws|Purpose|
|---|---|---|---|
|**`AutoCloseable`**|`java.lang`|`Exception` (broader)|Parent — introduced in Java 7 for try-with-resources. Any resource type|
|**`Closeable`**|`java.io`|`IOException` (narrower)|Extends `AutoCloseable` — specifically for I/O streams (older, since Java 5)|

- **`Closeable extends AutoCloseable`** — so any `Closeable` is automatically an `AutoCloseable`
- Only classes implementing one of these can appear inside `try(...)`
- Built-in classes like `BufferedReader`, `FileInputStream`, `Connection`, `Scanner` already implement `Closeable` or `AutoCloseable`

### Custom Resource Example

```java
// Your class MUST implement AutoCloseable (or Closeable)
class MyResource implements AutoCloseable {
    @Override
    public void close() throws Exception {
        System.out.println("Resource closed");
    }
}

// Now it works with try-with-resources
try (MyResource r = new MyResource()) {
    // use resource
}  // r.close() called automatically
```

If your class does **not** implement either interface, the compiler will throw an error when you try to use it in `try(...)`.

### Which One to Use When Writing Your Own Class?

- Use **`AutoCloseable`** for general resources (DB connections, locks, etc.)
- Use **`Closeable`** only if your resource is I/O-related and throws `IOException`

## 6. throw vs throws

||`throw`|`throws`|
|---|---|---|
|What|Explicitly throw an exception|Declare that method may throw|
|Where|Inside method body|In method signature|
|Followed by|Exception object|Exception class name(s)|

```java
// throw — create and throw an exception
public void validate(int age) {
    if (age < 0) {
        throw new IllegalArgumentException("Age can't be negative");
    }
}

// throws — declare in method signature
public void readFile() throws IOException {
    // ⚠️ Don't declare redundant subclasses — FileNotFoundException is already a child of IOException
    // method may throw these — caller must handle
}

// Combined usage
public void process(String data) throws CustomException {
    if (data == null) {
        throw new CustomException("Data is null");  // throw inside
    }
}
```

### throws propagation

```java
// If you don't handle, you must declare — it bubbles up
public void methodA() throws IOException {
    methodB();   // methodB throws IOException
}

public void methodB() throws IOException {
    throw new IOException("failed");
}

// Eventually someone must catch it
public void caller() {
    try {
        methodA();
    } catch (IOException e) {
        // finally handled here
    }
}
```

## 7. Custom Exceptions

```java
// Checked custom exception — extends Exception
class InsufficientFundsException extends Exception {
    private static final long serialVersionUID = 1L;  // needed if exception is serialized
    private double amount;

    public InsufficientFundsException(String message, double amount) {
        super(message);
        this.amount = amount;
    }

    public double getAmount() { return amount; }
}

// Unchecked custom exception — extends RuntimeException
class InvalidUserException extends RuntimeException {
    public InvalidUserException(String message) {
        super(message);
    }

    public InvalidUserException(String message, Throwable cause) {
        super(message, cause);   // chaining — wraps original exception
    }
}

// Usage
public void withdraw(double amount) throws InsufficientFundsException {
    if (amount > balance) {
        throw new InsufficientFundsException("Not enough balance", amount);
    }
}
```

**When to create custom exceptions:**

- Domain-specific errors (InsufficientFundsException, UserNotFoundException)
- When built-in exceptions don't convey the right meaning
- When you need extra fields (amount, errorCode, etc.)

**Checked or Unchecked?**

- Checked — if caller can reasonably recover (retry, fallback, ask user)
- Unchecked — if it's a programming error or nothing useful the caller can do

## 8. Exception Chaining (Wrapping)

Preserve the original cause when throwing a new exception.

```java
try {
    // database operation
} catch (SQLException e) {
    // Wrap SQL exception in a business exception — original cause preserved
    throw new ServiceException("Failed to save user", e);  // e is the cause
}

// Later, retrieve the chain
catch (ServiceException e) {
    System.out.println(e.getMessage());       // "Failed to save user"
    System.out.println(e.getCause());         // original SQLException
    e.printStackTrace();                       // shows full chain
}
```

**Why chain?** — Lower layers throw technical exceptions (SQL, IO). Upper layers should throw business exceptions. Chaining keeps the original cause for debugging while presenting meaningful errors to callers.

### Wrapping Checked as Unchecked

A very common real-world pattern — **convert checked exceptions into unchecked** to avoid forcing every caller to declare `throws`. Frameworks like Spring do this everywhere (e.g., `SQLException` → `DataAccessException`).

```java
public User findUser(long id) {
    try {
        // low-level call that throws checked SQLException
        return jdbc.query("SELECT * FROM users WHERE id = ?", id);
    } catch (SQLException e) {
        // Wrap as unchecked — callers don't have to declare throws
        throw new RuntimeException("DB lookup failed for id=" + id, e);
    }
}
```

**Why?**
- Checked exceptions in lower layers (JDBC, IO) pollute every method signature up the stack
- Business code rarely handles them meaningfully at the point they occur
- Wrapping lets exceptions propagate to a central handler (e.g., global `@ControllerAdvice` in Spring)
- Original cause is preserved via `getCause()` — nothing is lost

**When NOT to wrap:** if the caller genuinely can recover (retry, fallback, ask user), keep it checked so the compiler enforces handling.

## 9. Exception Methods

```java
e.getMessage()          // error message string
e.getCause()            // original exception (if chained)
e.getStackTrace()       // array of StackTraceElement
e.printStackTrace()     // prints full stack trace — avoid in production, use logging frameworks
e.toString()            // class name + message
e.getSuppressed()       // suppressed exceptions (from try-with-resources)
```

### Suppressed Exceptions (try-with-resources)

When both try block and close() throw exceptions, the close() exception is **suppressed** (not lost).

```java
try (MyResource r = new MyResource()) {
    throw new Exception("try failed");
    // r.close() also throws → that exception is suppressed
} catch (Exception e) {
    System.out.println(e.getMessage());           // "try failed"
    System.out.println(e.getSuppressed()[0]);     // close() exception
}
```

## 10. Exception Handling with Inheritance

### Method overriding rules

```java
class Parent {
    void doWork() throws IOException { }
}

class Child extends Parent {
    // ✅ Same exception
    void doWork() throws IOException { }

    // ✅ Narrower (subclass) exception
    void doWork() throws FileNotFoundException { }

    // ✅ No exception at all
    void doWork() { }

    // ❌ Broader (parent) exception — COMPILE ERROR
    void doWork() throws Exception { }

    // ❌ New checked exception — COMPILE ERROR
    void doWork() throws SQLException { }

    // ✅ Can add unchecked exceptions freely
    void doWork() throws IOException, NullPointerException { }
}
```

**Rule:** Overriding method can throw **same, narrower, or no** checked exception — never broader or new checked. Unchecked exceptions have no restriction.

**Why?** — If parent says `throws IOException`, callers only catch `IOException`. If child throws `Exception`, the caller's catch block wouldn't handle it — breaks the contract.

## 11. Best Practices

1. **Catch specific exceptions, not generic `Exception`** — `catch(Exception e)` hides bugs.
    
2. **Don't swallow exceptions silently** — empty catch blocks hide errors.
    

```java
// ❌ Bad — error disappears
catch (Exception e) { }

// ✅ At minimum, log it
catch (Exception e) { logger.error("Failed", e); }
```

3. **Don't use exceptions for flow control** — exceptions are expensive. Avoid when the condition is frequent and cheap to check.

```java
// ❌ Bad — using exception for expected case
try { int val = Integer.parseInt(str); }
catch (NumberFormatException e) { val = 0; }

// ✅ Better — check first if cheap
if (str != null && !str.isEmpty()) {
    int val = Integer.parseInt(str);
}
```

4. **Close resources with try-with-resources** — not manually in finally.
    
5. **Prefer unchecked exceptions** for programming errors — don't force callers to handle bugs.
    
6. **Document exceptions with `@throws` in Javadoc** — even unchecked ones.
    
7. **Don't catch `Error`** — `OutOfMemoryError`, `StackOverflowError` are not recoverable.
    
8. **Throw early, catch late** — validate inputs early, let exceptions propagate to where they can be meaningfully handled.

## 12. Commonly Asked Gotchas

1. **`finally` runs even with `return`** — if both try and finally have `return`, finally's return wins. Also, finally can silently swallow exceptions.

```java
// Return override
public int test() {
    try { return 1; }
    finally { return 2; }  // returns 2, not 1!
}

// Exception lost — finally return swallows the exception
try {
    throw new IOException();
} finally {
    return;  // ❌ IOException is silently lost!
}

// Execution order — finally runs before return completes
try {
    System.out.println("try");
    return;
} finally {
    System.out.println("finally");
}
// Output: try, finally (both print, finally runs before method returns)
```

2. **`finally` doesn't run only if** — `System.exit()` is called or JVM crashes.
    
3. **Multi-catch variable is effectively final** — can't reassign `e` inside a multi-catch block.
    

```java
catch (IOException | SQLException e) {
    e = new IOException();  // ❌ COMPILE ERROR
}
```

4. **try-with-resources can exist without catch/finally** — the `try` alone is valid.

```java
try (BufferedReader br = new BufferedReader(new FileReader("file.txt"))) {
    // no catch, no finally — still valid, resource auto-closed
}
```

5. **`StackOverflowError` is an Error, not Exception** — caused by infinite recursion. Don't catch it.
    
6. **`NullPointerException` — most common runtime exception.** Java 14+ gives helpful messages showing exactly which variable was null.
    

```java
String s = null;
s.length();   // Java 14+: "Cannot invoke String.length() because s is null"
```

7. **Return in catch + finally** — finally's return overwrites catch's return. Avoid returning in finally.
    
8. **Unreachable catch block** — catching a checked exception that the try block can never throw is a compile error.
    

```java
try {
    System.out.println("hello");
} catch (IOException e) { }   // ❌ COMPILE ERROR — nothing throws IOException here
```

9. **Re-throwing exceptions** — you can catch and re-throw the same or a different exception.

```java
// Option 1: re-throw same exception
catch (SQLException e) {
    logger.error("query failed", e);
    throw e;
}

// Option 2: wrap and throw a new one (exception chaining)
catch (SQLException e) {
    throw new ServiceException("failed", e);
}

// Note: Java doesn't support bare "throw;" like C# — must specify the exception object
```

10. **throw vs throws vs Throwable** — three different things: `Throwable` = root class of all errors and exceptions. `throw` = action — explicitly throw an exception object inside method body. `throws` = declaration — in method signature, tells caller what exceptions to expect.
    
11. **Checked exceptions in lambdas** — built-in functional interfaces don't declare checked exceptions. See FP notes (gotcha #6) for solutions.

## 13. Quick Revision Cheat Sheet

|Concept|Remember|
|---|---|
|Checked exception|Must handle (try-catch) or declare (throws). External failures|
|Unchecked exception|Optional to handle. Programming bugs. Extends RuntimeException|
|Error|Don't catch. JVM/system level. Unrecoverable|
|try-with-resources|Auto-closes AutoCloseable/Closeable resources. Java 7+|
|throw|Create and throw exception object inside method body|
|throws|Declare in method signature that method may throw|
|finally|Always runs (except System.exit/JVM crash). Cleanup code|
|Multi-catch|`catch(A \| B e)` — Java 7+, can't be parent-child|
|Custom exception|Extend Exception (checked) or RuntimeException (unchecked)|
|Exception chaining|`throw new X("msg", cause)` — preserves original exception|
|Override rule|Same, narrower, or no checked exception — never broader|
|Best practice|Catch specific, don't swallow, throw early, catch late|
