## Overview

Java's numeric ecosystem spans primitives, wrappers, arbitrary-precision math, and thread-safe atomics. This file covers:

- `java.lang.Number` — the abstract parent of every numeric class
- `java.lang.Math` — stateless utility functions (abs, min, pow, sqrt, trig, rounding, overflow-checked ops)
- `BigInteger` — unlimited-size integers
- `BigDecimal` — unlimited-precision decimals (money, exact math)
- Parsing and formatting numbers
- Unsigned operations (Java 8+)
- Random number generation

For **bit-level representation** (two's complement, `~x = -x - 1`), see [Operators — Two's Complement Appendix](<05. Operators & Type System.md#Two's Complement — How Negative Integers Actually Work>). For **`Atomic*`, `LongAdder`, `LongAccumulator`**, see [Concurrency — Atomic Classes](<26. Multithreading & Concurrency.md#Atomic Classes (java.util.concurrent.atomic)>).

## `java.lang.Number` — the numeric root

All "single value" numeric classes extend the abstract `Number` class. It defines conversion methods that every subclass implements:

```java
public abstract class Number {
    public abstract int    intValue();
    public abstract long   longValue();
    public abstract float  floatValue();
    public abstract double doubleValue();
    public byte  byteValue();      // default via intValue() cast
    public short shortValue();     // default via intValue() cast
}
```

### Summary — Java numeric classes

| Class | Purpose | Extends `Number`? |
|---|---|---|
| `Character`, `Boolean` | Wrap non-numeric primitives | ❌ |
| `Byte`, `Short`, `Integer`, `Long`, `Float`, `Double` | Wrap numeric primitives | ✅ |
| `BigInteger`, `BigDecimal` | Arbitrary precision | ✅ |
| `AtomicInteger`, `AtomicLong` | Thread-safe single value | ✅ |
| `LongAdder`, `DoubleAdder`, `LongAccumulator`, `DoubleAccumulator` | High-contention counters | ✅ |

## `java.lang.Math` — core utilities

Stateless, all static methods. Operates on primitives (no boxing). `java.lang` is auto-imported.

### Absolute value, min, max

```java
Math.abs(-5)        // 5
Math.abs(-5.5)      // 5.5
Math.min(3, 7)      // 3
Math.max(3, 7)      // 7
```

### Powers, roots, logs

```java
Math.pow(2, 10)     // 1024.0  (ALWAYS returns double, even for int args)
Math.sqrt(16)       // 4.0
Math.cbrt(27)       // 3.0     (cube root)
Math.exp(1)         // 2.718...  (e^x)
Math.log(Math.E)    // 1.0       (natural log)
Math.log10(1000)    // 3.0
Math.log(8) / Math.log(2)   // 3.0  (log base 2 — no built-in method)
```

### Rounding — the three most confused methods

```java
Math.round(2.5)     // 3    (ties go toward +∞ — NOT "away from zero")
Math.round(-2.5)    // -2   (not -3!)
Math.floor(2.7)     // 2.0  (largest int ≤ x, returns double)
Math.ceil(2.1)      // 3.0  (smallest int ≥ x, returns double)
Math.floor(-2.1)    // -3.0 (floor goes toward -∞)
Math.ceil(-2.1)     // -2.0
```

| Method | Rule | `-2.5` → | `2.5` → | Returns |
|---|---|---|---|---|
| `Math.round` | Half toward `+∞` | `-2` | `3` | `long` / `int` |
| `Math.floor` | Toward `-∞` | `-3.0` | `2.0` | `double` |
| `Math.ceil`  | Toward `+∞` | `-2.0` | `3.0` | `double` |
| `(int) x`    | Toward `0` (truncate) | `-2` | `2` | `int` |

### Floor-division: `floorDiv` and `floorMod`

`/` and `%` truncate toward zero. `Math.floorDiv` and `Math.floorMod` floor toward `-∞`. The difference only matters for negatives.

```java
-7 / 2                    // -3  (truncates)
-7 % 2                    // -1  (sign matches dividend)

Math.floorDiv(-7, 2)      // -4  (floors)
Math.floorMod(-7, 2)      //  1  (always non-negative when divisor is positive)
```

Use `floorMod` when you need a non-negative remainder (e.g., wrap-around array index):

```java
int idx = Math.floorMod(-1, arr.length);   // cleanly wraps to last index
```

### Overflow-checked arithmetic (Java 8+)

Normal `+`, `*` silently overflow. The `Exact` variants throw on overflow — use them when correctness matters more than speed:

```java
Math.addExact(Integer.MAX_VALUE, 1);       // ❌ ArithmeticException
Math.multiplyExact(100_000, 100_000);      // ❌ ArithmeticException (overflow)
Math.subtractExact(Integer.MIN_VALUE, 1);  // ❌ ArithmeticException
Math.incrementExact(x);   Math.decrementExact(x);   Math.negateExact(x);

Math.toIntExact(someLong);   // throws if the long doesn't fit in int
```

### Trigonometry

```java
Math.sin(Math.PI / 2)    // 1.0   (radians, NOT degrees!)
Math.cos(0)              // 1.0
Math.tan(Math.PI / 4)    // ~1.0
Math.toRadians(180)      // π
Math.toDegrees(Math.PI)  // 180.0
```

### Constants

```java
Math.PI    // 3.141592653589793
Math.E     // 2.718281828459045
```

## BigInteger — unlimited-size integers

Rarely needed in typical app code. Reach for it when values exceed `long` range (±9.2 × 10¹⁸) — number theory, crypto, combinatorics.

```java
BigInteger a = new BigInteger("123456789012345678901234567890");
BigInteger b = BigInteger.valueOf(42);

a.add(b); a.subtract(b); a.multiply(b); a.divide(b); a.mod(b); a.pow(10);

a.compareTo(b);      // -1, 0, or 1 — don't use < == > on objects
a.gcd(b); a.modPow(exp, mod); a.isProbablePrime(certainty);  // crypto helpers

BigInteger.ZERO;  BigInteger.ONE;  BigInteger.TEN;
```

**Immutable** — `total.add(x)` returns a new object; must reassign (`total = total.add(x)`), otherwise the result is silently discarded.

## BigDecimal — exact decimal math (money, finance)

`double` can't represent `0.1` exactly. For money, tax, billing, invoicing — use `BigDecimal`.

### Creation — always use the `String` constructor (or `valueOf`)

```java
BigDecimal good = new BigDecimal("0.1");        // ✅ exact 0.1
BigDecimal bad  = new BigDecimal(0.1);          // ❌ 0.1000000000000000055511151231...
BigDecimal ok   = BigDecimal.valueOf(0.1);      // ✅ uses Double.toString internally → "0.1"
```

The `double` constructor faithfully reflects the *imprecise* binary representation. The `String` constructor preserves the exact decimal you typed.

### Arithmetic

```java
BigDecimal a = new BigDecimal("10.00");
BigDecimal b = new BigDecimal("3.00");

a.add(b);                    // 13.00
a.subtract(b);               // 7.00
a.multiply(b);               // 30.0000     (scale adds)
a.divide(b);                 // ❌ ArithmeticException — non-terminating decimal
a.divide(b, 2, RoundingMode.HALF_UP);   // ✅ 3.33 (scale=2, half-up)
```

`divide` **requires** a scale + `RoundingMode` whenever the result doesn't terminate. The default "just divide" fails for irrational decimal quotients like 10/3.

### Scale and RoundingMode

```java
new BigDecimal("3.14").scale();    // 2  (digits after decimal)
new BigDecimal("100").scale();     // 0

BigDecimal rounded = a.setScale(2, RoundingMode.HALF_UP);
```

| RoundingMode | `2.5` → | `2.35` → (scale 1) | Use when |
|---|---|---|---|
| `HALF_UP` | `3` | `2.4` | Common — "normal" rounding |
| `HALF_DOWN` | `2` | `2.3` | Rarely used |
| `HALF_EVEN` | `2` | `2.4` | Banker's rounding — reduces bias |
| `UP` | `3` | `2.4` | Always away from zero |
| `DOWN` | `2` | `2.3` | Always toward zero (truncate) |
| `CEILING` | `3` | `2.4` | Always toward `+∞` |
| `FLOOR` | `2` | `2.3` | Always toward `-∞` |

Banks and finance typically use **`HALF_EVEN`** — it's the IEEE 754 default and reduces cumulative rounding bias over many transactions.

### Comparison — `compareTo` vs `equals`

```java
new BigDecimal("2.0").equals(new BigDecimal("2.00"));     // false! (scale differs)
new BigDecimal("2.0").compareTo(new BigDecimal("2.00"));  // 0 (same value)
```

`equals` checks value **and** scale. Use `compareTo` for value equality.

## Parsing and formatting

### Parsing strings → numbers

```java
Integer.parseInt("42");              // 42
Long.parseLong("9000000000");        // 9_000_000_000L
Double.parseDouble("3.14");          // 3.14

// Radix variants
Integer.parseInt("ff", 16);          // 255 (hex)
Integer.parseInt("1010", 2);         // 10  (binary)

// Unsigned (Java 8+)
Integer.parseUnsignedInt("4000000000");
```

### Parsing edge cases

```java
Integer.parseInt(null);      // ❌ NullPointerException
Integer.parseInt("");        // ❌ NumberFormatException
Integer.parseInt(" 42 ");    // ❌ NumberFormatException — no whitespace tolerance
Integer.parseInt("1_000");   // ❌ NumberFormatException — underscore is SOURCE-CODE-only
Integer.parseInt("+42");     // ✅ 42 (leading + is allowed since Java 7)
```

For safe parsing with fallbacks, wrap in try/catch or use library helpers (Apache Commons `NumberUtils.toInt(s, default)`).

### Formatting — `String.format`, `printf`, `NumberFormat`

```java
String.format("%d", 42);               // "42"
String.format("%,d", 1_000_000);       // "1,000,000"    (grouping)
String.format("%.2f", 3.14159);        // "3.14"         (2 decimals)
String.format("%10.2f", 3.14159);      // "      3.14"   (min width 10)
String.format("%-10.2f|", 3.14);       // "3.14      |"  (left-aligned)
String.format("%08.2f", 3.14);         // "00003.14"     (zero-pad)
String.format("%e", 1234567.89);       // "1.234568e+06" (scientific)

// Locale-aware
NumberFormat fmt = NumberFormat.getCurrencyInstance(Locale.US);
fmt.format(1234.56);   // "$1,234.56"

NumberFormat pct = NumberFormat.getPercentInstance();
pct.format(0.75);      // "75%"

// Custom pattern
DecimalFormat df = new DecimalFormat("#,##0.00");
df.format(1234.5);     // "1,234.50"
```

> `DecimalFormat` is **not thread-safe**. In multi-threaded code, use a `ThreadLocal<DecimalFormat>` or re-create per call. Same for `SimpleDateFormat`.

## Unsigned operations (Java 8+)

Rarely needed in typical app code — mostly for low-level byte/network protocol parsing and C interop. Java has no `uint` / `ulong` type; `int` and `long` are always signed. Static helpers on `Integer` / `Long` reinterpret the bits as unsigned:

```java
Integer.toUnsignedLong(-1);           // 4,294,967,295 (same bits, unsigned lens)
Integer.divideUnsigned(a, b);
Integer.compareUnsigned(a, b);
Integer.parseUnsignedInt("4294967295");
```

Bit-level "same bits, different lens" explanation: [Operators — Two's Complement](<05. Operators & Type System.md#Two's Complement — How Negative Integers Actually Work>).

## Random number generation

### `java.util.Random` — classic (Java 1.0)

```java
Random r = new Random();              // seeded from nanoTime
Random r = new Random(42L);           // deterministic seed (reproducible tests)

r.nextInt();                          // any int
r.nextInt(100);                       // 0..99 inclusive
r.nextInt(10, 20);                    // 10..19 (Java 17+)
r.nextLong();
r.nextDouble();                       // [0.0, 1.0)
r.nextBoolean();
r.nextGaussian();                     // normal distribution (mean 0, stddev 1)

// Stream API (Java 8+)
r.ints(10, 1, 100)                    // 10 ints in [1, 100)
 .forEach(System.out::println);
```

### `ThreadLocalRandom` — for multi-threaded code

`Random` is thread-safe but contended (a shared seed under CAS). In concurrent code, use `ThreadLocalRandom` — one instance per thread, zero contention.

```java
int n = ThreadLocalRandom.current().nextInt(100);
double d = ThreadLocalRandom.current().nextDouble();
```

### `SplittableRandom` — for parallel streams / fork-join

Designed for recursive splitting. Each `split()` yields an independent sequence. Use inside `parallelStream()` / fork-join tasks.

```java
SplittableRandom r = new SplittableRandom();
SplittableRandom child = r.split();   // independent sequence, safe to pass to another task
```

### `SecureRandom` — cryptographic

Use for tokens, passwords, session IDs, nonces. Non-deterministic by design; slower but unpredictable.

```java
SecureRandom secure = new SecureRandom();
byte[] token = new byte[16];
secure.nextBytes(token);
```

### Choosing the right one

| Use case | Class |
|---|---|
| Single-threaded, need reproducibility | `Random(seed)` |
| Multi-threaded | `ThreadLocalRandom.current()` |
| Parallel streams / divide-and-conquer | `SplittableRandom` |
| Security (tokens, passwords, nonces) | `SecureRandom` |

> **Never** use `Random` or `ThreadLocalRandom` for anything security-sensitive — their sequences are predictable if the seed leaks (or via statistical analysis). Use `SecureRandom`.

## Commonly Asked Gotchas

1. **`Math.pow(2, 10)` returns `double`** — even with int args. Cast back: `(int) Math.pow(2, 10)`. For exact integer powers, use `BigInteger.valueOf(2).pow(10)` or a loop.
2. **`double` for money is a bug** — `0.1 + 0.2 = 0.30000000000000004`. Use `BigDecimal` with the `String` constructor.
3. **`BigDecimal.equals` checks scale** — `new BigDecimal("2.0").equals(new BigDecimal("2.00")) == false`. Use `compareTo` for value equality.
4. **`BigDecimal(double)` captures imprecision** — `new BigDecimal(0.1)` stores the inexact binary value. Always pass a `String`, or use `BigDecimal.valueOf(double)`.
5. **`BigDecimal.divide` without scale throws** — for non-terminating decimals (10/3). Always pass `(scale, RoundingMode)`.
6. **Integer overflow is silent** — `Integer.MAX_VALUE + 1` wraps to `Integer.MIN_VALUE`. Use `Math.addExact` / `multiplyExact` / `toIntExact` when correctness matters.
7. **`new Random()` uses nanoTime seed** — two `Random` instances created in the same tick may yield the same sequence. In parallel code, prefer `ThreadLocalRandom`.
8. **`DecimalFormat` is not thread-safe** — same rule as `SimpleDateFormat`. Either `ThreadLocal` it or re-create per call.
9. **`Integer.parseInt("1_000")` fails** — underscores are a *source-code* feature (`int x = 1_000;`), not a runtime one. Strip them first if your input has them.
