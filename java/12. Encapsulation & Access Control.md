## 1. What is Encapsulation

**Hiding internal data** and controlling access through public methods. The object decides what's visible and how it can be modified.

```java
// Without encapsulation — anyone can set invalid data
class Employee {
    public int salary;
}
Employee e = new Employee();
e.salary = -5000;    // no validation — invalid state

// With encapsulation — controlled access
class Employee {
    private int salary;

    public void setSalary(int salary) {
        if (salary < 0) throw new IllegalArgumentException("Salary can't be negative");
        this.salary = salary;
    }

    public int getSalary() { return salary; }
}
```

**Encapsulation = data hiding + controlled access.** Not just "private fields + getters/setters" — the point is **validation and control**.

## 2. Access Modifiers

|Modifier|Same Class|Same Package|Subclass (diff package)|Other Package|
|---|---|---|---|---|
|`private`|✅|❌|❌|❌|
|`default` (no keyword)|✅|✅|❌|❌|
|`protected`|✅|✅|✅|❌|
|`public`|✅|✅|✅|✅|

**Memory aid:** private → default → protected → public (least to most visible)

### Where each modifier can be used

||Class (top-level)|Field|Method|Constructor|Inner Class|
|---|---|---|---|---|---|
|`private`|❌|✅|✅|✅|✅|
|`default`|✅|✅|✅|✅|✅|
|`protected`|❌|✅|✅|✅|✅|
|`public`|✅|✅|✅|✅|✅|

**Top-level classes can only be `public` or `default`** — `private` and `protected` don't make sense for top-level (who would access them?).

## 3. Access Modifier Details

### private

```java
class Account {
    private double balance;          // only this class can access
    private Account() { }           // private constructor — prevents external instantiation

    public double getBalance() {     // controlled access
        return balance;
    }
}

// Private members accessible within same class — even from other instances
class Account {
    private double balance;

    boolean hasMoreThan(Account other) {
        return this.balance > other.balance;   // ✅ can access other's private field
    }
}
```

### default (package-private)

```java
// No keyword — just leave it blank
class Helper {          // default class — only visible within same package
    void process() { }  // default method — same package only
}
```

### protected

```java
// Accessible in: same class + same package + subclass (even different package)
// ⚠️ In different package: accessible ONLY via inheritance (this), NOT via parent reference
class Animal {
    protected String name;
    protected void makeSound() { }
}

// Different package
class Dog extends Animal {
    void test() {
        this.name = "Buddy";    // ✅ via inheritance (child reference)
        makeSound();             // ✅ inherited method

        Animal a = new Animal();
        a.name = "X";           // ❌ NOT allowed — parent reference in different package
    }
}
```

### public

```java
public class UserService {              // accessible everywhere
    public void createUser() { }        // accessible everywhere
}
```

## 4. Getter and Setter Methods

```java
class Employee {
    private String name;
    private int age;

    // Getter — read access
    public String getName() { return name; }

    // Setter — write access with validation
    public void setAge(int age) {
        if (age < 18 || age > 65) {
            throw new IllegalArgumentException("Invalid age");
        }
        this.age = age;
    }

    // Read-only field — getter only, no setter
    public int getAge() { return age; }
}
```

**Not all fields need both getter and setter:**

- Read-only → getter only
- Write-only → setter only (rare — usually for write-only APIs like passwords)
- Internal-only → no getter/setter
- Computed field → getter with logic, no underlying field

```java
// Computed getter — no matching field
public String getFullName() {
    return firstName + " " + lastName;
}
```

## 5. Mutable vs Immutable Objects

### Mutable — state can change after creation

```java
class Employee {
    private String name;
    public void setName(String name) { this.name = name; }  // state changes
}

Employee e = new Employee();
e.setName("Alice");
e.setName("Bob");    // object changed — same reference, different state
```

### Immutable — state cannot change after creation

```java
// String is immutable
String s = "hello";
s.toUpperCase();           // returns new String, original unchanged
System.out.println(s);     // still "hello"

// Integer, LocalDate, etc. are also immutable
```

**Why immutable?**

- Thread-safe — no synchronization needed
- Predictable — no side effects
- Safe as HashMap keys — hashCode never changes
- Cacheable — can be shared freely

## 6. Creating an Immutable Class (5 Steps)

```java
// 1. Class is final — can't extend (prevents overriding methods to break immutability)
public final class Employee {

    // 2. Fields are private final — can't reassign
    private final String name;
    private final int salary;
    private final List<String> skills;

    // 3. Initialize via constructor
    public Employee(String name, int salary, List<String> skills) {
        this.name = name;
        this.salary = salary;
        // 5. Defensive copy of mutable fields — prevent external modification
        this.skills = new ArrayList<>(skills);
    }

    // 4. No setters — only getters
    public String getName() { return name; }
    public int getSalary() { return salary; }

    // 5. Return defensive copy from getter — prevent modification via returned reference
    public List<String> getSkills() {
        return new ArrayList<>(skills);    // copy, not original
        // NOT: return skills;             // ❌ caller could modify internal list
    }
}
```

### The 5 Rules

1. **`final` class** — prevent subclasses from breaking immutability
2. **`private final` fields** — no direct access, no reassignment
3. **No setters** — no way to change state
4. **Constructor initialization** — all state set at creation
5. **Defensive copies** — for mutable fields (List, Date, arrays), copy in constructor AND in getter. If fields are already immutable (e.g., String, Integer), no defensive copy needed.

### Why defensive copies matter

```java
// Without defensive copy — broken immutability
List<String> skills = new ArrayList<>(List.of("Java", "SQL"));
Employee emp = new Employee("Alice", 50000, skills);
skills.add("Python");              // ❌ modifies internal state if no copy in constructor!
emp.getSkills().add("Python");     // ❌ modifies internal state if no copy in getter!

// With defensive copy — truly immutable
// Constructor copies input → external list changes don't affect object
// Getter copies output → caller can't modify internal list
```

## 7. Singleton Class

Only **one instance** of the class can exist. Common for configs, connection pools, loggers.

### Eager Initialization

```java
public class Config {
    private static final Config INSTANCE = new Config();   // created at class loading
    private Config() { }                                    // private constructor
    public static Config getInstance() { return INSTANCE; }
}
```

### Lazy Initialization (Thread-safe — Bill Pugh)

```java
public class Config {
    private Config() { }

    private static class Holder {
        private static final Config INSTANCE = new Config();  // created only when accessed
    }

    public static Config getInstance() {
        return Holder.INSTANCE;    // inner class loaded on first call
    }
}
```

### Double-Checked Locking

```java
public class Config {
    private static volatile Config instance;
    private Config() { }

    public static Config getInstance() {
        if (instance == null) {                    // first check (no lock)
            synchronized (Config.class) {
                if (instance == null) {            // second check (with lock)
                    instance = new Config();
                }
            }
        }
        return instance;
    }
}
```

### Enum Singleton (Best — recommended)

```java
public enum Config {
    INSTANCE;

    private String dbUrl;

    public String getDbUrl() { return dbUrl; }
    public void setDbUrl(String url) { this.dbUrl = url; }
}

Config.INSTANCE.getDbUrl();
```

**Why enum is best:** inherently serialization-safe, reflection-safe, thread-safe, concise.

### Singleton Comparison

|Approach|Thread-safe|Lazy|Serialization-safe|Reflection-safe|
|---|---|---|---|---|
|Eager|✅|❌|❌ (unless `readResolve()`)|❌|
|Bill Pugh (Holder)|✅|✅|❌ (unless `readResolve()`)|❌|
|Double-checked|✅|✅|❌ (unless `readResolve()`)|❌|
|Enum|✅|Effectively eager (class-loading based)|✅|✅|

**Breaking Singleton:** Reflection can invoke private constructor via `setAccessible(true)` → creates new instance. Serialization and Cloning can also create multiple instances. Enum is immune to all three. **Fix for Serialization:** implement `readResolve()` (covered in Serialization notes).

## 8. Commonly Asked Gotchas

1. **Encapsulation ≠ just getters/setters** — the point is validation and controlled access. Getters/setters without validation are just extra code.
    
2. **`protected` is wider than default** — many people think `protected` is more restrictive. It's not. protected = package + subclass. Default = package only.
    
3. **`private` doesn't mean other objects can't access it** — other instances of the SAME class can access private members.
    

```java
class Account {
    private int balance;
    boolean richerThan(Account other) {
        return this.balance > other.balance;   // ✅ accessing other's private field
    }
}
```

4. **Immutable class with mutable field — defensive copy is essential:**

```java
// Without defensive copy → list can be modified externally
// Always copy in constructor (input) AND getter (output)
```

5. **`final` field ≠ immutable** — `final` means the reference can't change, but the object it points to can still be modified.

```java
final List<String> list = new ArrayList<>();
list.add("hello");     // ✅ list content changes — final only prevents reassignment
// list = new ArrayList<>();  // ❌ can't reassign reference
```

6. **Singleton is broken by:** Reflection (`setAccessible(true)` on private constructor), Serialization (deserialization creates new instance), Cloning. Enum singleton is immune to all three.
    
7. **`protected` constructor** — allows subclasses in other packages to instantiate via `super()`, but prevents direct `new` from non-subclass code in other packages.
    
8. **Default access (no modifier) is NOT `public`** — common beginner mistake. No modifier = package-private.
    
9. **Immutable objects can be shared freely between threads** — no synchronization needed. This is why `String`, `Integer`, `LocalDate` are thread-safe.
    
10. **Singleton anti-pattern** — overuse of Singleton makes code hard to test (hidden dependencies). Prefer Dependency Injection (e.g., Spring-managed beans) to avoid hidden dependencies.

## 9. Quick Revision Cheat Sheet

|Concept|Remember|
|---|---|
|Encapsulation|Data hiding + controlled access (not just getters/setters)|
|`private`|Same class only. Other instances of same class CAN access|
|`default`|Same package only. No keyword|
|`protected`|Same package + subclasses. Wider than default|
|`public`|Everywhere|
|Top-level class|Only `public` or `default`|
|Immutable class|final class + private final fields + no setters + defensive copies|
|Defensive copy|Copy mutable fields in constructor AND getter|
|`final` ≠ immutable|final = can't reassign reference. Object content can still change|
|Singleton|One instance. Enum is best. Bill Pugh for lazy|
|Singleton breakers|Reflection, Serialization, Cloning (Enum is immune)|
|Mutable vs Immutable|Mutable = state changes. Immutable = state fixed at creation|
