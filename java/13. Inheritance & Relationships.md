## 1. What is Inheritance

A child class **acquires fields and methods** of a parent class. Promotes code reuse and establishes IS-A relationship.

```java
class Animal {
    String name;
    void eat() { System.out.println(name + " is eating"); }
}

class Dog extends Animal {
    void bark() { System.out.println(name + " is barking"); }
}

Dog d = new Dog();
d.name = "Buddy";    // inherited field
d.eat();              // inherited method
d.bark();             // own method
```

**What is inherited:**

- public and protected members — always
- default (package-private) members — only if same package
- static members — inherited but resolved at compile-time (method hiding, not overriding)
- **NOT** inherited: private members (exist in object memory but not accessible), constructors

**What child can do:**

- Use inherited members directly
- Override inherited methods
- Add new fields and methods
- Call parent constructor via `super()`

## 2. `extends` Keyword

```java
class Child extends Parent { }

// Java supports single inheritance for classes — one parent only
class A extends B { }        // ✅
class A extends B, C { }     // ❌ COMPILE ERROR
```

**Every class implicitly extends `Object`** — if no explicit parent, compiler adds `extends Object`. So `Object` is the root of all classes.

```java
class Employee { }
// Compiler sees: class Employee extends Object { }
// That's why toString(), equals(), hashCode() are available on every object
```

## 3. `super` Keyword

### Call parent constructor

```java
class Animal {
    String name;
    Animal(String name) { this.name = name; }
}

class Dog extends Animal {
    String breed;
    Dog(String name, String breed) {
        super(name);           // calls Animal(String) — MUST be first line
        this.breed = breed;
    }
}
```

**Constructor chaining rules:**

- `super()` is implicitly added if you don't write it — calls parent no-arg constructor
- If parent has NO no-arg constructor → child MUST explicitly call `super(args)`
- `super()` must be first line — can't coexist with `this()` on first line

```java
class Animal {
    Animal(String name) { }     // no no-arg constructor
}

class Dog extends Animal {
    Dog() { }                   // ❌ COMPILE ERROR — no super() match
    Dog() { super("Dog"); }    // ✅ explicitly calls parent constructor
}
```

### Access parent method/field

```java
class Animal {
    String type = "Animal";
    void sound() { System.out.println("Some sound"); }
}

class Dog extends Animal {
    String type = "Dog";        // hides parent field (not override — fields don't override)

    void sound() {
        super.sound();          // calls parent's sound()
        System.out.println("Bark");
    }

    void printType() {
        System.out.println(this.type);    // "Dog"
        System.out.println(super.type);   // "Animal"
    }
}
```

## 4. Types of Inheritance

```java
// Single — A → B
class Animal { }
class Dog extends Animal { }

// Multilevel — A → B → C
class Animal { }
class Dog extends Animal { }
class Puppy extends Dog { }

// Hierarchical — A → B, A → C
class Animal { }
class Dog extends Animal { }
class Cat extends Animal { }

// Multiple (NOT supported with classes — supported with interfaces)
class A extends B, C { }     // ❌
class A implements X, Y { }  // ✅ (interfaces)

// Hybrid — combination of above (possible only through interfaces)
```

### Why Java Doesn't Support Multiple Inheritance (with classes)

**Diamond Problem:**

```
    Animal
   /      \
  Dog     Cat
   \      /
    Hybrid      ← which eat() method?
```

If `Dog` and `Cat` both override `eat()`, `Hybrid` doesn't know which one to use → ambiguity in method resolution.

**Interfaces solve this** — because interfaces don't carry instance state and force explicit resolution of default method conflicts. If two interfaces have the same default method → implementing class MUST override it.

```java
interface Flyable { default void move() { System.out.println("Fly"); } }
interface Swimmable { default void move() { System.out.println("Swim"); } }

class Duck implements Flyable, Swimmable {
    // Must resolve conflict explicitly
    public void move() {
        Flyable.super.move();    // choose one, or write own logic
    }
}
```

## 5. Generalization and Specialization

```
        Vehicle            ← Generalization (moving UP — abstract common traits)
       /       \
    Car       Bike          ← Specialization (moving DOWN — adding specific traits)
   /    \
Sedan   SUV
```

- **Generalization** — extracting common behavior into a parent class (Car and Bike both move → Vehicle)
- **Specialization** — adding specific behavior in child class (SUV has offRoad(), Sedan doesn't)

## 6. IS-A vs HAS-A Relationships

### IS-A (Inheritance) — `extends` / `implements`

```java
class Dog extends Animal { }      // Dog IS-A Animal
class ArrayList implements List { } // ArrayList IS-A List

// Tested with instanceof
Dog d = new Dog();
d instanceof Animal    // true
d instanceof Dog       // true
d instanceof Object    // true
```

### HAS-A (Composition/Aggregation) — field reference

```java
class Engine { }
class Car {
    Engine engine;      // Car HAS-A Engine
}
```

### Association, Aggregation, Composition

||Association|Aggregation|Composition|
|---|---|---|---|
|Relationship|Uses|Has (weak)|Has (strong)|
|Lifecycle|Independent|Independent|Dependent|
|Example|Teacher-Student|Department-Employee|House-Room|
|If parent destroyed|Other survives|Other survives|Other destroyed too|

```java
// Aggregation — Employee can exist without Department
class Department {
    List<Employee> employees;    // employees exist independently
}

// Composition — Room can't exist without House
class House {
    private List<Room> rooms;

    House() {
        rooms = new ArrayList<>();
        rooms.add(new Room());     // created inside — lifecycle tied
    }
}

class Room { }     // meaningless without a House
```

**When to use which:**

- IS-A → "Dog is a type of Animal" → inheritance
- HAS-A → "Car has an Engine" → composition
- **Prefer composition over inheritance** — more flexible, avoids tight coupling, easier to change

## 7. Object Creation in Inheritance

```java
class Animal {
    Animal() { System.out.println("Animal constructor"); }
}

class Dog extends Animal {
    Dog() {
        // super() added implicitly
        System.out.println("Dog constructor");
    }
}

new Dog();
// Output:
// Animal constructor    ← parent first
// Dog constructor       ← then child
```

**Constructor chain always goes up to `Object`:**

```
new Dog() → Dog() calls super() → Animal() calls super() → Object()
```

`Object` has only a no-arg constructor and is always the last call in the chain. Parent is **always constructed first** — child may depend on parent's state being initialized.

## 8. Commonly Asked Gotchas

1. **Constructors are NOT inherited** — child must define its own. But child's constructor always calls parent's constructor (explicitly or implicitly via `super()`).
    
2. **Private members are NOT inherited** — but they exist in the child object (parent's private field is in heap memory, just not accessible directly). Accessible via parent's public/protected methods.
    

```java
class Parent {
    private int x = 10;
    public int getX() { return x; }
}
class Child extends Parent {
    // x is not accessible directly
    // but getX() works — accesses parent's private field
}
```

3. **Fields don't override — they hide.** Field access is compile-time binding, unlike method overriding which is runtime (dynamic binding).

```java
class Parent { int x = 10; }
class Child extends Parent { int x = 20; }

Child c = new Child();
Parent p = c;
System.out.println(c.x);    // 20 — Child's x
System.out.println(p.x);    // 10 — Parent's x (field access is based on reference type)
// Unlike methods, field access is NOT polymorphic
```

4. **`instanceof` with null is always false:**

```java
Dog d = null;
d instanceof Animal    // false — null is not an instance of anything
```

5. **Casting in inheritance:**

```java
Animal a = new Dog();          // ✅ upcasting — implicit, always safe
Dog d = (Dog) a;               // ✅ downcasting — explicit, works if actual object is Dog
Cat c = (Cat) a;               // ❌ ClassCastException — actual object is Dog, not Cat

// Always check before downcasting
if (a instanceof Dog d) {      // Java 16+ pattern matching      // Java 16+ pattern matching
    d.bark();
}
```

6. **`super()` is added implicitly** — if you don't write `super()` or `this()` on the first line, compiler adds `super()`. If parent has no no-arg constructor → compile error.
    
7. **Cyclic inheritance is not allowed:**
    

```java
class A extends B { }
class B extends A { }     // ❌ COMPILE ERROR
```

8. **`final` class can't be extended:**

```java
final class String { }
class MyString extends String { }   // ❌ COMPILE ERROR
```

9. **Prefer composition over inheritance** — inheritance creates tight coupling (child depends on parent's implementation). Composition is more flexible and testable. Use inheritance only when there is a true IS-A relationship and behavioral substitution (Liskov Substitution Principle).
    
10. **Covariant return type** — overriding method can return a subtype of the parent's return type:
    

```java
class Animal {
    Animal create() { return new Animal(); }
}
class Dog extends Animal {
    Dog create() { return new Dog(); }    // ✅ Dog is subtype of Animal
}
```

11. **Fragile base class problem** — changes in parent class can unintentionally break child classes. Another reason to prefer composition over inheritance.

## 9. Quick Revision Cheat Sheet

|Concept|Remember|
|---|---|
|`extends`|Single inheritance only. Every class extends Object implicitly|
|`super()`|Calls parent constructor. Must be first line. Added implicitly if missing|
|Inherited|public, protected members. NOT private, NOT constructors|
|Fields|Don't override — they hide. Access based on reference type, not object type|
|Diamond problem|Why no multiple inheritance with classes. Interfaces resolve it|
|IS-A|Inheritance (`extends`, `implements`). Tested with `instanceof`|
|HAS-A|Composition/Aggregation. Field reference to another object|
|Aggregation|Weak HAS-A. Parts exist independently (Department-Employee)|
|Composition|Strong HAS-A. Parts die with parent (House-Room)|
|Prefer|Composition over inheritance — more flexible, less coupling|
|Upcasting|Child → Parent. Implicit, always safe|
|Downcasting|Parent → Child. Explicit, check with `instanceof` first|
|Covariant return|Override can return subtype of parent's return type|
|Constructor chain|Always goes up: Child → Parent → ... → Object|
