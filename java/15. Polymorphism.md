## What is Polymorphism

**Same method name, different behavior** depending on context. "Many forms."

Two types:

- **Compile-time** (Static/Early Binding) — resolved at compile time → **Method Overloading**
- **Runtime** (Dynamic/Late Binding) — resolved at runtime → **Method Overriding**

---

## Compile-Time Polymorphism — Method Overloading

Same method name, **different parameter list** (number, type, or order). Compiler decides which to call based on arguments.

```java
class Calculator {
    int add(int a, int b) { return a + b; }
    double add(double a, double b) { return a + b; }
    int add(int a, int b, int c) { return a + b + c; }
}

Calculator c = new Calculator();
c.add(1, 2);           // calls add(int, int)
c.add(1.5, 2.5);       // calls add(double, double)
c.add(1, 2, 3);        // calls add(int, int, int)
```

### Rules

- **CAN change:** parameter list, return type, access modifier, exceptions
- **MUST change:** parameter list (otherwise it's not overloading)
- **Can't overload by return type alone:**

```java
int add(int a, int b) { }
double add(int a, int b) { }     // ❌ COMPILE ERROR — same parameters
```

### Types of overloading

```java
// Instance method overloading
class Test {
    void show(int x) { }
    void show(String s) { }
}

// Static method overloading
class MathUtil {
    static int max(int a, int b) { }
    static double max(double a, double b) { }
}

// Constructor overloading
class Employee {
    Employee() { }
    Employee(String name) { }
    Employee(String name, int salary) { }
}

// Can overload main method (but JVM only calls String[] version)
public static void main(String[] args) { }     // ← JVM calls this
public static void main(int x) { }             // valid overload, but not entry point
```

### Overloading Resolution Priority

When multiple methods could match, Java resolves in this order:

```
1. Exact match
2. Widening (int → long)
3. Autoboxing (int → Integer)
4. Varargs (int... → last resort)
```

```java
void process(int x) { }        // 1. exact
void process(long x) { }       // 2. widening
void process(Integer x) { }    // 3. autoboxing
void process(int... x) { }     // 4. varargs

process(5);    // calls process(int) — exact match

// Ambiguity example
void process(int x, double y) { }
void process(double x, int y) { }
process(5, 5);    // ❌ COMPILE ERROR — ambiguous, both match via widening
```

### Autoboxing + Widening Interaction

```java
// Java allows:
// - Widening (int → long)
// - Boxing (int → Integer)
// - Boxing + widening (int → Integer → Object)

// Java does NOT allow:
// - Widening + boxing (int → long → Long)

void m(Long x) { }
m(5);              // ❌ COMPILE ERROR — can't widen int→long THEN box long→Long

void m(Object x) { }
m(5);              // ✅ boxes int→Integer, then widens Integer→Object
```

## Runtime Polymorphism — Method Overriding

Child class provides **specific implementation** of a method already defined in parent. JVM decides which method to call based on **actual object type** at runtime.

```java
class Animal {
    void sound() { System.out.println("Some sound"); }
}

class Dog extends Animal {
    @Override
    void sound() { System.out.println("Bark"); }
}

class Cat extends Animal {
    @Override
    void sound() { System.out.println("Meow"); }
}

Animal a = new Dog();    // reference: Animal, object: Dog
a.sound();               // "Bark" — calls Dog's method, not Animal's (runtime decision)

Animal b = new Cat();
b.sound();               // "Meow"
```

### Rules for Overriding

|Rule|Requirement|
|---|---|
|Method name|Must be same|
|Parameters|Must be same (exact match)|
|Return type|Same or covariant (subtype)|
|Access modifier|Same or wider (can't reduce visibility)|
|Exceptions|Same, narrower, or no checked exception (can't add new/broader)|
|`static` methods|Can't override — only hide|
|`final` methods|Can't override|
|`private` methods|Can't override (not inherited)|
|Constructors|Can't override (not inherited)|

```java
class Parent {
    protected Animal create() throws IOException { return new Animal(); }
}

class Child extends Parent {
    @Override
    public Dog create() throws FileNotFoundException {    // ✅ all valid:
        return new Dog();                                  // public > protected (wider access)
    }                                                      // Dog is subtype of Animal (covariant)
}                                                          // FileNotFoundException is subtype of IOException (narrower)
```

### `@Override` Annotation

```java
@Override    // optional but strongly recommended
void sound() { }

// Benefits:
// - Compiler verifies it's actually overriding (catches typos)
// - Documents intent
// - Compile error if parent method doesn't exist or signature doesn't match
```

## Static Method Hiding (Not Overriding)

Static methods are **hidden**, not overridden. Resolved at **compile time** based on reference type.

```java
class Parent {
    static void greet() { System.out.println("Parent"); }
}

class Child extends Parent {
    static void greet() { System.out.println("Child"); }   // hides, not overrides
}

Parent p = new Child();
p.greet();          // "Parent" — resolved by reference type (compile time)
                    // If it were overriding → would print "Child"

Child c = new Child();
c.greet();          // "Child" — reference type is Child
```

**Key difference:** Overriding = runtime (based on object type). Hiding = compile time (based on reference type).

## Polymorphism in Action

```java
// Method accepting parent type — works with any child
void makeSound(Animal a) {
    a.sound();    // calls appropriate version based on actual object
}

makeSound(new Dog());     // "Bark"
makeSound(new Cat());     // "Meow"

// Collections of parent type — store any child
List<Animal> animals = new ArrayList<>();
animals.add(new Dog());
animals.add(new Cat());

for (Animal a : animals) {
    a.sound();    // each calls its own version — polymorphism
}
```

**This is the real power** — write code against the parent type/interface, and it works with any implementation. New animal types can be added without changing existing code.

## Overloading vs Overriding

||Overloading|Overriding|
|---|---|---|
|When|Compile time|Runtime|
|Where|Same class (or inherited)|Parent-child|
|What changes|Parameter list|Implementation (body)|
|What stays same|Method name|Method name + parameters|
|Return type|Can differ|Same or covariant|
|Access|Can differ|Same or wider|
|Static methods|Can overload|Can't override (only hide)|
|Private methods|Can overload|Can't override|
|Binding|Static/Early|Dynamic/Late|

## Commonly Asked Gotchas

1. **Compile-time vs runtime polymorphism** — Java defines both overloading (compile-time) and overriding (runtime) as polymorphism. In practice, "polymorphism" usually refers to runtime polymorphism. The runtime mechanism is called **Dynamic Method Dispatch** — method call resolved at runtime based on actual object type.

2. **Overloading with `null`:**

```java
void process(String s) { }
void process(Object o) { }
process(null);    // calls process(String) — most specific type wins

void process(String s) { }
void process(Integer i) { }
process(null);    // ❌ COMPILE ERROR — ambiguous, both are equally specific
```

3. **Variable/field access is NOT polymorphic:**

```java
class Parent { int x = 10; }
class Child extends Parent { int x = 20; }

Parent p = new Child();
p.x;     // 10 — fields resolved by reference type, not object type
// Only methods are polymorphic, not fields
```

4. **`instanceof` check before casting in polymorphism:**

```java
Animal a = getAnimal();
if (a instanceof Dog d) {     // Java 16+ pattern matching
    d.fetch();                 // safe — only runs if actually a Dog
}
```

5. **Can you override `main()`?** — No. `main()` is static → can only be hidden, not overridden. JVM looks at the class you run, not polymorphism.

6. **Method dispatch with overloading + overriding together:**


```java
class Parent {
    void process(Object o) { System.out.println("Parent-Object"); }
}
class Child extends Parent {
    void process(Object o) { System.out.println("Child-Object"); }   // override
    void process(String s) { System.out.println("Child-String"); }   // overload
}

Parent p = new Child();
p.process("hello");    // "Child-Object" — NOT "Child-String"!
// Compiler sees Parent reference → only knows process(Object)
// At runtime → Child's override of process(Object) is called
// process(String) is not visible through Parent reference
```

## Quick Revision Cheat Sheet

|Concept|Remember|
|---|---|
|Overloading|Same name, different params. Compile-time. Same or different class|
|Overriding|Same name + params, different body. Runtime. Parent-child only|
|Resolution priority|Exact → widening → autoboxing → varargs|
|Return type|Overloading: can differ. Overriding: same or covariant|
|Access modifier|Overloading: can differ. Overriding: same or wider|
|Static methods|Can overload. Can't override (hide only — compile-time)|
|`final` methods|Can overload. Can't override|
|`private` methods|Can overload. Can't override (not inherited)|
|`@Override`|Optional but recommended. Compiler checks correctness|
|Fields|NOT polymorphic — resolved by reference type|
|null resolution|Most specific type wins. Equal specificity → ambiguous|
|Real power|Code to parent type/interface → works with any child|
