## What is Abstraction

**Hiding implementation details, showing only essential functionality.** The user knows _what_ it does, not _how_ it does it.

Real-world: you drive a car (steering, pedals) without knowing engine internals. Java achieves this through **abstract classes** and **interfaces**.

---

## Abstract Classes

A class that **can't be instantiated** and may contain abstract methods (no body — child must implement).

```java
abstract class Shape {
    String color;

    // Abstract method — no body, child MUST implement
    abstract double area();

    // Concrete method — has body, child inherits as-is (can override)
    void display() {
        System.out.println("Color: " + color + ", Area: " + area());
    }
}

class Circle extends Shape {
    double radius;

    Circle(double radius) { this.radius = radius; }

    @Override
    double area() { return Math.PI * radius * radius; }   // must implement
}

// Shape s = new Shape();    // ❌ can't instantiate abstract class
Shape s = new Circle(5);     // ✅ can use as reference type
s.area();                     // calls Circle's implementation
s.display();                  // inherited concrete method
```

### Rules

- Can have **both** abstract and concrete methods
- Can have constructors (called via `super()` from child)
- Can have fields (any access modifier, any type — not just constants)
- Can have `static` methods
- **Cannot be instantiated** — but can be used as reference type
- Child **must implement all** abstract methods — or be abstract itself
- Can extend only ONE abstract class (single inheritance)

### Abstract Methods

```java
abstract void process();          // ✅ instance abstract method
// abstract static void util();   // ❌ static abstract doesn't make sense
                                   //    static = class-level, abstract = needs override in child
```

- Must be in an abstract class (or interface)
- Cannot be `static`, `final`, or `private` (all prevent overriding)

### Class vs Abstract Class

||Class|Abstract Class|
|---|---|---|
|Instantiate|Yes|No|
|Abstract methods|No|Yes (can also have concrete)|
|Constructor|Yes|Yes (called by child via super)|
|Fields|Any|Any|
|Purpose|Complete implementation|Partial implementation — template for children|

## Interfaces

A **contract** — defines what a class must do, not how. Purely about capability.

```java
interface Flyable {
    void fly();       // public abstract by default
}

interface Swimmable {
    void swim();
}

// A class can implement multiple interfaces
class Duck implements Flyable, Swimmable {
    @Override
    public void fly() { System.out.println("Duck flying"); }

    @Override
    public void swim() { System.out.println("Duck swimming"); }
}
```

### Rules

- Methods are `public abstract` by default (no need to write it)
- Variables are `public static final` by default (constants only)
- **No constructors** — can't instantiate
- **No instance fields** — only constants
- A class can implement **multiple** interfaces
- An interface can extend **multiple** interfaces

```java
interface A { void methodA(); }
interface B { void methodB(); }
interface C extends A, B { }       // ✅ interface extends multiple interfaces

class MyClass implements C {
    public void methodA() { }      // must implement all
    public void methodB() { }
}
```

### Interface Variables

```java
interface Constants {
    int MAX = 100;              // public static final by default
    // int x;                   // ❌ must be initialized (it's final)
    // private int y = 5;       // ❌ must be public
}

// Accessed via interface name
Constants.MAX;
```

### Can an Interface Have a Constructor?

**No.** Interfaces can't be instantiated, so constructors don't make sense. All initialization is through constants (`static final`).

## Concrete Methods in Interfaces (Java 8+)

### Default Methods (Java 8)

Methods with a body in interfaces — provides backward compatibility (add methods without breaking existing implementations).

```java
interface Sortable {
    void sort();

    default void bubbleSort() {
        System.out.println("Default bubble sort");
    }
}

class MyList implements Sortable {
    public void sort() { /* must implement */ }
    // bubbleSort() is inherited — can use or override
}
```

**Why added:** When `stream()` was added to `Collection` in Java 8, all existing implementations would break. Default methods solved this — `Collection` got a default `stream()` method.

### Static Methods (Java 8)

```java
interface Validator {
    boolean validate(String input);

    static boolean isNotEmpty(String s) {
        return s != null && !s.isEmpty();
    }
}

// Called on interface — NOT inherited by implementing classes
Validator.isNotEmpty("hello");    // ✅
// myValidator.isNotEmpty("x");   // ❌ not inherited
```

### Private Methods (Java 9+)

Helper methods shared between default methods — not visible to implementing classes.

```java
interface Logger {
    default void logInfo(String msg) { log("INFO", msg); }
    default void logError(String msg) { log("ERROR", msg); }

    private void log(String level, String msg) {    // shared helper
        System.out.println(level + ": " + msg);
    }
}
```

## Default Method Conflict (Diamond Problem with Interfaces)

```java
interface A { default void show() { System.out.println("A"); } }
interface B { default void show() { System.out.println("B"); } }

class C implements A, B {
    // ❌ COMPILE ERROR if not resolved — must override
    @Override
    public void show() {
        A.super.show();     // explicitly choose A's version
        // or B.super.show();
        // or write own logic
    }
}
```

**Rules:**

- Class method always wins over interface default method
- More specific interface wins (child interface over parent interface)
- If ambiguous → must override and resolve explicitly

```
Method resolution priority:
1. Class method (highest)
2. Most specific interface (child > parent interface)
3. Explicit override required (if conflict remains)
```

```java
class Parent {
    void show() { System.out.println("Parent"); }
}
interface I {
    default void show() { System.out.println("Interface"); }
}

class Child extends Parent implements I {
    // No conflict — Parent's show() wins (class > interface)
}
new Child().show();    // "Parent"
```

## Multiple Inheritance with Interfaces

```java
interface Printable { void print(); }
interface Scannable { void scan(); }
interface Faxable { void fax(); }

// One class, multiple capabilities
class AllInOnePrinter implements Printable, Scannable, Faxable {
    public void print() { }
    public void scan() { }
    public void fax() { }
}

// Interface extending multiple interfaces
interface MultiFunctionDevice extends Printable, Scannable, Faxable { }
```

## Marker Interfaces

Interfaces with **no methods** — just a tag/signal to JVM or frameworks.

|Marker Interface|Purpose|
|---|---|
|`Serializable`|Marks class as serializable (JVM checks before serialization)|
|`Cloneable`|Marks class as cloneable (`clone()` throws exception without it)|
|`Remote`|Marks class for RMI|

```java
class Employee implements Serializable {
    // No methods to implement — just a signal
}
```

**Modern alternative:** Annotations (`@Entity`, `@Component`) largely replace marker interfaces.

## Class vs Abstract Class vs Interface

||Class|Abstract Class|Interface|
|---|---|---|---|
|Instantiate|✅|❌|❌|
|Methods|Concrete only|Abstract + Concrete|Abstract + Default + Static + Private|
|Fields|Any|Any|`public static final` only|
|Constructor|✅|✅|❌|
|Inheritance|Single (`extends`)|Single (`extends`)|Multiple (`implements`)|
|Access modifiers|Any|Any|`public` (methods), `public static final` (fields)|
|Purpose|Full implementation|Template with partial implementation|Contract / capability|
|Use when|Complete, ready-to-use class|IS-A with shared state/code|CAN-DO capability|

**Punchline:** Abstract class for IS-A with shared state. Interface for CAN-DO capability.

## Abstract Class vs Interface — When to Use

|Scenario|Use|
|---|---|
|Share code among related classes|Abstract class|
|Need constructor or instance fields|Abstract class|
|Define a contract for unrelated classes|Interface|
|Need multiple inheritance|Interface|
|Provide common default behavior|Interface with default methods|
|Template method pattern|Abstract class|

```java
// Abstract class — related classes, shared state
abstract class Vehicle {
    int speed;
    abstract void start();
    void stop() { speed = 0; }    // shared code
}

// Interface — unrelated classes, common capability
interface Loggable {
    default void log(String msg) { System.out.println(msg); }
}

// A class can use both
class Car extends Vehicle implements Loggable {
    void start() { speed = 60; log("Car started"); }
}
```

## Commonly Asked Gotchas

1. **Abstract class CAN have a constructor** — called via `super()` from child. Used to initialize shared fields.

2. **An abstract class can have zero abstract methods** — it's still abstract (can't instantiate). Useful when you want to prevent instantiation.

3. **Can't use `super` to call interface default method from outside:**

```java
// Only works inside implementing class
A.super.show();    // ✅ inside implementing class
// Can't do: myObj.A.super.show() from outside
```

4. **Abstract class can implement interface without providing implementations** — pushes responsibility to concrete child.

```java
interface I { void method(); }
abstract class A implements I { }      // ✅ no error — A is abstract
class B extends A {
    public void method() { }           // B must implement
}
```

5. **Functional interface = interface with exactly ONE abstract method** — enables lambdas. Multiple default/static methods don't count.

6. **Interface can't extend a class** — but a class can extend another class AND implement interfaces.

```java
// interface I extends MyClass { }     // ❌
class C extends MyClass implements I { }  // ✅
```

7. **Since Java 8, abstract class and interface are closer** — both can have concrete methods. Key remaining differences: interfaces cannot hold instance state or constructors. This is the real differentiator, not "methods with body."

## Quick Revision Cheat Sheet

|Concept|Remember|
|---|---|
|Abstract class|Can't instantiate. Mix of abstract + concrete. Has constructors, fields|
|Interface|Contract. No state (only constants). Multiple inheritance|
|Abstract method|No body. Must be overridden. Can't be static/final/private|
|Default method|Java 8+. Interface method with body. Backward compatibility|
|Static method|Java 8+. Called on interface, NOT inherited|
|Private method|Java 9+. Helper for default methods. Not visible to implementors|
|Marker interface|No methods — just a tag (Serializable, Cloneable)|
|Diamond conflict|Must resolve explicitly. Class wins over interface|
|When abstract class|IS-A + shared state/code + template pattern|
|When interface|CAN-DO capability + multiple inheritance + API design|
|Interface variables|Always `public static final` — constants only|
|Functional interface|One abstract method → enables lambda|
