## What is the Object Class

`java.lang.Object` is the **root of every class hierarchy** — every Java class implicitly extends `Object` (directly or indirectly). This is why methods like `toString()`, `equals()`, and `hashCode()` are available on every object.

```java
class Employee { }
// Compiler sees: class Employee extends Object { }

Employee e = new Employee();
e.toString();    // inherited from Object
e.equals(other); // inherited from Object
e.hashCode();    // inherited from Object
```

The three most-overridden methods are **`toString()`, `equals()`, and `hashCode()`** — properly overriding them is essential for debugging, collections, and general correctness.

### All Object Methods

|Method|Purpose|Full coverage|
|---|---|---|
|`toString()`, `equals()`, `hashCode()`|String repr, equality, hashing|This chapter|
|`getClass()`|Returns runtime class (`final` — can't override)|[18. Keywords](<18. Important Keywords (Deep Dive).md>)|
|`clone()`|Shallow copy (requires `Cloneable` marker)|[13. Encapsulation](<13. Encapsulation & Access Control.md>)|
|`wait()`, `notify()`, `notifyAll()`|Thread coordination on object's monitor|[26. Multithreading](<26. Multithreading & Concurrency.md>)|
|`finalize()`|Called before GC — **deprecated since Java 9, slated for removal**|—|

## `toString()`

Returns a String representation of the object. Default implementation: `ClassName@hexHashCode` — rarely useful.

```java
class Employee {
    String name;
    int salary;
}

Employee e = new Employee();
System.out.println(e);    // "Employee@1a2b3c" — useless
```

### Override for meaningful output

```java
class Employee {
    String name;
    int salary;

    @Override
    public String toString() {
        return "Employee{name='" + name + "', salary=" + salary + "}";
    }
}

System.out.println(e);    // "Employee{name='Alice', salary=50000}"
```

**When `toString()` is called:**

- Directly: `obj.toString()`
- Implicitly by `System.out.println(obj)`, string concatenation `"User: " + obj`, logging frameworks
- By debuggers and IDEs (watch windows)

**Use `Objects.toString(obj, defaultIfNull)`** for null-safe printing:

```java
Objects.toString(null, "N/A");    // "N/A"
Objects.toString(e, "N/A");       // uses e.toString()
```

## `equals()`

Compares objects for **logical equality**. Default implementation: reference equality (same as `==`).

### `==` vs `.equals()`

|Operator|Compares|Works on|
|---|---|---|
|`==`|References (same object in memory?)|Primitives use value; objects use reference|
|`.equals()`|Content (logical equality) — if overridden|Objects only|

```java
String a = new String("hello");
String b = new String("hello");

a == b         // false — different objects
a.equals(b)    // true — same content (String overrides equals)

Employee e1 = new Employee("Alice");
Employee e2 = new Employee("Alice");
e1 == e2       // false
e1.equals(e2)  // false — Employee hasn't overridden equals, falls back to reference equality
```

### Override `equals()` correctly

```java
class Employee {
    String name;
    int salary;

    @Override
    public boolean equals(Object o) {
        if (this == o) return true;                        // same reference → equal
        if (!(o instanceof Employee e)) return false;      // type check + pattern var
        return salary == e.salary && Objects.equals(name, e.name);
    }
}
```

### `equals()` contract

For any non-null `x`, `y`, `z`:

- **Reflexive** — `x.equals(x)` is true
- **Symmetric** — `x.equals(y)` ↔ `y.equals(x)`
- **Transitive** — if `x.equals(y)` and `y.equals(z)` then `x.equals(z)`
- **Consistent** — repeated calls return the same result (if the objects don't change)
- **null-safe** — `x.equals(null)` is always false

Violating these breaks collections that rely on `equals` (`HashSet`, `HashMap`, `List.contains`, etc.).

### Use `Objects.equals()` for null-safe comparison

```java
Objects.equals(a, b)    // handles null on either side — no NPE
// Internally: (a == b) || (a != null && a.equals(b))
```

## `hashCode()`

Returns an `int` hash used by hash-based collections (`HashMap`, `HashSet`, `Hashtable`).

### Default implementation

- Returns an int derived from the object's memory address (not the content)
- Two logically-equal objects get **different** hash codes by default → breaks `HashMap`/`HashSet`

### Override `hashCode()` alongside `equals()`

```java
class Employee {
    String name;
    int salary;

    @Override
    public int hashCode() {
        return Objects.hash(name, salary);    // Java 7+ — clean and correct
    }
}
```

`Objects.hash(...)` combines the hashes of its arguments using a standard algorithm — equivalent to writing the combining code manually.

## Contract Between `equals()` and `hashCode()`

**The rule:** if `a.equals(b)` is true, then `a.hashCode() == b.hashCode()` MUST be true.

The reverse is not required — two objects with the same hash can be unequal (that's a hash collision, handled by the collection).

### Why `HashSet`/`HashMap` Depend on It

Hash-based collections look up entries in two steps:

1. Compute `hashCode()` → find the bucket
2. Walk the bucket, calling `equals()` on each entry to find a match

If `hashCode()` is inconsistent with `equals()`, step 1 sends the lookup to the wrong bucket — the entry is "lost" even though it's in the collection.

```java
class Employee {
    String name;
    Employee(String name) { this.name = name; }

    @Override
    public boolean equals(Object o) {
        return o instanceof Employee e && name.equals(e.name);
    }
    // ❌ forgot to override hashCode()
}

Set<Employee> set = new HashSet<>();
set.add(new Employee("Alice"));
set.contains(new Employee("Alice"));   // false! — different hashCode → different bucket
```

### What Breaks if Not Overridden

- `HashSet.contains()` / `HashMap.get()` return wrong results (usually "not found" even when the object is there)
- `HashMap.remove()` fails silently
- `List.contains()` / `indexOf()` return wrong results (uses `equals` only, but inconsistency still bites when lists are compared to sets)
- De-duplication (`new HashSet<>(list)`) doesn't actually de-duplicate

### Golden rule

**Always override `hashCode()` when you override `equals()`** — and use the SAME fields in both. IDE shortcuts and Lombok `@EqualsAndHashCode` do this automatically.

## Commonly Asked Gotchas

1. **Overriding `equals()` without `hashCode()` is a classic bug** — breaks hash-based collections silently. IDE usually warns; pay attention.

2. **`==` vs `.equals()` on Strings from different sources** — `String` literals go to the pool (`==` works accidentally), but `new String("x")` creates a heap object (`==` fails). Always use `.equals()` for Strings.

3. **Integer cache makes `==` on `Integer` *accidentally* work for small values** — `-128..127` are cached and shared. `Integer a = 127, b = 127; a == b` is true; `Integer a = 200, b = 200; a == b` is false. Never use `==` on wrapper objects.

4. **`equals()` must accept `Object`, not your class type** — `public boolean equals(Employee e)` is NOT overriding, it's overloading. Without `@Override`, this slips past silently.

5. **Immutable objects make `hashCode()` safe as map keys** — if a field used in `hashCode()` changes after the object is inserted into a `HashMap`, it gets "lost". Prefer immutable keys.

6. **Default `toString()` is rarely useful** — `Object@1a2b3c`. Always override for classes you'll log or debug.

7. **`Objects.equals(a, b)` and `Objects.hash(...)`** — `java.util.Objects` utility methods handle null safely and produce correct hashes. Prefer them over manual implementations.

|Concept|Remember|
|---|---|
|`Object`|Root of all classes. Every class implicitly extends it|

|`==`|Reference equality for objects (value equality for primitives)|
|`.equals()`|Content/logical equality. Default = reference equality (like `==`)|
|`equals()` contract|Reflexive, symmetric, transitive, consistent, null-safe|
|`equals()` signature|Must take `Object` param — not your class type|
|`hashCode()`|Used by `HashMap`/`HashSet`. Default = memory-based|
|Golden rule|Override `hashCode()` whenever you override `equals()` — same fields|
|Contract|`a.equals(b)` ⇒ `a.hashCode() == b.hashCode()` (reverse NOT required)|
|Breaks if missing|`HashSet.contains()`, `HashMap.get/remove()` fail silently|
|Null-safe helpers|`Objects.equals(a, b)`, `Objects.hash(...)`, `Objects.toString(o, default)`|
