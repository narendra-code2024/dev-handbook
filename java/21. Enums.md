## What & Why

An enum is a **fixed set of named constants** with full class capabilities — fields, methods, constructors, interface implementation.

Use enums for any value that has a **known, finite set of valid options**:

- **States** — `OrderStatus`, `PaymentStatus`, `JobStatus`
- **Roles / permissions** — `UserRole`, `AccessLevel`
- **Feature flags, log levels, config**
- **Strategy pattern** per constant
- **Singletons** (Effective Java pattern)

```java
enum OrderStatus { PENDING, PAID, SHIPPED, DELIVERED, CANCELLED }

OrderStatus status = OrderStatus.PAID;
if (status == OrderStatus.PAID) { /* ... */ }   // == is safe on enum constants
```

**Why not just `public static final` constants?** Enums are type-safe (compiler catches wrong values), introspectable (`values()`, `name()`, `ordinal()`), usable in `switch`, can carry behavior, enable `EnumSet`/`EnumMap` optimizations.

## Declaration & Basics

```java
enum Color { RED, GREEN, BLUE }

Color c = Color.RED;
System.out.println(c);            // RED
System.out.println(c.name());     // RED
System.out.println(c.ordinal());  // 0 — position in declaration order
System.out.println(c == Color.RED); // true — enum constants are singletons
```

- Each constant is a **singleton instance** — guaranteed unique reference
- `==` comparison is safe and idiomatic (unlike general objects, where `equals()` is required for content)
- Enums implicitly extend `java.lang.Enum` — you can't `extends` another class, but you can implement interfaces

## Built-in Methods

### Static (generated by the compiler)

```java
Color.values();          // returns Color[] — all constants in declaration order
Color.valueOf("RED");    // returns Color.RED — throws IllegalArgumentException if no match
```

### Instance (inherited from `java.lang.Enum`)

```java
Color c = Color.GREEN;
c.name();            // "GREEN" — declared constant name
c.ordinal();         // 1 — position in declaration (don't rely on this!)
c.toString();        // "GREEN" — same as name() by default; override to customize
c.compareTo(Color.RED); // ordinal difference: 1 - 0 = 1
```

> **`ordinal()` pitfall**: don't use it for business logic or persist it. Reordering constants or inserting a new one silently shifts `ordinal()` values.

## Enums with Fields, Constructors, and Methods

Enums can carry per-constant state and behavior.

```java
enum Planet {
    MERCURY(3.303e+23, 2.4397e6),
    VENUS  (4.869e+24, 6.0518e6),
    EARTH  (5.976e+24, 6.37814e6);

    private final double mass;     // instance fields
    private final double radius;

    // Constructor — always private (explicitly or implicitly)
    Planet(double mass, double radius) {
        this.mass = mass;
        this.radius = radius;
    }

    public double getMass()   { return mass; }
    public double getRadius() { return radius; }

    public double surfaceGravity() {
        return 6.67300E-11 * mass / (radius * radius);
    }
}

Planet.EARTH.surfaceGravity();   // 9.80...
```

- **Constructors are private** by default — you can't call `new Planet(...)` externally
- All constants are declared at the top; each invokes the constructor
- Fields are conventionally `final` (immutability) but not required

## Abstract Methods per Constant (Strategy Pattern)

Each constant can override a method — encoding *different behavior per constant* without `switch`.

```java
enum Operation {
    ADD      { public int apply(int a, int b) { return a + b; } },
    SUBTRACT { public int apply(int a, int b) { return a - b; } },
    MULTIPLY { public int apply(int a, int b) { return a * b; } },
    DIVIDE   { public int apply(int a, int b) { return a / b; } };

    public abstract int apply(int a, int b);
}

Operation.ADD.apply(2, 3);       // 5
Operation.MULTIPLY.apply(4, 5);  // 20
```

The classic enum strategy pattern — each constant carries its own implementation, no giant `switch` needed.

## Implementing Interfaces

Enums can implement interfaces, enabling polymorphism.

```java
interface Describable {
    String describe();
}

enum Status implements Describable {
    ACTIVE   { public String describe() { return "running"; } },
    INACTIVE { public String describe() { return "stopped"; } },
    ERROR    { public String describe() { return "failed"; } };
}

Describable d = Status.ACTIVE;
d.describe();   // "running"
```

Useful when an enum should fit into an existing type hierarchy.

## Enums in `switch`

Switch on enums is clean — constants don't need qualification.

```java
switch (status) {
    case PENDING   -> notify("waiting");
    case PAID      -> shipOrder();
    case SHIPPED   -> track();
    case DELIVERED -> thankCustomer();
    case CANCELLED -> refund();
}
```

**Pattern matching (Java 21+)** — exhaustive switch expressions:

```java
String label = switch (status) {
    case PENDING, PAID -> "In Progress";
    case SHIPPED       -> "On the way";
    case DELIVERED     -> "Done";
    case CANCELLED     -> "Refunded";
};
```

> **Gotcha**: adding a new enum constant doesn't error existing switches that lack `default`. Use exhaustive switch expressions (Java 14+) or always include `default`.

## `EnumSet` and `EnumMap`

Specialized high-performance collections designed for enum keys. Internally use bit-vectors / arrays — **much faster** than `HashSet<Enum>` or `HashMap<Enum, V>`.

### `EnumSet`

```java
EnumSet<OrderStatus> active = EnumSet.of(OrderStatus.PENDING, OrderStatus.PAID);
EnumSet<OrderStatus> terminal = EnumSet.of(OrderStatus.DELIVERED, OrderStatus.CANCELLED);
EnumSet<OrderStatus> all = EnumSet.allOf(OrderStatus.class);
EnumSet<OrderStatus> none = EnumSet.noneOf(OrderStatus.class);
EnumSet<OrderStatus> range = EnumSet.range(OrderStatus.PENDING, OrderStatus.SHIPPED);

active.contains(OrderStatus.PAID);   // O(1) bitwise check — blazing fast
```

### `EnumMap`

```java
EnumMap<OrderStatus, String> labels = new EnumMap<>(OrderStatus.class);
labels.put(OrderStatus.PENDING, "Waiting for payment");
labels.put(OrderStatus.PAID,    "Payment received");
```

Use `EnumSet`/`EnumMap` over `HashSet`/`HashMap` when keys are enums — smaller footprint, faster operations, preserves declaration order.

## Enum as Singleton (Effective Java pattern)

From *Effective Java* (Joshua Bloch) — the **safest, most concise way to implement Singleton**:

```java
enum Database {
    INSTANCE;

    private Connection conn;

    Database() {
        this.conn = connect();   // initialized once, on class load
    }

    public void query(String sql) {
        // use conn...
    }
}

// Usage
Database.INSTANCE.query("SELECT * FROM users");
```

Why enum singleton beats the traditional approach:

- **Serialization-safe** — deserialization returns the same instance
- **Reflection-safe** — can't invoke enum constructors via reflection
- **Thread-safe** by default (JVM guarantees via class loading)
- **Concise** — 2 lines vs 10+ for double-checked-locking singleton

## Real-World Patterns

### State — `OrderStatus` with transitions

```java
enum OrderStatus {
    PENDING, PAID, SHIPPED, DELIVERED, CANCELLED;

    public boolean isTerminal() {
        return this == DELIVERED || this == CANCELLED;
    }

    public boolean canTransitionTo(OrderStatus next) {
        return switch (this) {
            case PENDING   -> next == PAID || next == CANCELLED;
            case PAID      -> next == SHIPPED || next == CANCELLED;
            case SHIPPED   -> next == DELIVERED;
            case DELIVERED, CANCELLED -> false;
        };
    }
}
```

### Roles / Permissions

```java
enum UserRole {
    ADMIN  (Set.of("read", "write", "delete", "manage")),
    EDITOR (Set.of("read", "write")),
    VIEWER (Set.of("read"));

    private final Set<String> permissions;
    UserRole(Set<String> permissions) { this.permissions = permissions; }

    public boolean can(String action) {
        return permissions.contains(action);
    }
}

if (currentUser.getRole().can("delete")) { /* ... */ }
```

### Feature Flag / Config

```java
enum Feature {
    DARK_MODE    (true),
    BETA_CHECKOUT(false),
    NEW_DASHBOARD(true);

    private final boolean enabled;
    Feature(boolean enabled) { this.enabled = enabled; }
    public boolean isEnabled() { return enabled; }
}

if (Feature.DARK_MODE.isEnabled()) { /* ... */ }
```

### Log Level

```java
enum LogLevel {
    DEBUG(0), INFO(1), WARN(2), ERROR(3), FATAL(4);

    private final int severity;
    LogLevel(int severity) { this.severity = severity; }

    public boolean isAtLeast(LogLevel other) {
        return this.severity >= other.severity;
    }
}
```

## Commonly Asked Gotchas

1. **`ordinal()` is fragile** — it's the declaration position. Reordering or inserting constants silently shifts values. Don't persist `ordinal()` to DB; persist `name()` instead.

2. **Adding a new constant doesn't error old switches** (without `default` or exhaustive expressions) — silently falls through. Use exhaustive switch expressions (Java 14+) or always include `default`.

3. **Enum implements `Comparable`** — natural order is `ordinal()`. Use a `Comparator` for custom order.

4. **Enum constants load lazily** — static initialization runs once, on first use of the class.

5. **Serialization** — enums are serialized by `name()`, not by fields. Renaming a constant breaks old serialized data.

6. **Not for dynamic sets** — enums are for **compile-time-known** values. For categories that come from a DB or config, use a different structure.

7. **`EnumSet.of()` needs at least one arg** (overloaded up to 5 varargs). For empty sets: `EnumSet.noneOf(EnumType.class)`.

## Quick Revision Cheat Sheet

|Concept|Remember|
|---|---|
|Declaration|`enum X { A, B, C }`|
|Built-in methods|`values()`, `valueOf()`, `name()`, `ordinal()`|
|Per-constant state|Fields + private constructor + constants with args|
|Per-constant behavior|Abstract method + override in each constant (strategy pattern)|
|Comparison|`==` safe and idiomatic|
|Switch|No qualification needed — `case PAID`, not `case OrderStatus.PAID`|
|`EnumSet`|Fast bit-vector set; prefer over `HashSet<Enum>`|
|`EnumMap`|Fast array-backed map; prefer over `HashMap<Enum, V>`|
|Singleton|`enum Singleton { INSTANCE; }` — safest pattern|
|`ordinal()` gotcha|Don't persist — fragile to reordering|
|Switch gotcha|Adding constants doesn't error — use exhaustive switch or `default`|
|Can extend?|No classes, only interfaces|
