## What & Why

Serialization is converting a **Java object into a byte stream** so it can be saved to a file, sent over a network, or stored in a database. Deserialization is the reverse — byte stream back to object.

```
Object → byte stream → file / network / DB    (Serialization)
file / network / DB → byte stream → Object    (Deserialization)
```

**Real-world use cases:**

- Saving session state (web apps)
- Sending objects over network (RMI, sockets)
- Caching objects (Redis, Memcached)
- Deep cloning objects (only if entire object graph is Serializable)
- Storing objects in files

## How to Serialize

A class must implement `java.io.Serializable` — a **marker interface** (no methods, just a signal to JVM that this class can be serialized).

```java
import java.io.*;

class Employee implements Serializable {
    private static final long serialVersionUID = 1L;  // version control
    String name;
    int salary;
    transient String password;  // won't be serialized

    public Employee(String name, int salary, String password) {
        this.name = name;
        this.salary = salary;
        this.password = password;
    }
}
```

### Serialize (write object to file)

```java
Employee emp = new Employee("Alice", 50000, "secret123");

try (ObjectOutputStream oos = new ObjectOutputStream(new FileOutputStream("emp.ser"))) {
    oos.writeObject(emp);
}
```

### Deserialize (read object from file)

```java
try (ObjectInputStream ois = new ObjectInputStream(new FileInputStream("emp.ser"))) {
    Employee emp = (Employee) ois.readObject();
    System.out.println(emp.name);      // "Alice"
    System.out.println(emp.salary);    // 50000
    System.out.println(emp.password);  // null (transient — not serialized)
}
```

## serialVersionUID

A version number for the serialized class. JVM uses it to verify that the sender and receiver of a serialized object have the same class version.

```java
private static final long serialVersionUID = 1L;
```

**What happens without it?**

- JVM auto-generates one based on class structure (fields, methods, etc.)
- If you change the class (add/remove fields), the auto-generated UID changes
- Deserialization fails with `InvalidClassException` — even if the change was minor

**Best practice:** Always declare `serialVersionUID` explicitly. This gives you control over compatibility.

```java
// Class v1 — serialized to file
class Employee implements Serializable {
    private static final long serialVersionUID = 1L;
    String name;
}

// Class v2 — added field, same UID → deserialization still works
class Employee implements Serializable {
    private static final long serialVersionUID = 1L;
    String name;
    int age;    // new field — defaults to 0 on deserialization of old objects
}
```

## transient Keyword

Marks a field to be **excluded from serialization**. On deserialization, transient fields get default values (`null`, `0`, `false`).

```java
class User implements Serializable {
    String username;
    transient String password;     // not serialized — security
    transient int sessionId;       // not serialized — temporary data
    transient Logger logger;       // not serialized — non-serializable object
}

// After deserialization:
// username = "Alice"
// password = null
// sessionId = 0
// logger = null
```

**When to use transient:**

- Sensitive data (passwords, tokens)
- Derived/calculated fields (can be recomputed)
- Non-serializable objects (Logger, Thread, DB connections)
- Temporary state (session data, caches)

## static Fields and Serialization

`static` fields are **NOT serialized** — they belong to the class, not the object.

```java
class Config implements Serializable {
    static String appName = "MyApp";  // NOT serialized
    String env;                        // serialized
}

// After deserialization, static fields have whatever value
// the class currently has — NOT the value at serialization time
```

## Inheritance and Serialization

### Parent is Serializable → child is automatically Serializable

```java
class Parent implements Serializable {
    int x = 10;
}

class Child extends Parent {    // also Serializable (inherited)
    int y = 20;
}
// Both x and y are serialized
```

### Parent is NOT Serializable → child implements Serializable

```java
class Parent {                  // NOT Serializable
    int x = 10;
    Parent() { x = 10; }       // must have no-arg constructor
}

class Child extends Parent implements Serializable {
    int y = 20;
}

// On deserialization:
// y = 20 (restored from byte stream)
// x = 10 (NOT restored — Parent's no-arg constructor is called)
```

**Key rule:** If the first non-serializable parent does NOT have a no-arg constructor, deserialization fails with `InvalidClassException`.

## Custom Serialization (writeObject / readObject)

Override default behavior for special handling (encryption, validation, reinitializing transient fields).

```java
class Account implements Serializable {
    private static final long serialVersionUID = 1L;
    String name;
    transient String password;

    // Custom serialization
    private void writeObject(ObjectOutputStream oos) throws IOException {
        oos.defaultWriteObject();                    // serialize normal fields
        oos.writeObject(encrypt(password));           // manually serialize transient field (encrypted)
    }

    // Custom deserialization
    private void readObject(ObjectInputStream ois) throws IOException, ClassNotFoundException {
        ois.defaultReadObject();                     // deserialize normal fields
        this.password = decrypt((String) ois.readObject());  // manually restore transient field
    }
}
```

**Use cases:**

- Encrypting sensitive fields before serialization
- Validating data during deserialization
- Reinitializing transient fields (DB connections, loggers)
- Maintaining backward compatibility

## Externalizable Interface

Full control over serialization — you implement everything yourself. Must provide no-arg constructor.

```java
class Employee implements Externalizable {
    String name;
    int salary;

    public Employee() { }   // mandatory no-arg constructor

    @Override
    public void writeExternal(ObjectOutput out) throws IOException {
        out.writeUTF(name);
        out.writeInt(salary);
    }

    @Override
    public void readExternal(ObjectInput in) throws IOException {
        name = in.readUTF();
        salary = in.readInt();
    }
}
```

### Serializable vs Externalizable

||Serializable|Externalizable|
|---|---|---|
|Type|Marker interface (no methods)|Has `writeExternal()` / `readExternal()`|
|Default behavior|JVM handles everything|You handle everything|
|Performance|Slower (reflection-based)|Potentially faster (you control what's written, but depends on implementation)|
|No-arg constructor|Not required|Required|
|transient|Respected|Irrelevant (you control fields)|
|Use case|Most cases (simple)|Performance-critical / full control|

## Serialization and Singleton

Serialization can **break Singleton** — deserializing creates a new instance.

```java
// Problem
Singleton obj1 = Singleton.getInstance();
// serialize obj1 → deserialize → obj2 is a NEW instance
// obj1 != obj2 — singleton broken!

// Fix: implement readResolve()
class Singleton implements Serializable {
    private static final Singleton INSTANCE = new Singleton();
    private Singleton() { }

    public static Singleton getInstance() { return INSTANCE; }

    // This is called after deserialization — return existing instance
    private Object readResolve() { return INSTANCE; }
}
```

**Best alternative:** Use `enum` Singleton — inherently safe from serialization attacks.

```java
enum Singleton {
    INSTANCE;
    // enum serialization is handled by JVM — always returns same instance
}
```

## Modern Alternatives to Java Serialization

Java's built-in serialization has known issues (security vulnerabilities, performance, verbosity). Modern alternatives are preferred in production:

|Alternative|Format|Use case|
|---|---|---|
|**JSON** (Jackson, Gson)|Text|REST APIs, config files, human-readable|
|**XML** (JAXB)|Text|Legacy systems, SOAP|
|**Protocol Buffers** (Google)|Binary|High-performance, cross-language|
|**Avro** (Apache)|Binary|Big data, Kafka|
|**Kryo**|Binary|Fast JVM serialization (Spark, Flink)|

```java
// Jackson — most common in Spring Boot
ObjectMapper mapper = new ObjectMapper();

// Object → JSON string
String json = mapper.writeValueAsString(employee);

// JSON string → Object
Employee emp = mapper.readValue(json, Employee.class);

// Object → JSON file
mapper.writeValue(new File("emp.json"), employee);
```

**In Spring Boot:** Jackson is auto-configured. `@RestController` automatically serializes/deserializes objects to/from JSON using Jackson.

## Commonly Asked Gotchas

1. **`NotSerializableException`** — thrown if any field in the object graph is not Serializable and not transient. All referenced objects must be Serializable too.

```java
class Employee implements Serializable {
    Address address;   // Address must also implement Serializable!
}
```

2. **Constructor is NOT called during deserialization** — for Serializable classes. Object is created via reflection without calling any constructor. But constructors of the first non-serializable superclass ARE called. For Externalizable, no-arg constructor IS called.
    
3. **`transient` + `static` — both excluded** but for different reasons. `transient` = intentionally excluded. `static` = belongs to class, not object.

4. **Serialization is recursive** — if Employee has Address, and Address has City, all must be Serializable. The entire object graph is serialized.

5. **`serialVersionUID` should be `private static final long`** — any other signature won't be recognized by the serialization mechanism.

6. **Deserialization of removed fields** — if a field was serialized but the class no longer has it, it's silently ignored (if serialVersionUID matches). New fields get default values.

7. **`final` fields are serialized and restored via reflection** — during deserialization, even `final` fields are set through reflection, bypassing the normal "can't reassign final" rule.

8. **Serialization preserves object identity** — if two fields reference the same object, after deserialization they still reference the same object (not two copies).

## Quick Revision Cheat Sheet

|Concept|Remember|
|---|---|
|Serializable|Marker interface, no methods. JVM handles serialization|
|serialVersionUID|Always declare. Controls version compatibility|
|transient|Excluded from serialization. Gets default value on deserialization|
|static|Not serialized — belongs to class, not object|
|Externalizable|Full control, must implement read/write, needs no-arg constructor|
|writeObject/readObject|Custom serialization hooks for Serializable classes|
|readResolve()|Prevents Singleton from breaking during deserialization|
|Inheritance|Non-serializable parent must have no-arg constructor|
|Constructor|NOT called for Serializable (except first non-serializable parent's no-arg). Called for Externalizable|
|Object graph|Entire graph must be Serializable|
|Security|Never deserialize untrusted data — gadget chain attacks. Prefer JSON/Protobuf|
|Jackson|Standard in Spring Boot. Auto-configured for JSON serialization|
