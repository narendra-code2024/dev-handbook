## Beans & IoC Container

- **Bean** — a Java object whose creation, configuration, and lifecycle are managed by Spring's **IoC container** (IoC = **Inversion of Control** — the framework creates and wires your objects instead of your code doing it)
- **IoC container** — `ApplicationContext` is the main implementation; `BeanFactory` is a lightweight parent interface, rarely used directly

---

## How to Create Beans

**1. Using `@Component`** (and stereotype annotations: `@Service`, `@Repository`, `@Controller`) — detected via component scanning:

```java
@Service
public class GreetingService {
    public String greet(String name) {
        return "Hello, " + name;
    }
}
```

**2. Using `@Bean` in a `@Configuration` class** — explicit bean definition, useful for third-party or JDK classes you can't annotate:

```java
@Configuration
public class AppConfig {

    @Bean
    public Clock clock() {
        return Clock.systemUTC();   // java.time.Clock is a JDK class — can't put @Component on it
    }
}
```

### Using the beans

`SpringApplication.run()` returns the `ApplicationContext` — you can retrieve beans from it directly using `getBean()`. This is the most explicit way to see the container in action (DI is covered in [1.4](<1.4. Dependency Injection.md>)):

```java
@SpringBootApplication
public class DemoApplication {

    public static void main(String[] args) {
        ApplicationContext context = SpringApplication.run(DemoApplication.class, args);

        GreetingService greetingService = context.getBean(GreetingService.class);
        Clock clock = context.getBean(Clock.class);

        System.out.println(greetingService.greet("Alice"));   // "Hello, Alice"
        System.out.println("Started at " + clock.instant());
    }
}
```

`context.getBean()` has three forms:

- `getBean(Clock.class)` — by type (shown above); throws if multiple beans of that type exist
- `getBean("clock")` — by name; returns `Object` (needs casting)
- `getBean("clock", Clock.class)` — by name with type check; use when multiple beans share a type

The bean name for a `@Bean` method is the method name (so `public Clock clock()` → bean name `"clock"`).

## Bean Lifecycle

| Phase | What happens |
|---|---|
| **Instantiation** | `new MyBean()` runs — constructor executes. Fields are still `null` / default. |
| **Dependency Injection** | Spring populates `@Autowired` fields/setters. This is where `this.someService = injectedService` happens. |
| **Initialization** | `@PostConstruct` / `afterPropertiesSet()` / custom init-method run. Your setup code that requires dependencies to already be injected. |
| **Destruction** | `@PreDestroy` / `destroy()` / custom destroy-method run on container shutdown. |

> **Common confusion:** "Initialization" doesn't mean *assigning fields* — that happens during the DI phase. Initialization runs **your own setup code** (via `@PostConstruct`) **after** dependencies are in place.

```java
@Service
public class GreetingService {

    @Value("Hello")                   // ← set during DEPENDENCY INJECTION phase
    private String greetingPrefix;     //   (@Value injects a literal or a property value)

    private String greetMessage;

    @PostConstruct
    void init() {                     // ← runs during INITIALIZATION phase
        this.greetMessage = greetingPrefix + ", ";   // greetingPrefix is already set, use it here
    }

    public String greet(String name) {
        return greetMessage + name + "!";
    }
}
```

- `greetingPrefix` is set during **DI** — Spring fills the field before `init()` runs
- `greetMessage` is computed inside `@PostConstruct` — the **Initialization** phase, guaranteed to run *after* all fields have values

### Full lifecycle order

**Creation & initialization:**

1. **Constructor called** — the bean object is instantiated
2. **Dependencies injected** — fields and setter-injected `@Autowired` deps populated
3. **`Aware` interfaces** — container calls `BeanNameAware.setBeanName()`, `ApplicationContextAware.setApplicationContext()`, etc., so the bean can access container metadata (rarely needed in application code; mostly for framework-internal beans)
4. **`BeanPostProcessor.postProcessBeforeInitialization()`** — first hook; runs for every bean in the container, typically used by framework extensions to inspect or modify beans before init
5. **`@PostConstruct`** — your init callback (preferred)
6. **`InitializingBean.afterPropertiesSet()`** — legacy interface equivalent to `@PostConstruct`. Prefer the annotation.
7. **Custom init method** — declared via `@Bean(initMethod = "init")` or XML `init-method` (legacy)
8. **`BeanPostProcessor.postProcessAfterInitialization()`** — second hook; this is **where AOP proxies get wrapped around the bean** (so `@Transactional`, `@Async`, security proxies are applied here)

**Destruction (when the container shuts down):**

1. **`@PreDestroy`** — your cleanup callback (preferred)
2. **`DisposableBean.destroy()`** — legacy interface equivalent
3. **Custom destroy method** — declared via `@Bean(destroyMethod = "close")`

> **Which to use:** prefer `@PostConstruct` / `@PreDestroy` — standard JSR-250 annotations, not tied to Spring interfaces. `InitializingBean` / `DisposableBean` and custom init/destroy methods are legacy patterns you'll see in older codebases.

## Bean Scopes

A bean's scope controls its lifetime and how many instances exist. Spring has 6 built-in scopes; `singleton` is the default.

| Scope | Lifetime | Example use case |
|---|---|---|
| `singleton` | One instance per `ApplicationContext` (default) | Services, repositories, any stateless component |
| `prototype` | New instance every time it's requested | Form handlers, per-call calculators, builders with mutable state |
| `request` | One instance per HTTP request | Per-request logging context, request-scoped auth/user info |
| `session` | One instance per user HTTP session | Shopping cart, user preferences, logged-in user profile |
| `application` | One instance per `ServletContext` | App-wide config shared across all users |
| `websocket` | One instance per WebSocket session | Connected-user state, per-connection chat context |

### Declaring a non-default scope

```java
@Component
@Scope("prototype")
public class ShoppingCart { }

@Bean
@Scope("request")
public UserContext userContext() { return new UserContext(); }
```

### Gotcha — singleton holding a shorter-lived bean

**The problem:** DI happens **once**, at bean creation. A singleton is created once, so its dependencies are injected once. If it holds a `prototype` (or `request`/`session`) bean, that "shorter-lived" bean gets injected one time and never refreshed — defeating the shorter scope.

```java
@Component                          // singleton (default)
public class OrderService {
    @Autowired
    private ShoppingCart cart;       // injected ONCE when OrderService is built

    public void add(String item) { cart.add(item); }
}

@Component
@Scope("prototype")                  // "new instance every time" ❌ only in theory
public class ShoppingCart {
    private final List<String> items = new ArrayList<>();
    public void add(String i) { items.add(i); }
}
```

Every call to `orderService.add(...)` adds to the **same** cart — `ShoppingCart` was created once when `OrderService` was built. `prototype` is ignored because Spring never re-injects.

**The fix:** tell Spring to inject a **proxy** instead of the actual bean. The proxy looks up the correct scoped instance from the container on every method call.

```java
@Component
@Scope(value = "prototype", proxyMode = ScopedProxyMode.TARGET_CLASS)
public class ShoppingCart { /* ... */ }
```

Now `orderService.cart.add(...)` goes through the proxy, which asks the container for a fresh `ShoppingCart` (prototype) or the request-scoped one (for `request`) on each call. Same pattern for `request`/`session` scopes injected into singletons.

## Important Interview Questions

1. **What is a Spring bean?**
   Any Java object whose creation, wiring, and lifecycle are managed by Spring's IoC container (`ApplicationContext`).

2. **Difference between `@Component` and `@Bean`?**
   `@Component` (+ `@Service`, `@Repository`, `@Controller`) marks a class for auto-detection via component scanning. `@Bean` explicitly declares a bean inside a `@Configuration` method — used for third-party classes you can't annotate.

3. **What is the default scope of a bean? What other scopes exist?**
   Default is `singleton` (one instance per ApplicationContext). Others: `prototype`, `request`, `session`, `application`, `websocket`.

4. **`@PostConstruct` vs `@PreDestroy`?**
   `@PostConstruct` runs after dependencies are injected — use for init logic. `@PreDestroy` runs just before the container destroys the bean — use for cleanup.

5. **What's the role of `BeanPostProcessor`?**
   An extension point that lets you modify a bean after instantiation but before/after initialization (e.g., to apply proxies or validation). Runs between dependency injection and `@PostConstruct`.

6. **When would you use `prototype` over `singleton`?**
   When the bean holds per-request or per-call mutable state that shouldn't be shared across threads. Rare in typical service/repository layers.

## Quick Reference

|Concept|Remember|
|---|---|
| Bean | Object managed by Spring's IoC container |
| `ApplicationContext` | Main IoC container — creates, wires, destroys beans |
| `@Component` | Auto-detected via component scanning |
| `@Bean` | Explicit bean in `@Configuration` (for third-party classes) |
| Default scope | `singleton` |
| `@PostConstruct` | Runs after DI, before bean is used |
| `@PreDestroy` | Runs before destruction (cleanup) |
| `BeanPostProcessor` | Extension hook before/after initialization |
| Scope proxy | Inject proxy for shorter-lived bean into singleton — `@Scope(value = "...", proxyMode = ScopedProxyMode.TARGET_CLASS)` |
