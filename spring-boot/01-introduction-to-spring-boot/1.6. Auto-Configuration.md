## What Is It?

A mechanism that **auto-configures beans** based on:

- Dependencies present on the **classpath**
- **Properties** defined in configuration files
- **Conditional annotations** that evaluate at startup

Enabled by `@EnableAutoConfiguration`, which is included in `@SpringBootApplication`.

In traditional Spring, you had to define many beans manually and write XML or Java config for MVC, `DataSource`, JPA, etc. Spring Boot removes most of this setup and lets you focus on business logic.

---

## How It Works

1. **Classpath scanning** ŌĆö identifies project dependencies (from `pom.xml` or `build.gradle`). For what the classpath actually is, see [Java ŌĆö Classpath](<../../java/02. JDK, JRE & JVM.md#Classpath>).
2. **Locating configuration classes** ŌĆö `spring-boot-autoconfigure.jar` contains all auto-configuration logic. Spring Boot reads metadata from `META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports` (modern) or `META-INF/spring.factories` (legacy) via `@EnableAutoConfiguration` during startup.
3. **Conditional configuration** ŌĆö each auto-config class is gated by `@Conditional*` annotations:
    - `@ConditionalOnClass` ŌĆö only if a specific class exists on the classpath
    - `@ConditionalOnMissingBean` ŌĆö only if the user has NOT already defined that bean
    - `@ConditionalOnProperty` ŌĆö only if a specific property is set
    - `@ConditionalOnWebApplication` ŌĆö only for web applications
4. **Bean registration** ŌĆö if all conditions are satisfied, Spring Boot instantiates the configuration class, registers beans with sensible defaults, and adds them to the `ApplicationContext`.

## Where the JAR Lives and How Spring Finds It

**Physical location:**

- **Dev** ŌĆö Maven cache: `~/.m2/repository/org/springframework/boot/spring-boot-autoconfigure/<version>/...jar`
- **Prod** ŌĆö packaged inside your fat JAR at `BOOT-INF/lib/...jar`, loaded by Spring Boot's `LaunchedClassLoader`

**How Spring finds the configs:** it doesn't search by filename ŌĆö it asks the JVM classloader for a specific resource path, and every JAR on the classpath that ships that file contributes.

```
@EnableAutoConfiguration (inside @SpringBootApplication)
          Ōåō
AutoConfigurationImportSelector
          Ōåō
ImportCandidates.load(AutoConfiguration.class, classLoader)
          Ōåō
classLoader.getResources("META-INF/spring/...AutoConfiguration.imports")
          Ōåō
Returns one entry per JAR that ships this file:
  ŌĆó spring-boot-autoconfigure.jar  ŌåÆ its copy of the imports file
  ŌĆó springdoc-openapi-starter.jar  ŌåÆ its copy
  ŌĆó spring-cloud-config.jar        ŌåÆ its copy
          Ōåō
Spring reads each file, gets fully-qualified class names,
evaluates @Conditional* annotations, registers matching beans
```

Classloader-level, not filename-based ŌĆö any JAR that ships the `.imports` file participates.

## `@ConditionalOnProperty`

Conditionally enables or disables beans and configuration classes based on property values.

Key attributes: `name` (or `value`), `prefix`, `havingValue`, `matchIfMissing`.

```java
@Configuration
@ConditionalOnProperty(name = "feature.notifications.enabled", havingValue = "true")
public class NotificationConfig {
    // Only loaded if feature.notifications.enabled=true
}
```

## Overriding and Debugging

**Opt out of a specific auto-config class** via `exclude`:

```java
@SpringBootApplication(exclude = DataSourceAutoConfiguration.class)
```

**See what got auto-configured** by setting `debug=true` in `application.properties` ŌĆö Spring Boot prints a report showing which auto-configs matched, which didn't, and why.

## Important Interview Questions

1. **What is auto-configuration?**
   Spring Boot's ability to configure beans automatically based on classpath dependencies, properties, and conditional annotations ŌĆö all enabled via `@EnableAutoConfiguration` (included in `@SpringBootApplication`).

2. **What makes Spring Boot "opinionated"?**
   Auto-configuration picks sensible defaults for common setups ŌĆö Tomcat for web, Jackson for JSON, HikariCP for connection pools. You get production-ready wiring without deciding on every library; override only what deviates from the norm.

3. **How does Spring Boot know what to auto-configure?**
   It reads `META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports` (modern) or `META-INF/spring.factories` (legacy) from `spring-boot-autoconfigure.jar`, then evaluates `@Conditional*` annotations on each config class.

4. **What are `@ConditionalOnClass`, `@ConditionalOnMissingBean`, `@ConditionalOnProperty`?**
   Conditional annotations that gate auto-config beans on (a) a class being on classpath, (b) no user-defined bean of that type, (c) a specific property being set. Together they let Spring Boot defer to your configuration when you provide your own.

5. **How do you override an auto-configured bean?**
   Define your own bean of the same type ŌĆö `@ConditionalOnMissingBean` on the auto-config side makes Spring skip the default. Or exclude the auto-config class: `@SpringBootApplication(exclude = DataSourceAutoConfiguration.class)`.

6. **What does `@SpringBootApplication` actually include?**
   Meta-annotation combining `@SpringBootConfiguration` (which is `@Configuration`), `@EnableAutoConfiguration`, and `@ComponentScan`.
