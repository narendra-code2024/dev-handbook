### What is an API?

- An API (Application Programming Interface) is a set of rules or protocols that enables software applications to communicate with each other to exchange data, features, and functionality.

---

### MVC: Model-View-Controller

MVC stands for Model, View, Controller — a design pattern based on **separation of concerns**.

#### Model

- Encapsulates the application data
- Interacts with the database

#### View

- UI layer, displays the data
- In Spring MVC → Thymeleaf / JSP
- In REST APIs → JSON is the view

#### Controller

- Handles incoming HTTP requests
- Delegates to service layer
- Returns response (view name or data)

---

### Spring MVC Flow

#### 1. Client sends HTTP request

> `GET /api/hello`

#### 2. DispatcherServlet receives the request

- It receives `HttpServletRequest` and `HttpServletResponse`
- These are created by the **servlet container (Tomcat)**, not by DispatcherServlet
- DispatcherServlet is the **front controller** — all requests pass through it
- In Spring Boot, DispatcherServlet is registered automatically via auto-configuration (in traditional Spring, it had to be configured in `web.xml` or via `AbstractDispatcherServletInitializer`)

#### 3. HandlerMapping is consulted

- DispatcherServlet asks: "Which controller method can handle this request?"
- Matches URL, HTTP method, headers, etc. to the correct handler

#### 4. Controller method is invoked

- DispatcherServlet forwards request to the matched controller
- Method parameters are auto-resolved:
  - `@PathVariable`
  - `@RequestParam`
  - `@RequestBody`

#### 5. Controller processes the request

- Calls service layer
- Applies business logic
- Returns a response

#### 6. Response handling (differs by app type)

**Traditional MVC:**
- Controller returns a **view name** (String) or `ModelAndView`
- **ViewResolver** resolves the view name to an actual view template (JSP/Thymeleaf)
- View is rendered and sent to client

**REST API:**
- Controller returns an Object/DTO
- **HttpMessageConverter** (Jackson) serializes it to JSON
- ViewResolver is **NOT** used

---

### MVC Flow: Traditional vs REST API

#### Controller Return Types

*Traditional MVC* (JSP / Thymeleaf views):
```java
// Returns a view name — ViewResolver finds the template
@GetMapping("/home")
public String home(Model model) {
    model.addAttribute("message", "Hello");
    return "home"; // resolves to home.jsp or home.html
}

// Alternative: returns ModelAndView explicitly
@GetMapping("/home")
public ModelAndView home() {
    return new ModelAndView("home", "message", "Hello");
}
```

*Modern REST API* (Spring Boot — most common):

- No view name or ModelAndView returned
- JSON is returned directly
- ViewResolver is NOT used
```java
@GetMapping("/api/hello")
public String hello() {
    return "Hello World";
}

@GetMapping("/products")
public List<Product> getProducts() {
    return productService.getAll();
}
```

#### Flow Comparison

> **Traditional MVC:** Controller → View Name / ModelAndView → ViewResolver → JSP / Thymeleaf → HTML Response

> **REST API:** Controller → Object/DTO → HttpMessageConverter (Jackson) → JSON Response

---

### Interview Summary: Spring MVC

- DispatcherServlet is the front controller
- Tomcat creates `HttpServletRequest` & `HttpServletResponse`
- DispatcherServlet uses HandlerMapping to find the correct controller
- Controller processes request and returns:
  - View name or ModelAndView → Traditional MVC
  - Object / DTO → REST API (serialized to JSON)
- ViewResolver is used only for MVC with views
- REST APIs use HttpMessageConverters (Jackson)

> **One-line Interview Answer:** Spring MVC uses DispatcherServlet as a front controller which routes requests to controllers using handler mappings, processes business logic, and returns either views or JSON responses based on application type.
