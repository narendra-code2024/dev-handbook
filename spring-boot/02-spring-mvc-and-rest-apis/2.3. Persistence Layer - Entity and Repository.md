### Overview

The Persistence Layer is responsible for:
- Mapping Java objects to database tables
- Performing CRUD operations
- Managing database interactions

---

### Spring Data JPA

In Spring Boot, this layer is provided by Spring Data JPA.

Starter dependency: `spring-boot-starter-data-jpa`

This starter includes:
- JPA APIs
- Hibernate (default JPA provider in Spring Boot)
- Spring Data JPA abstractions

### JPA Architecture Flow

> Application
> ↓
> Spring Data JPA (Repository)
> ↓
> JPA EntityManager
> ↓
> JPA Provider (Hibernate)
> ↓
> JDBC API
> ↓
> JDBC Driver
> ↓
> Database

Key points:
- JPA is a **specification**, not an implementation
- Hibernate is the default **implementation** (JPA provider) in Spring Boot
- JDBC handles low-level DB communication

### @Entity Annotation
```java
@Entity
@Table(name = "employees")
public class Employee { }
```

What `@Entity` means:
- Marks a class as persistent (managed by JPA)
- Maps the class to a database table
- Enables ORM (Object–Relational Mapping)

Key points:
- Class-level annotation
- Must have a primary key (`@Id`)
- Table name defaults to class name if `@Table` is not used

### Primary Key Annotations
```java
@Id
@GeneratedValue(strategy = GenerationType.IDENTITY)
private Long id;
```

Explanation:
- `@Id` → Marks the primary key
- `@GeneratedValue` → Auto-generates ID values
- `GenerationType.IDENTITY` → Uses database auto-increment (most common with MySQL/PostgreSQL)
- `GenerationType.AUTO` → Lets the persistence provider (Hibernate) choose strategy based on database dialect (may use sequence or table strategy — not always auto-increment)
- `GenerationType.SEQUENCE` → Uses database sequence (preferred for PostgreSQL/Oracle)
- `GenerationType.TABLE` → Uses a separate table to simulate sequences (least efficient)

### JPA Repository
```java
public interface EmployeeRepository extends JpaRepository<Employee, Long> {}
```

#### Repository Interface Hierarchy

> `Repository` (marker)
> ↓
> `CrudRepository` (basic CRUD)
> ↓
> `PagingAndSortingRepository` (pagination & sorting)
> ↓
> `JpaRepository` (batch ops, flush)

#### What JpaRepository Provides

- CRUD operations
- Pagination & sorting
- Batch operations (`saveAll`, `deleteAllInBatch`)
- Flushing (`flush`, `saveAndFlush`)

#### Built-in Methods

- `findAll()`, `findById(id)`, `save(entity)`, `deleteById(id)`, `count()`, `existsById(id)`

#### Derived Query Methods (Important Feature)
```java
List<Employee> findByName(String name);
List<Employee> findByDepartmentAndSalaryGreaterThan(String dept, Double salary);
```

Key points:
- No implementation required — just define the method signature in the repository interface
- Method name defines the query
- Spring Data JPA generates the query at runtime

#### @Query for Custom Queries

When derived method names get too complex, use `@Query`:
```java
@Query("SELECT e FROM Employee e WHERE e.department = :dept AND e.salary > :minSalary")
List<Employee> findByDeptAndMinSalary(@Param("dept") String dept, 
                                       @Param("minSalary") Double minSalary);
```

- Supports JPQL (default) and native SQL (`nativeQuery = true`)

#### findById & Optional
```java
Optional<Employee> employee = repository.findById(id);
```

`Optional` (java.util.Optional):
- Helps avoid `NullPointerException`
- Forces explicit null handling
- Introduced in Java 8
```java
Employee emp = employee.orElseThrow(() -> new ResourceNotFoundException("Employee not found"));
```

### Lombok (Reducing Boilerplate Code)

Dependency:
```xml
<groupId>org.projectlombok</groupId>
<artifactId>lombok</artifactId>
```

Lombok generates common methods using annotations at **compile time**.

#### Common Annotations

| Annotation | Purpose |
|-----------|---------|
| `@Getter` | Generates getter methods |
| `@Setter` | Generates setter methods |
| `@AllArgsConstructor` | Constructor with all fields |
| `@NoArgsConstructor` | No-argument constructor |
| `@Data` | Combines @Getter, @Setter, @ToString, @EqualsAndHashCode, @RequiredArgsConstructor |
| `@Builder` | Builder pattern for object creation |

Benefits:
- Reduces boilerplate
- Cleaner entities and DTOs
- Improves readability

### Hibernate DDL Auto Configuration
```properties
spring.jpa.hibernate.ddl-auto=update
```

What it does:
- Controls how Hibernate manages the database schema on startup

#### Modes

| Mode | Behavior |
|------|----------|
| `none` | Does nothing (default for non-embedded databases) |
| `validate` | Validates schema against entities, throws error on mismatch |
| `update` | Updates schema to match entities (adds columns/tables, never drops) |
| `create` | Drops and recreates schema on every startup |
| `create-drop` | Creates on startup, drops on shutdown |

**Do NOT use `update` or `create` in production.**
Use migration tools like **Flyway** or **Liquibase** for production schema management.

### H2 In-Memory Database

What is H2?
- Lightweight, in-memory relational database
- Used for development & testing
- Data is lost when application stops

Dependency:
