### What is the Service Layer?

The Service Layer acts as a bridge between the Controller (Presentation Layer) and the Repository (Persistence Layer). It contains the **business logic** of the application.

### Responsibilities

- Implement business rules
- Coordinate multiple repositories
- Handle transactions
- Perform validations not related to HTTP
- Trigger domain events (if any)

### @Service Annotation
```java
@Service
public class OrderService {}
```

- Specialization of `@Component`
- Marks class as a Spring-managed bean
- Indicates business logic layer
- Enables: Dependency injection, Transaction management, AOP (logging, security, etc.)

**Best Practice:** One service = one responsibility. Keep controllers thin — they should only handle HTTP concerns.

> **One-line Interview Answer:** The Service Layer contains business logic, coordinates repositories, manages transactions, and ensures separation of concerns between controllers and persistence.

### Transactions

#### What is a Transaction?

A transaction is a sequence of operations that must be completed **atomically** — either all succeed or all are rolled back.

#### @Transactional Annotation
```java
@Transactional
public void placeOrder(Order order) {
    orderRepository.save(order);
    paymentService.charge(order);
}
```

Key points:
- Ensures **ACID** properties (Atomicity, Consistency, Isolation, Durability)
- If any step fails → entire transaction is rolled back
- Can be applied at method level or class level
- Uses **proxy-based AOP** internally

#### Important Gotchas

*Self-invocation problem:* Calling a `@Transactional` method from within the same class will NOT trigger the transaction, because the proxy is bypassed. Always call `@Transactional` methods from another bean.
```java
@Service
public class OrderService {
    
    public void process() {
        this.placeOrder(order); // Transaction NOT applied — self-invocation!
    }
    
    @Transactional
    public void placeOrder(Order order) { ... }
}
```

*Rollback behavior:*
- Rolls back on **unchecked exceptions** (`RuntimeException` and subclasses) by default
- Does NOT roll back on **checked exceptions** unless specified:
```java
@Transactional(rollbackFor = Exception.class)  // rolls back on all exceptions
```

#### Read-only Transactions

Improves performance and prevents accidental updates:
```java
@Transactional(readOnly = true)
public List<Order> getOrders() {
    return orderRepository.findAll();
}
```
