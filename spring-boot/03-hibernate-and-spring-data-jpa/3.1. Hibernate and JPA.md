### JDBC (Java Database Connectivity)

- Lowest-level API to connect Java applications to a database
- Executes raw SQL queries
- Handles: Database connections, Statements, Result sets
- Hibernate internally uses JDBC to execute SQL queries
- If you use JDBC directly, you must manually:
  - Write SQL
  - Map ResultSet to objects
  - Handle transactions

---

### JPA (Java Persistence API)

- **Specification** (not an implementation) for ORM in Java
- Defines:
  - Annotations (`@Entity`, `@Id`, etc.)
  - Interfaces (`EntityManager`)
  - JPQL (Java Persistence Query Language)
  - Transaction rules
  - Relationship mapping
- Popular JPA Providers:
  - Hibernate
  - EclipseLink
  - Apache OpenJPA
- JPA does NOT provide implementation. It only defines how ORM should work.

---

### Hibernate

- Most popular JPA implementation
- Full-featured ORM framework
- Converts: Entity objects → SQL, SQL results → Java objects
- Provides:
  - ORM mapping
  - Automatic SQL generation
  - Caching (1st level & 2nd level)
  - Lazy/Eager loading
  - Query optimization
  - Native SQL support
- Can be used:
  - As a standalone ORM framework
  - Or as a JPA provider (most common with Spring Boot)

---

### Spring Data JPA

- Built on top of JPA
- Main purpose:
  - Simplifies the Data Access Layer (DAO layer) using Repository
  - Eliminates boilerplate code
- Spring Data JPA automatically:
  - Generates queries
  - Implements repository methods
  - Handles transactions (with Spring)
- Note:
  - JPA provides the EntityManager API to perform CRUD operations
  - But it does not provide automatic repository abstraction like Spring Data JPA

### Hibernate Configuration (Spring Boot)

| Property | Description |
|----------|-------------|
| `spring.jpa.hibernate.ddl-auto` | Controls schema generation |
| `spring.jpa.show-sql` | Prints SQL in console |
| `spring.jpa.properties.hibernate.format_sql` | Formats SQL nicely |
| `spring.jpa.properties.hibernate.dialect` | Defines SQL dialect |

#### spring.jpa.hibernate.ddl-auto Options

| Value | Meaning |
|-------|---------|
| `none` | No action (default for non-embedded databases) |
| `validate` | Validates schema against entities |
| `update` | Updates schema if needed |
| `create` | Drops and recreates schema |
| `create-drop` | Drops schema at shutdown |

**Production recommendation → `validate` or `none`**

### Entity Annotations in JPA

#### @Entity

Marks class as a JPA entity.
```java
@Entity
public class Employee { }
```

#### @Table

Used to customize table mapping.
```java
@Table(
    name = "employees",
    catalog = "employee_catalog",
    schema = "hr",
    uniqueConstraints = {
        @UniqueConstraint(columnNames = {"email"})
    },
    indexes = {
        @Index(name = "idx_name", columnList = "name"),
        @Index(name = "idx_department", columnList = "department")
    }
)
```

| Attribute | Purpose |
|-----------|---------|
| `name` | Table name |
| `schema` | DB schema |
| `catalog` | DB catalog |
| `uniqueConstraints` | Unique keys |
| `indexes` | DB indexes |

#### @Id

Defines primary key.

#### @GeneratedValue
```java
@GeneratedValue(strategy = GenerationType.IDENTITY)
```

Strategies: `AUTO`, `IDENTITY`, `SEQUENCE`, `TABLE`

#### @Column
```java
@Column(name = "name", nullable = false, length = 50)
```

Options: `name`, `nullable`, `unique`, `length`, `updatable`, `insertable`

#### @CreationTimestamp & @UpdateTimestamp

Hibernate-specific annotations (not pure JPA).
```java
@CreationTimestamp
private LocalDateTime createdAt;

@UpdateTimestamp
private LocalDateTime updatedAt;
```

### spring.jpa.defer-datasource-initialization
```properties
spring.jpa.defer-datasource-initialization=true
```

Purpose:
- Hibernate creates tables first
- Then `data.sql` runs

Without this:
- `data.sql` may execute before tables exist

### Spring SQL Init Properties

| Property | Description | Default | Possible Values |
|----------|-------------|---------|-----------------|
| `spring.sql.init.mode` | Controls when SQL scripts run | `embedded` | `always`, `embedded`, `never` |
| `spring.sql.init.schema-locations` | Location of schema SQL scripts | `classpath:schema.sql` | Any valid path |
| `spring.sql.init.data-locations` | Location of data SQL scripts | `classpath:data.sql` | Any valid path |
| `spring.sql.init.platform` | Platform-specific script variant | `all` | Any platform name |
| `spring.sql.init.continue-on-error` | Continue even if error occurs | `false` | `true`/`false` |
| `spring.sql.init.encoding` | Script encoding | `UTF-8` | Any valid encoding |

### Key Features of JPA

#### 1. Entity Management

- Maps Java classes to DB tables
- Managed via `EntityManager`

#### 2. JPQL (Java Persistence Query Language)

- Object-oriented query language
- Uses entity names, not table names
- Converted internally to SQL by Hibernate (JPA provider)

#### 3. Transactions

- Managed via `@Transactional` and `PlatformTransactionManager`
- Ensures atomicity and rollback on failure

#### 4. Entity Relationships

| Annotation | Relationship |
|-----------|-------------|
| `@OneToOne` | One-to-One |
| `@OneToMany` | One-to-Many |
| `@ManyToOne` | Many-to-One |
| `@ManyToMany` | Many-to-Many |

Supports: Cascade types, Fetch types (LAZY / EAGER), Join columns

- **Hibernate** → Implementation of JPA
- **Spring Data JPA** → Simplifies repository layer
- **JDBC** → Executes SQL at lowest level
- **Entities** → Java classes mapped to DB tables
- **JPQL** → Query language based on objects
- **Hibernate Dialect** → DB-specific SQL generation

### JPA: Specification, Implementation, or Both?

The Java Persistence API (JPA) is a **specification** because it defines a standard set of rules, interfaces, and annotations for ORM, but it does not contain the concrete code to perform actual database operations. It relies on **JPA providers** (like Hibernate) to supply the underlying implementation.

So when we say "JPA provides EntityManager":
- JPA **defines** the `EntityManager` interface
- Hibernate **implements** it
- Spring **injects** the implementation

### Flow: When you call `entityManager.persist(user)`

> 1. Your code calls JPA interface
> 2. Hibernate implementation executes logic
> 3. Hibernate generates SQL
> 4. SQL executed via JDBC
> 5. JDBC talks to database
