### Sorting Using Method Name (OrderBy Keyword)
```java
@Repository
public interface EmployeeRepository extends JpaRepository<Employee, Long> {
    List<Employee> findAllByOrderByNameAsc();
    List<Employee> findAllByOrderByNameDesc();
}
```

Rules:
- `OrderBy` must come after the `By` clause and any query conditions
- Sorting direction (`Asc` / `Desc`) is optional. Default is `ASC`
- Limitation: Not dynamic — field name and direction are hardcoded, cannot change at runtime

Examples:
```java
findByDepartmentOrderByNameAsc()
findByDepartmentOrderByNameDesc()
findByDepartmentOrderByName()       // default ASC
findAllByOrderByNameAsc()           // no condition, just sort
```

### Sorting Using Sort Parameter (Dynamic)

Better approach for flexibility.

Repository:
```java
@Repository
public interface EmployeeRepository extends JpaRepository<Employee, Long> {
    List<Employee> findByDepartment(String department, Sort sort);
}
```

Usage:
```java
Sort sort = Sort.by(Sort.Direction.ASC, "name");
List<Employee> employees = repo.findByDepartment("IT", sort);
```

Multiple sort fields:
```java
Sort sort = Sort.by(
    Sort.Order.asc("name"),
    Sort.Order.desc("salary")
);
```

Advantages: Fully dynamic, multiple sorting fields supported.

### Sorting with findAll(Sort)

`JpaRepository` already provides:
```java
List<Employee> findAll(Sort sort);
```

Example:
```java
Sort sort = Sort.by("name").descending();
List<Employee> employees = repo.findAll(sort);
```

### Pagination in Spring Data JPA

Pagination allows splitting large datasets into smaller chunks (pages).

#### Key Interfaces

**Page** — Represents a page of results. Provides:
- `getContent()` — list of items on current page
- `getTotalElements()` — total count across all pages
- `getTotalPages()` — total number of pages
- `getNumber()` — current page number
- `isFirst()`, `isLast()` — boundary checks

**Pageable** — Input interface that provides page number, page size, and sorting options.

**PageRequest** — Concrete implementation of `Pageable`. Used to create pageable objects.

---

### Using Page, Pageable, PageRequest

Repository:
```java
public interface UserRepository extends JpaRepository<User, Long> {
    Page<User> findAll(Pageable pageable);
    Page<User> findByLastName(String lastName, Pageable pageable);
}
```

Creating Pageable instance:
```java
Pageable pageable = PageRequest.of(0, 10); // First page, 10 records
```

**Important: pageNumber is zero-based.**

Combining Pagination + Sorting:
```java
Pageable pageable = PageRequest.of(
    0,
    5,
    Sort.by("salary").descending()
);
Page<Employee> page = repo.findAll(pageable);
```

---

### How Pagination Works Internally
```java
Page<Employee> page = repo.findAll(pageable);
```

Spring Data JPA generates two queries:

> 1. Main Query: `SELECT * FROM employee LIMIT ? OFFSET ?`
> 2. Count Query: `SELECT COUNT(*) FROM employee`

---

### Page vs Slice (Very Important)

| | Page | Slice |
|-|------|-------|
| COUNT query | Yes | No |
| Knows total pages | Yes | Only knows if next page exists |
| Performance | Slightly heavier | More performant |
| Internal trick | Uses COUNT query | Fetches `LIMIT + 1` rows to check if more exist |

If you don't need total count → use `Slice`.

### Return Types for Pagination

- `Page<T>` — need total count
- `Slice<T>` — no total count, better performance
- `List<T>` — only items, no pagination metadata

---

### Is Sort/OrderBy Mandatory in Pagination?

Sorting is **not mandatory** for pagination. You can paginate using `PageRequest.of(page, size)` without sorting. However, defining a **stable sort order is highly recommended** to ensure consistent data across pages, as database results without explicit sorting can appear in arbitrary orders.
