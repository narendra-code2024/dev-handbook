### What is Projection?

- A mechanism in Spring Data JPA that allows you to fetch **only selected fields** from an entity instead of retrieving the entire entity
- Instead of: `SELECT * FROM employee`
- You can fetch: `SELECT name, salary FROM employee`
- Improves: Performance, Memory usage, Network overhead, Query execution time

### Why Use Projections?

Normally `List<Employee> findAll()` loads all columns and all relationships (depending on fetch type). Fetching the entire entity is wasteful if you only need name & email. Projection solves this.

---

### Types of Projections

#### 1. Interface-Based Projection (Most Common)

You define an interface with getter methods.

Entity:
```java
@Entity
public class Employee {
    private Long id;
    private String name;
    private String email;
    private Double salary;
}
```

Projection Interface:
```java
public interface EmployeeNameEmail {
    String getName();
    String getEmail();
}
```

Repository method:
```java
List<EmployeeNameEmail> findBySalaryGreaterThan(Double salary);
```

Spring automatically creates proxy, maps selected fields, and returns only name & email.

**Important Rules:**
- Getter names must match entity field names
- Only declared getters are fetched
- Works with derived queries and `@Query`

#### 2. Class-Based Projection (DTO Projection)

Instead of interface, use a class.

DTO:
```java
public class EmployeeDto {
    private String name;
    private String email;
    public EmployeeDto(String name, String email) {
        this.name = name;
        this.email = email;
    }
}
```

Repository:
```java
@Query("SELECT new com.example.EmployeeDto(e.name, e.email) FROM Employee e")
List<EmployeeDto> fetchEmployeeData();
```

**Constructor must match selected fields exactly.**

#### 3. Dynamic Projection (Advanced)

Allows choosing projection type at runtime.
```java
<T> List<T> findBySalaryGreaterThan(Double salary, Class<T> type);
```

Usage:
```java
repo.findBySalaryGreaterThan(50000, Employee.class);        // full entity
repo.findBySalaryGreaterThan(50000, EmployeeNameEmail.class); // projection
```

Very useful in reusable APIs.

---

### Closed vs Open Projections

**Closed Projection:** Only simple getters. Spring fetches only required columns.
```java
String getName();
```

**Open Projection:** Uses SpEL expressions. Entire entity is fetched — less efficient. Avoid for performance-critical code.
```java
@Value("#{target.name + ' - ' + target.email}")
String getDisplayInfo();
```

---

### Nested Projections

Works with relationships. Spring automatically performs JOIN.
```java
public interface EmployeeView {
    String getName();
    DepartmentView getDepartment();
}

public interface DepartmentView {
    String getDepartmentName();
}
```

---

### Entity vs Interface Projection vs DTO Projection

| Feature | Entity | Interface Projection | DTO Projection |
|---------|--------|---------------------|----------------|
| Fetch full row | Yes | No | No |
| Fetch selected fields | No | Yes | Yes |
| Requires constructor | No | No | Yes |
| Proxy based | No | Yes | No |
| Best performance | No | Yes | Yes |

---

### Important Clarifications

- Projection works only for SELECT queries
- Not usable for save/update
- Reduces over-fetching problem
- Helps avoid N+1 in some read cases (but not always)

### Internal Working (Interface Projection)

1. Spring generates proxy
2. JPQL selects only requested fields
3. Hibernate executes SQL
4. Proxy maps results to interface methods

> **Interview Summary:** Projection = Fetch partial data. 3 types → Interface, DTO, Dynamic. Interface projection = easiest & cleanest. Open projection fetches full entity (avoid). Improves performance by reducing over-fetching. Works with nested relationships.
