### What is EntityManager?

EntityManager is the main JPA interface used to:
- Persist entities
- Remove entities
- Find entities
- Manage transactions
- Manage persistence context

---

### What is Persistence Context?

- A set (cache) of managed entity instances associated with an EntityManager
- Also called **First-Level Cache**
- Properties:
  - Exists per EntityManager
  - Stores managed entities
  - Ensures only one entity instance per primary key
  - Tracks changes automatically (Dirty Checking)

Example: `emp` and `emp2` refer to the same object instance because persistence context caches it.
```java
Employee emp = entityManager.find(Employee.class, 1L);
Employee emp2 = entityManager.find(Employee.class, 1L);
// emp == emp2 → true (same reference)
```

---

### Entity Lifecycle States

#### 1. Transient (New)

- Just created using `new`
- Not associated with EntityManager
- Not stored in database
```java
Employee emp = new Employee(); // Transient
```

#### 2. Persistent (Managed)

- Associated with Persistence Context
- Tracked by EntityManager
- Changes automatically synchronized with DB
```java
entityManager.persist(emp); // Now Persistent
```

Or:
```java
Employee emp = entityManager.find(Employee.class, 1L); // Also Persistent
```

#### 3. Detached

- Was persistent before
- No longer associated with EntityManager
- Changes are NOT tracked

Happens when:
- Transaction ends
- `entityManager.clear()`
- `entityManager.detach(entity)`
```java
entityManager.detach(emp); // Now Detached
```

#### 4. Removed

- Marked for deletion
- Will be deleted on commit
```java
entityManager.remove(emp); // Marked for removal
```

---

### Dirty Checking (Very Important)

When entity is in **Persistent** state, you don't need to call update or save:
```java
Employee emp = entityManager.find(Employee.class, 1L);
emp.setName("Narendra");
// No update() call needed
```

At transaction commit:
- Hibernate compares original state vs current state
- Generates UPDATE automatically

This is called **Dirty Checking**.

---

### Flush vs Commit

**flush()**
- Synchronizes persistence context with database
- Executes SQL statements
- Does **NOT** commit transaction — changes can still be rolled back

**commit()**
- Permanently saves changes
- Ends transaction

Spring calls flush automatically before commit.

---

### Persistence Context Scope in Spring

In Spring Boot:
- Each transaction has its own persistence context
- Managed by `@Transactional`
- Closed after transaction ends

---

### First-Level Cache vs Second-Level Cache

| | First-Level Cache | Second-Level Cache |
|-|-------------------|-------------------|
| Scope | Per EntityManager | Shared across sessions |
| Enabled | By default | Must be configured |
| Can disable | No | Yes |

### Why Does LazyInitializationException Happen?

Because:
- Entity is **Detached** (persistence context is closed)
- Lazy relationship tries to load data
- No active session available to execute the query

This is one of the most common JPA errors and a frequent interview question.
