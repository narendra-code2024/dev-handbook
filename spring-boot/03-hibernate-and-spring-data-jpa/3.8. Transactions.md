A transaction is a unit of work that satisfies the **ACID** properties:

- **Atomicity** ‚Äî all operations succeed or all are rolled back
- **Consistency** ‚Äî data moves from one valid state to another
- **Isolation** ‚Äî concurrent transactions don't interfere
- **Durability** ‚Äî committed data survives crashes

```java
@Service
public class OrderService {

    @Transactional
    public void createOrder(Order order) {
        orderRepository.save(order);
        paymentService.processPayment(order);
        // if processPayment throws, the save is rolled back
    }
}
```

## Transactions and the Persistence Context

This is the angle that makes transactions matter in this Hibernate / Spring Data JPA section:

- A `@Transactional` method opens a transaction; JPA binds a **persistence context** (`EntityManager`) to that transaction
- All repository calls inside the method share the **same** persistence context ‚Äî entities loaded once stay managed; lazy associations can be navigated freely
- When the method returns successfully, the persistence context **flushes** pending changes and the transaction commits
- When it throws an unchecked exception, the persistence context discards changes and the transaction rolls back

Without `@Transactional`, **each repository call gets its own short-lived persistence context** ‚Äî entities returned are detached, and lazy associations throw `LazyInitializationException` the moment you touch them outside the call.

> The persistence context lifecycle is covered in detail in [3.5 `EntityManager`, Persistence Context & Entity Lifecycle](<3.5. EntityManager, Persistence Context & Entity Lifecycle.md>).

## JPA-Specific Pitfalls

- **Missing `@Transactional` in the service layer** ‚Üí `LazyInitializationException` because the persistence context closes between repository calls. Annotate the service method (or use OSIV as a workaround, but better to be explicit).
- **Cascade `ALL` everywhere** ‚Üí combined with transaction commits, accidental deletes propagate across the entity graph on a single rollback or detach. See [3.7 Cascading and Orphan Removal](<3.7. Cascading and Orphan Removal.md>).
- **Read-only `findAll()` + flush on commit** ‚Üí without `@Transactional(readOnly = true)`, Hibernate runs dirty-checking on every loaded entity. See [11.6 `@Transactional` Annotation](<../11-caching-and-concurrent-transaction-management/11.6. Transactional Annotation in Spring Boot.md>) ‚Üí `readOnly`.

> For non-JPA pitfalls ‚Äî self-invocation, private methods, missing `rollbackFor` for checked exceptions, propagation interactions ‚Äî see [11.6](<../11-caching-and-concurrent-transaction-management/11.6. Transactional Annotation in Spring Boot.md>).

## Important Interview Questions

1. **How does `@Transactional` interact with the JPA persistence context?**
   The proxy opens a transaction; JPA binds a persistence context (`EntityManager`) to it. All repository operations within the method share that context ‚Äî entities stay managed, lazy associations are navigable, and changes are flushed on commit. Without `@Transactional`, each repository call uses its own short-lived persistence context and entities are detached on return.
