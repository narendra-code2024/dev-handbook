### What is the N+1 Problem?

It happens when:
- 1 query loads parent entities
- Then N additional queries load each child entity
- Total queries = 1 + N

---

### Example

Entity:
```java
@Entity
class Order {
    @OneToMany(mappedBy = "order", fetch = FetchType.LAZY)
    private List<OrderItem> items;
}
```

Code:
```java
List<Order> orders = orderRepository.findAll();
for (Order order : orders) {
    System.out.println(order.getItems().size());
}
```

What happens internally:

**Step 1:** Fetch Orders
```sql
SELECT * FROM orders;
-- Returns 5 orders
```

**Step 2:** For each order → load items (LAZY triggers on access)
```sql
SELECT * FROM order_item WHERE order_id = 1;
SELECT * FROM order_item WHERE order_id = 2;
SELECT * FROM order_item WHERE order_id = 3;
SELECT * FROM order_item WHERE order_id = 4;
SELECT * FROM order_item WHERE order_id = 5;
```

Total: 1 (orders) + 5 (items) = **6 queries**. If 100 orders → 101 queries.

---

### Why It Happens

- `@OneToMany` is LAZY by default
- Hibernate loads children only when accessed
- Inside loop → triggers multiple selects

### Important Clarification

N+1 can happen with both LAZY and EAGER. Hibernate may still fire multiple queries with EAGER in some cases. Changing fetch type is **NOT** a reliable solution.

---

### Solutions

#### 1. Fetch Join (Best & Most Common)
```java
@Query("SELECT o FROM Order o JOIN FETCH o.items")
List<Order> findAllWithItems();
```

SQL generated:
```sql
SELECT o.*, i.*
FROM orders o
JOIN order_item i ON o.id = i.order_id;
```

Only 1 query.

#### 2. EntityGraph (Cleaner with Spring Data JPA)
```java
@EntityGraph(attributePaths = "items")
List<Order> findAll();
```

Spring automatically performs fetch join.

#### 3. Batch Fetching (Hibernate Specific)
```properties
spring.jpa.properties.hibernate.default_batch_fetch_size=20
```

Instead of 1 + 100 queries → becomes 1 + 5 queries (batch of 20 each). Better but not best.

#### 4. DTO Projection (Most Performant for Read-heavy Systems)
```java
@Query("""
    SELECT new com.example.OrderDTO(o.id, i.name)
    FROM Order o
    JOIN o.items i
""")
List<OrderDTO> findOrderData();
```

Loads only required fields. Best for reporting, APIs, and large datasets.

---

### Fetch Join vs Normal Join

- **JOIN** → used for filtering (WHERE clause)
- **JOIN FETCH** → used for fetching the association (loading related data)

If you use normal JOIN, Hibernate may still fire additional queries for the association.

---

### Limitations of Fetch Join

- Cannot use multiple bag collections (MultipleBagFetchException)
- Can cause Cartesian product explosion with multiple collections
- Not good with pagination (Hibernate fetches all rows in memory, then paginates)

---

### How to Detect N+1

Enable SQL logging:
```properties
spring.jpa.show-sql=true
spring.jpa.properties.hibernate.format_sql=true
```

Or use:
- Hibernate statistics
- p6spy (SQL logging library)
- Count executed queries in tests
