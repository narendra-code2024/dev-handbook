### What is Logging?

Logging is the process of tracking all events that happen after a piece of code is run. It helps track where exactly the code crashes and eases debugging.

A logging framework handles: setting log file destinations, customizing log messages, routing logs to different outputs, etc.

---

### SLF4J

Java provides various logging frameworks: Log4J, java.util.logging (JUL), Tinylog, Logback, etc.

Spring Boot comes with **SLF4J** inbuilt — an abstraction over all these logging frameworks.

- SLF4J stands for **Simple Logging Facade for Java**
- Allows you to work with any logging framework using a single dependency
- **Logback** is the default implementation in Spring Boot

---

### Elements of a Logging Framework

| Element | Purpose |
|---------|---------|
| **Logger** | Captures the messages |
| **Formatter** | Formats the messages captured by the logger |
| **Handler** | Dispatches messages — prints to console, stores in file, sends email, etc. |

---

### Log Levels

From most severe to most verbose:

| Level | Description |
|-------|-------------|
| `FATAL` | Fatal error crashing the system |
| `ERROR` | Runtime errors |
| `WARN` | Warnings |
| `INFO` | Events occurring at runtime |
| `DEBUG` | Information about the flow of the system |
| `TRACE` | More detailed information about system flow |

When you enable a level, it logs events at that level **and all levels above it**. For example, enabling `WARN` shows WARN, ERROR, and FATAL — but not INFO, DEBUG, or TRACE.

### Two Ways to Configure Logging

#### 1. application.properties (Simple)

Quick setup directly in properties. Good for basic needs.
```properties
# Set log levels
logging.level.root=INFO
logging.level.com.myPackageName=DEBUG

# Console output format
logging.pattern.console=%d [%level] %c{1.} [%t] %m%n

# File output
logging.file.name=logs/app.log
logging.pattern.file=%d{yyyy-MM-dd HH:mm:ss} [%level] %c{1.} [%t] %m%n
```

**Important:**
- `logging.file.name` — sets the output file path and name
- `logging.pattern.file` — sets the format of log messages in that file
- Both are independent — setting `logging.pattern.file` alone without `logging.file.name` won't create a file

**Limitation:** Properties-based config cannot do rolling files, size limits, or multiple appenders. For that, use Logback XML.

#### 2. logback-spring.xml (Advanced — Recommended for Production)

Create `logback-spring.xml` in `src/main/resources`. When this file exists, it **overrides** the logging properties in `application.properties`.

Use `logback-spring.xml` (not `logback.xml`) to leverage Spring Boot profile support.

---

### Pattern Placeholders

| Pattern                   | Meaning                               | Example Output                        |
| ------------------------- | ------------------------------------- | ------------------------------------- |
| `%d{yyyy-MM-dd HH:mm:ss}` | Date/timestamp                        | `2026-02-22 14:30:00`                 |
| `%-5level`                | Log level (left-padded to 5 chars)    | `INFO `, `DEBUG`                      |
| `%level`                  | Log level (no padding)                | `INFO`, `DEBUG`                       |
| `%c{1}`                   | Class name (short, last segment only) | `EmployeeService`                     |
| `%c`                      | Full class path                       | `com.example.service.EmployeeService` |
| `%t`                      | Thread name                           | `main`, `http-nio-8080-exec-1`        |
| `%msg` or `%m`            | Log message                           | `Fetching employee with id: 1`        |
| `%n`                      | New line                              | —                                     |

Example pattern:

> `%d{yyyy-MM-dd HH:mm:ss} %-5level %c{1} - %msg%n`

Output:

> `2026-02-22 14:30:00 INFO  EmployeeService - Fetching employee with id: 1`

### Understanding logback-spring.xml (Line by Line)
```xml
<configuration>

    <!-- Define reusable variables -->
    <property name="LOG_PATTERN" 
              value="%d{yyyy-MM-dd HH:mm:ss} %-5level %c{1} - %msg%n"/>
    <property name="LOG_PATH" value="logs"/>

    <!-- Appender: WHERE logs go and HOW they're managed -->
    <appender name="ROLLING_FILE" 
              class="ch.qos.logback.core.rolling.RollingFileAppender">

        <!-- Rolling Policy: WHEN to create new files -->
        <rollingPolicy 
            class="ch.qos.logback.core.rolling.TimeBasedRollingPolicy">

            <!-- File naming: logs/application.2026-02-22.0.log -->
            <fileNamePattern>
                ${LOG_PATH}/application.%d{yyyy-MM-dd}.%i.log
            </fileNamePattern>

            <!-- Keep last 30 days of logs, delete older -->
            <maxHistory>30</maxHistory>

            <!-- Also split by size within the same day -->
            <timeBasedFileNamingAndTriggeringPolicy 
                class="ch.qos.logback.core.rolling.SizeAndTimeBasedFNATP">
                <!-- Start new file when current reaches 10MB -->
                <maxFileSize>10MB</maxFileSize>
            </timeBasedFileNamingAndTriggeringPolicy>

        </rollingPolicy>

        <!-- Encoder: HOW logs are formatted -->
        <encoder>
            <pattern>${LOG_PATTERN}</pattern>
        </encoder>

    </appender>

    <!-- Root logger: minimum level + which appenders to use -->
    <root level="debug">
        <appender-ref ref="ROLLING_FILE"/>
    </root>

</configuration>
```

#### How the File Naming Works

With pattern `${LOG_PATH}/application.%d{yyyy-MM-dd}.%i.log`:

> `logs/application.2026-02-22.0.log` — first file of the day
> `logs/application.2026-02-22.1.log` — second file (when first hits 10MB)
> `logs/application.2026-02-23.0.log` — next day, starts fresh

After 30 days, oldest files are automatically deleted.

### Adding Console Appender Alongside File

To log to **both** console and file:
```xml
<appender name="CONSOLE" class="ch.qos.logback.core.ConsoleAppender">
    <encoder>
        <pattern>${LOG_PATTERN}</pattern>
    </encoder>
</appender>

<root level="info">
    <appender-ref ref="CONSOLE"/>
    <appender-ref ref="ROLLING_FILE"/>
</root>
```

---

### Package-Level Log Levels

Override root level for specific packages:
```xml
<!-- Root at INFO, but your package at DEBUG -->
<root level="info">
    <appender-ref ref="CONSOLE"/>
    <appender-ref ref="ROLLING_FILE"/>
</root>

<logger name="com.example.service" level="DEBUG"/>
<logger name="org.hibernate.SQL" level="DEBUG"/>
```

### Using Logger in Code
```java
import org.slf4j.Logger;
import org.slf4j.LoggerFactory;

@Service
public class EmployeeService {

    private static final Logger log = LoggerFactory.getLogger(EmployeeService.class);

    public Employee getEmployee(Long id) {
        log.info("Fetching employee with id: {}", id);
        log.debug("Querying database for employee: {}", id);
        log.error("Employee not found with id: {}", id);
        return repository.findById(id).orElseThrow();
    }
}
```

- Use `{}` placeholders instead of string concatenation — better performance
- Or use Lombok `@Slf4j` annotation to auto-generate the logger field:
```java
@Slf4j
@Service
public class EmployeeService {
    // log.info(), log.debug() etc. available directly
}
```
