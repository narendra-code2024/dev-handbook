### Adding Spring Security

Just add the dependency — Spring Boot auto-configures security with sensible defaults:

```xml
<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-security</artifactId>
</dependency>
```

---

### Authentication vs Authorization

- **Authentication** — verifying **who** the user is (validating credentials like email/password, creating a security context)
- **Authorization** — determining **what** an authenticated user can do (checking permissions to access resources)

### How Requests Flow

```
Client (HTTP Request)
       │
       ▼
Servlet Container (Tomcat)
       │
       ├── Filter-0
       ├── Filter-1
       │
       ├── DelegatingFilterProxy
       │        │
       │        ▼
       │   FilterChainProxy
       │        │
       │        ▼
       │   SecurityFilterChain
       │        ├── CorsFilter
       │        ├── CsrfFilter
       │        ├── LogoutFilter
       │        ├── JwtAuthFilter (custom)
       │        ├── UsernamePasswordAuthFilter
       │        ├── ExceptionTranslationFilter
       │        └── AuthorizationFilter
       │
       ├── Filter-2
       ▼
DispatcherServlet → Controller
```

- `SecurityFilterAutoConfiguration` automatically registers `DelegatingFilterProxy` with the name `springSecurityFilterChain`
- `DelegatingFilterProxy` is a regular Servlet filter that **delegates** to `FilterChainProxy` (a Spring bean)
- `FilterChainProxy` executes all security filters inside `SecurityFilterChain` for the current request
- After security filters pass, the request continues through remaining Servlet filters to `DispatcherServlet`

---

### The Filter Chain Concept

- When you add `spring-boot-starter-security`, Spring inserts ~15 filters into the request pipeline
- Each filter handles a specific security concern
- Request passes through ALL filters in order
- If any filter rejects the request, the chain stops

### Key Filters (in order)

|Filter|Purpose|
|---|---|
|`DisableEncodeUrlFilter`|Prevents session IDs from leaking into URLs|
|`SecurityContextPersistenceFilter`|Loads/saves SecurityContext from session|
|`CsrfFilter`|Validates CSRF tokens on POST/PUT/DELETE|
|`UsernamePasswordAuthenticationFilter`|Handles form login (POST /login)|
|`ExceptionTranslationFilter`|Catches security exceptions, redirects to login or returns 403|
|`AuthorizationFilter`|Final gate — checks if user has permission|

### Full Request Lifecycle (with Authentication)

```
HTTP Request
       │
       ▼
 1. Security Filters Chain
       │
       ▼
 2. Authentication is delegated
       │
       ▼
   AuthenticationManager
       │
       ▼
 3. Uses the appropriate AuthenticationProvider
    [DaoAuthenticationProvider, InMemoryAuthenticationProvider,
     OAuth2AuthenticationProvider, ...]
       │
       ▼
 4. Returns the authenticated user
       │
       ▼
   AuthenticationManager
       │
       ▼
 5. Gets the authenticated user
       │
       ▼
 6. Sets authenticated user in SecurityContextHolder → SecurityContext
       │
       ▼
 7. DispatcherServlet → Controller
```

### Custom Filters

- You can add your own filter (like `JwtAuthFilter`) using `addFilterBefore()` or `addFilterAfter()`
- `OncePerRequestFilter` — base class that guarantees your filter runs exactly once per request

> `.addFilterBefore(jwtAuthFilter, UsernamePasswordAuthenticationFilter.class)`

---

### Default Behavior (no config)

When you just add the security dependency with no configuration:

- Every endpoint requires authentication
- A default login form is generated at `/login`
- A user `user` is created with a random password (printed in console)
- Password storage protected with BCrypt
- CSRF protection is enabled
- Session fixation protection is on
- HTTP basic authentication enabled
- Logout feature enabled
