### What is JWT?

- **JSON Web Token** βÄî a compact, self-contained token for authentication
- Server creates it, client stores it, client sends it with every request
- Server validates it without storing anything βÄî **stateless**

---

### Token Structure

> `eyJhbGciOi.eyJzdWIiOi.SflKxwRJSM` `|_header___|__payload__|_signature_|`

Three parts separated by dots:

- **Header** βÄî algorithm used (e.g., HS256)
- **Payload** βÄî data (email, roles, expiry). Anyone can decode this βÄî **never put passwords here**
- **Signature** βÄî created using a secret key only your server knows. Proves token wasn't tampered with

---

### Why Use JWT?

1. Stateless βÄî server stores nothing
2. Scalable in distributed systems βÄî any server can validate
3. Cross-domain authentication
4. Ideal for microservices
5. Highly secure (when used correctly)

---

### Session vs JWT

|Aspect|Session|JWT|
|---|---|---|
|State|Server stores sessions (stateful)|Server stores nothing (stateless)|
|Storage|JSESSIONID cookie|Token in header or HttpOnly cookie|
|Scalability|Hard across multiple servers|Easy βÄî any server can validate|
|Logout|Destroy session on server|Can't truly invalidate (use refresh tokens)|
|DB hit|Every request (session lookup)|Only during login and refresh|

> **Note:** Our implementation does a DB lookup on every request inside `JwtAuthFilter` to load `UserDetails`. This is a design choice for accuracy (e.g., if roles change). Some implementations skip this and trust the token claims alone for better performance.

---

### JWT Flow

> 1. **LOGIN:** `POST /api/auth/login` βÜ£ server validates credentials βÜ£ creates JWT βÜ£ returns token (JSON body + HttpOnly cookie)
> 2. **API CALL:** `GET /api/posts` with `Authorization: Bearer eyJ...` βÜ£ JwtAuthFilter extracts token βÜ£ validates signature + expiry βÜ£ extracts email βÜ£ loads user from DB βÜ£ sets SecurityContext βÜ£ request reaches controller
> 3. **LOGOUT:** Client deletes token / server clears cookie

### Token Delivery βÄî Two Approaches

|Approach|How It Works|Used By|
|---|---|---|
|**JSON body**|Server returns `{ token: "eyJ..." }`, client stores it and sends via `Authorization: Bearer` header|Microservices, Postman, mobile apps|
|**HttpOnly cookie**|Server sets `Set-Cookie: jwt=eyJ...; HttpOnly; Secure`, browser sends it automatically|Browser SPAs|

> SPA: `fetch('/api/posts', { credentials: 'include' });` βÄî browser sends cookie automatically Microservice: `Authorization: Bearer eyJhbGciOi...` βÄî sends token in header manually

### HttpOnly Cookie + CSRF Risk

Storing JWT in an HttpOnly cookie protects against **XSS** (JavaScript can't read it), but introduces a **CSRF risk** βÄî the browser sends cookies automatically on every request to that domain, including requests triggered by malicious sites.

```
Attacker's site makes a request to your API
       βî²
       β•Φ
Browser automatically attaches your jwt cookie
       βî²
       β•Φ
Server sees valid JWT βÜ£ executes the request
       βî²
       β•Φ
CSRF attack succeeds βÄî attacker didn't need to read the cookie
```

#### How to Protect HttpOnly JWT Cookies from CSRF

Use `SameSite=Lax` or `SameSite=Strict` on the cookie, or enable CSRF token protection. See Page 1 (CSRF Prevention) for full breakdown of SameSite values and CSRF token mechanism.

> **Our implementation** disables CSRF and uses `STATELESS` sessions because the `JwtAuthFilter` also checks the `Authorization: Bearer` header. If your SPA only uses the cookie path, you should add `SameSite=Lax` or `SameSite=Strict` to the cookie, or enable CSRF token protection.

### Dual Delivery βÄî Trade-off Warning

Returning the token in **both** the JSON body **and** an HttpOnly cookie from the same endpoint defeats the purpose of HttpOnly βÄî JavaScript can read the token from the response body, making XSS protection useless.

**In production, keep the two flows separate:**

```
Browser SPA βÜ£ POST /api/auth/login
  Response: HttpOnly cookie only (no token in body)

Microservice βÜ£ POST /api/auth/login
  Response: Token in JSON body only (no cookie)
```

> **Our implementation** uses a single endpoint that does both for simplicity during development. See the trade-off comment in AuthService.
