### The Problem with a Single JWT Access Token

#### Current Setup

```
Login → Server creates JWT (24h expiry) → Client sends it on every request
```

This works, but has two problems:

#### Problem 1: Long-Lived Token = Bigger Risk

A token with 24h expiry is sent with **every request**. If stolen (via network interception, logs, XSS), the attacker has **24 hours** to use it freely. The server can't revoke it — JWT is stateless, there's no "invalidate" button.

#### Problem 2: Short-Lived Token = Bad UX

If you shorten expiry to 15 minutes for safety, the user gets logged out every 15 minutes and has to re-enter credentials. Terrible experience.

#### The Dilemma

```
Long expiry  → Good UX, Bad security
Short expiry → Good security, Bad UX
```

You can't have both with a single token.

---

### The Solution: Two Tokens

Instead of one token trying to do everything, split the responsibility:

|Token|Purpose|Lifetime|Sent With|
|---|---|---|---|
|**Access Token**|Authenticate API requests|Short (15–30 min)|Every request (`Authorization: Bearer`)|
|**Refresh Token**|Get a new access token|Long (7–30 days)|Only sent to `/api/auth/refresh`|

#### Why This Is Better

```
Access token stolen?
  → Attacker has 15 minutes, not 24 hours
  → Damage is limited

User's access token expired?
  → Frontend silently calls /refresh
  → Gets a new access token
  → User never notices
```

**Short-lived access token = security. Long-lived refresh token = UX. Best of both.**

### How It Works

#### Login

```
Client: POST /api/auth/login { email, password }
         │
         ▼
Server validates credentials
         │
         ▼
Server creates two tokens:
  ├── Access Token  (JWT, 15 min, returned in JSON body)
  └── Refresh Token (JWT, 7 days, HttpOnly cookie)
         │
         ▼
Client stores access token in memory (SPA) or env variable (Postman)
Browser stores refresh cookie automatically
```

#### API Calls

```
Client: GET /api/posts
        Authorization: Bearer <access_token>
         │
         ▼
JwtAuthFilter validates access token (from header only)
         │
         ▼
Valid → Request proceeds to controller
```

#### Token Refresh (Access Token Expired)

```
Client: POST /api/auth/refresh
        Cookie: refresh_token=eyJ... (sent automatically by browser)
         │
         ▼
Server validates refresh token against DB
         │
         ▼
Valid → Returns new access token in JSON body
        Sets new refresh cookie
         │
         ▼
Client stores new access token in memory
Client continues making API calls
```

#### Logout

```
Client: POST /api/auth/logout
         │
         ▼
Server invalidates refresh token
Server clears refresh cookie
         │
         ▼
Access token will expire on its own (15 min max)
```

### Refresh Token Storage: Stateless vs DB-Backed

#### Approach 1: Stateless (Refresh Token as JWT in HttpOnly Cookie)

```
Refresh token = another JWT with longer expiry
Stored only in HttpOnly cookie
Server doesn't store anything
```

|Pros|Cons|
|---|---|
|Simple — no DB table needed|Can't revoke a specific refresh token|
|No DB lookup on refresh|Can't do "logout from all devices"|
|Truly stateless|If stolen, valid until it expires|
|Fast — no DB hit|No way to track active sessions|

> **This is almost the same as a single JWT** — you've just split one long-lived token into two. The access token window is shorter (15 min vs 24h), so a stolen access token is less dangerous. But the refresh token itself has the same weakness as a single long-lived JWT — if stolen, it's valid until expiry and you can't revoke it. The real security jump comes with DB-backed approaches below.

#### Approach 2: DB-Backed (Refresh Token Stored in Database)

```
Refresh token = random string or JWT
Stored in a Session/RefreshToken table in DB
Server validates by checking DB
```

|Pros|Cons|
|---|---|
|Can revoke individual tokens|DB lookup on every refresh|
|Can invalidate all sessions (logout everywhere)|Not purely stateless|
|Can detect token reuse (security)|Extra table and cleanup logic|
|Track active sessions per user|Slightly more complex|

#### Approach 3: DB-Backed + Token Rotation (Recommended)

```
Same as Approach 2, but:
  → Every /refresh call returns a NEW refresh token
  → Old refresh token is invalidated immediately
  → If old token is reused → possible theft → invalidate ALL tokens for that user
```

|Pros|Cons|
|---|---|
|All benefits of Approach 2|Most complex to implement|
|Detects stolen refresh tokens|Requires careful concurrency handling|
|Most secure approach||
|Industry standard (OAuth2 spec recommends this)||

**How token reuse detection works:**

```
Normal flow:
  Refresh Token A → used → returns new Access Token + Refresh Token B
  Refresh Token B → used → returns new Access Token + Refresh Token C
  (Token A and B are now invalid)

Theft scenario:
  Attacker stole Token A
  Legitimate user uses Token A → gets Token B (A invalidated)
  Attacker tries Token A → already used!
  Server detects reuse → invalidates ALL tokens for this user
  Both attacker and user must re-login
```

### Isn't DB-Backed Refresh Token Just Sessions?

Similar idea (server stores something), but very different performance:

||Session-Based Auth|DB-Backed Refresh Token|
|---|---|---|
|DB hit on **every API call**|✅ Yes|❌ No (JWT signature only)|
|DB hit on **refresh**|Every request|Only every 15-30 min|
|What's stored in DB|Full session data|Just the refresh token string|
|Stateless API calls?|❌ Never|✅ Yes (99% of requests)|

> With sessions, **every single request** hits the DB. With refresh tokens, API calls are fully stateless — the DB is only touched during login, refresh (every 15-30 min), and logout. **99% stateless, 1% stateful.**

---

### What If the Refresh Token Is Stolen?

|Approach|What Happens|
|---|---|
|**Stateless**|Can't do anything — token works until it expires|
|**DB-backed**|Can revoke manually — user re-login, logout, or admin action deletes it from DB|
|**DB + Rotation**|Auto-detected — token reuse triggers invalidation of all tokens for that user|

DB-backed doesn't prevent theft, but gives you **tools to respond instantly**. DB + Rotation can **detect** it automatically.

### Comparison Summary

|Feature|Single JWT|Stateless Refresh|DB-Backed Refresh|DB + Rotation|
|---|---|---|---|---|
|Implementation complexity|Simple|Simple (marginal change)|Moderate|Complex|
|DB required for auth|No|No|Yes|Yes|
|Revoke specific token|❌|❌|✅|✅|
|Logout from all devices|❌|❌|✅|✅|
|Detect token theft|❌|❌|❌|✅|
|Stolen token risk|Full expiry window|Shorter for access, same for refresh|Can revoke|Can revoke + detect|
|Scalability|Best|Best|Good|Good|
|Security level|Low|Low–Medium|High|Highest|

### What Should You Pick?

```
Side project / learning → Single JWT or Stateless Refresh
Production app → DB-Backed Refresh
Sensitive app (banking, health) → DB-Backed + Token Rotation
```

#### Industry Standard (What Most Production Apps Use)

- **Access Token:** JWT, 15 min expiry, returned in JSON body, stored in memory, sent via `Authorization: Bearer` header
- **Refresh Token:** JWT, 7-30 days, stored in DB, sent in `HttpOnly + Secure + SameSite=Lax` cookie
- **On refresh:** Old refresh token invalidated, new one issued (rotation), new access token in body
- **On logout:** Refresh token deleted from DB, cookie cleared
- **Access token:** Not stored in DB — validated by signature only (stays stateless for API calls)
- **CSRF:** Not a concern — access token in header (not cookie), refresh cookie protected by SameSite=Lax

---

### Where Refresh Tokens Should Be Stored

**HttpOnly + Secure + SameSite=Lax cookie** — protects against both XSS and CSRF without needing a CSRF token mechanism.

> For detailed comparison of all storage options (localStorage, HttpOnly cookie, in-memory, Bearer header) and their XSS/CSRF trade-offs, see Fundamentals → Page 1 (CSRF section) and Page 5 (JWT Storage).

### Frontend Integration (How SPA Handles This)

> This is the "silent refresh" pattern mentioned in Fundamentals → Page 10 (Section 15). Here's the full flow:

```
App loads (first visit or browser refresh F5)
       │
       ▼
Access token in memory? ── No ──→ Call /api/auth/refresh
       │                                    │
      Yes                          Cookie sent automatically
       │                                    │
       ▼                                    ▼
Make API call                    Server validates refresh token
       │                                    │
       ▼                                    ▼
401 Unauthorized?                Returns new access token
       │                                    │
      Yes                           Store in memory
       │                                    │
       ▼                                    ▼
Call /api/auth/refresh           Retry original API call
       │
       ▼
Refresh also fails (401)?
       │
      Yes → Redirect to login page (session fully expired)
```

> The user never sees a login screen unless the refresh token itself has expired (after 7-30 days of inactivity).
