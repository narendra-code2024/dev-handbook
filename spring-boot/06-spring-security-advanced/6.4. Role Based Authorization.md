### Authentication vs Authorization

```
Authentication → Who are you?
  Verifying identity (email + password, OAuth2, JWT)
  Result: SecurityContext with authenticated user

Authorization → What can you do?
  Checking permissions to access resources
  Result: Allow (200) or Deny (403 Forbidden)
```

```
Request
  │
  ▼
Authentication (JwtAuthFilter)
  → Is this user valid?
  │
  ▼
Authorization (AuthorizationFilter)
  → Does this user have permission?
  │
  ├── Yes → Controller handles request
  └── No  → 403 Forbidden
```

### Roles vs Authorities

In Spring Security, **roles and authorities are the same thing** — both are strings stored in the `GrantedAuthority` list. The only difference is a naming convention:

||Authority|Role|
|---|---|---|
|Stored as|`ROLE_USER`, `ROLE_ADMIN`|Same — `ROLE_USER`, `ROLE_ADMIN`|
|Checked with|`hasAuthority("ROLE_USER")`|`hasRole("USER")`|
|Prefix|You include `ROLE_` yourself|Spring adds `ROLE_` automatically|

> `hasRole("ADMIN")` is just shorthand for `hasAuthority("ROLE_ADMIN")`. Spring automatically prepends `ROLE_` when using `hasRole()`.

```java
// These two are identical:
.hasAuthority("ROLE_ADMIN")
.hasRole("ADMIN")          // Spring checks for "ROLE_ADMIN" internally
```

**Important:** When storing roles in the database, always include the `ROLE_` prefix. Spring expects it.

```
DB stores: ROLE_USER, ROLE_ADMIN
hasRole("USER")      → checks for ROLE_USER ✅
hasRole("ROLE_USER") → checks for ROLE_ROLE_USER ❌ (double prefix)
hasAuthority("ROLE_USER") → checks for ROLE_USER ✅
```

### Implementation

#### Role Enum

```java
public enum Role {
    USER, ADMIN, MANAGER
}
```

> Stored as `USER`, `ADMIN` in DB. The `ROLE_` prefix is added in `getAuthorities()` — Spring expects it but we don't pollute our enum with it.

#### User Entity — Implement UserDetails

Instead of converting User to UserDetails in `CustomUserDetailsService`, you can make User implement `UserDetails` directly:

```java
@Entity
@Table(name = "users")
@Getter @Setter @NoArgsConstructor @AllArgsConstructor @Builder
public class User implements UserDetails {

    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    @Column(nullable = false)
    private String firstName;

    @Column(nullable = false)
    private String lastName;

    @Column(nullable = false, unique = true)
    private String email;

    @Column(nullable = true)
    private String password;

    @ElementCollection(fetch = FetchType.EAGER)
    @CollectionTable(name = "user_role",
        joinColumns = @JoinColumn(name = "user_id"))
    @Enumerated(EnumType.STRING)
    @Column(name = "role")
    private Set<Role> roles;

    @Override
    public Collection<? extends GrantedAuthority> getAuthorities() {
        return roles.stream()
            .map(role -> new SimpleGrantedAuthority("ROLE_" + role.name()))
            .collect(Collectors.toList());
    }

    @Override
    public String getUsername() {
        return email;
    }
}
```

> `getAuthorities()` adds the `ROLE_` prefix: `Role.ADMIN` → `"ROLE_ADMIN"`. This way your enum stays clean (`USER`, `ADMIN`) and Spring gets the prefix it expects.

#### CustomUserDetailsService (Simplified)

Since User now implements UserDetails, the service just returns the User directly:

```java
@Service
@RequiredArgsConstructor
public class CustomUserDetailsService implements UserDetailsService {

    private final UserRepository userRepository;

    @Override
    public UserDetails loadUserByUsername(String email)
            throws UsernameNotFoundException {
        return userRepository.findByEmail(email)
            .orElseThrow(() ->
                new UsernameNotFoundException("Invalid credentials"));
    }
}
```

> No more manual conversion — `User` is already a `UserDetails`.

### SecurityConfig — Request Matchers

#### Common Authorization Methods

|Method|Purpose|Example|
|---|---|---|
|`permitAll()`|Anyone (no auth needed)|Public endpoints, login, register|
|`authenticated()`|Any logged-in user|General protected endpoints|
|`hasRole("ADMIN")`|User must have `ROLE_ADMIN`|Admin panel|
|`hasAnyRole("ADMIN", "MANAGER")`|User must have any of these roles|Management endpoints|
|`hasAuthority("ROLE_ADMIN")`|Same as `hasRole("ADMIN")`|Explicit prefix version|
|`hasAnyAuthority(...)`|Same as `hasAnyRole()` with prefix|Explicit prefix version|
|`denyAll()`|Block everyone|Disabled endpoints|

#### SecurityConfig (Updated)

```java
@Bean
public SecurityFilterChain securityFilterChain(HttpSecurity http)
        throws Exception {

    http
        .csrf(csrf -> csrf.disable())

        .sessionManagement(session ->
            session.sessionCreationPolicy(SessionCreationPolicy.STATELESS))

        .authorizeHttpRequests(auth -> auth
            // Public
            .requestMatchers("/api/auth/register").permitAll()
            .requestMatchers("/api/auth/login").permitAll()
            .requestMatchers("/api/auth/refresh").permitAll()

            // Role-based
            .requestMatchers("/api/admin/**").hasRole("ADMIN")
            .requestMatchers("/api/management/**")
                .hasAnyRole("ADMIN", "MANAGER")

            // Method-specific
            .requestMatchers(HttpMethod.GET, "/api/posts/**").permitAll()
            .requestMatchers(HttpMethod.POST, "/api/posts/**")
                .hasRole("USER")
            .requestMatchers(HttpMethod.DELETE, "/api/posts/**")
                .hasRole("ADMIN")

            // Catch-all
            .anyRequest().authenticated()
        )

        .oauth2Login(oauth2 -> oauth2
            .failureUrl("/login?error=true")
            .successHandler(oAuth2SuccessHandler)
        )

        .addFilterBefore(jwtAuthFilter,
            UsernamePasswordAuthenticationFilter.class);

    return http.build();
}
```

> **Order matters.** Spring evaluates rules top to bottom — first match wins. Put specific rules before `.anyRequest().authenticated()`.

### Request Matcher Patterns

|Pattern|Matches|
|---|---|
|`/api/admin`|Exactly `/api/admin`|
|`/api/admin/*`|One level: `/api/admin/users`, `/api/admin/settings`|
|`/api/admin/**`|All levels: `/api/admin/users`, `/api/admin/users/1/roles`|

### How It Works End-to-End

```
GET /api/admin/users
Authorization: Bearer <token>
       │
       ▼
JwtAuthFilter
  → Extracts token → validates → loads User from DB
  → User has roles: [ROLE_USER]
  → Sets SecurityContext
       │
       ▼
AuthorizationFilter
  → /api/admin/** requires hasRole("ADMIN")
  → User has ROLE_USER, not ROLE_ADMIN
  → 403 Forbidden ❌
```

```
GET /api/admin/users
Authorization: Bearer <token>
       │
       ▼
JwtAuthFilter
  → User has roles: [ROLE_USER, ROLE_ADMIN]
  → Sets SecurityContext
       │
       ▼
AuthorizationFilter
  → /api/admin/** requires hasRole("ADMIN")
  → User has ROLE_ADMIN
  → Request proceeds to controller ✅
```

### Assigning Roles

#### During Registration

```java
User user = User.builder()
    .email(request.getEmail())
    .password(passwordEncoder.encode(request.getPassword()))
    .roles(Set.of(Role.USER))  // Default role
    .build();
```

#### Admin Assigning Roles

```java
public void assignRole(Long userId, Role role) {
    User user = userRepository.findById(userId).orElseThrow();
    user.getRoles().add(role); // e.g., Role.ADMIN
    userRepository.save(user);
}
```

> Role changes take effect on the **next token refresh** — the current access token still has the old roles encoded. When the user refreshes, `loadUserByUsername()` fetches updated roles from DB.

### Getting Current User Info in Controller

```java
// Get email
Authentication auth = SecurityContextHolder.getContext().getAuthentication();
String email = auth.getName();

// Check role programmatically
boolean isAdmin = auth.getAuthorities().stream()
    .anyMatch(a -> a.getAuthority().equals("ROLE_ADMIN"));

// Or inject directly
@GetMapping("/profile")
public ResponseEntity<UserResponse> profile(
        @AuthenticationPrincipal User user) {
    // user is your User entity (implements UserDetails)
    // user.getRoles() returns Set<Role> — clean enum values
    return ResponseEntity.ok(userMapper.toDto(user));
}
```

> `@AuthenticationPrincipal` extracts the User directly from SecurityContext — no need for manual `SecurityContextHolder` calls.
