### Role vs Authority (Permission)

||Role|Authority (Permission)|
|---|---|---|
|Purpose|High-level group|Fine-grained access right|
|Granularity|Broad|Specific|
|Examples|`USER`, `ADMIN`, `MANAGER`|`POST_CREATE`, `POST_DELETE`, `USER_VIEW`|
|Checked with|`hasRole("ADMIN")`|`hasAuthority("POST_DELETE")`|
|Prefix|Spring adds `ROLE_` automatically|No prefix — used as-is|

**Roles** answer: "What type of user is this?" **Permissions** answer: "What specific actions can this user perform?"

---

### Why Permissions?

With roles alone:

```java
.requestMatchers(HttpMethod.DELETE, "/api/posts/**").hasRole("ADMIN")
```

This works, but what if managers should also delete posts? You'd add `hasAnyRole("ADMIN", "MANAGER")`. Then product owners need it too. The rules keep growing and changing.

With permissions:

```java
.requestMatchers(HttpMethod.DELETE, "/api/posts/**").hasAuthority("POST_DELETE")
```

Now you just assign `POST_DELETE` to whichever roles need it. The security rule never changes — only the role-permission mapping does.

```
Without permissions:
  Rule knows about roles → rule changes when roles change

With permissions:
  Rule knows about permissions → role-permission mapping changes
  Rule stays stable
```

---

### How It Works — Roles Own Permissions

```
Role: ADMIN
  └── Permissions: POST_CREATE, POST_UPDATE, POST_DELETE,
                   USER_VIEW, USER_CREATE, USER_DELETE

Role: MANAGER
  └── Permissions: POST_CREATE, POST_UPDATE, POST_DELETE,
                   USER_VIEW

Role: USER
  └── Permissions: POST_CREATE, POST_UPDATE
```

When a user with `Role.MANAGER` authenticates, Spring sees their authorities as:

```
[ROLE_MANAGER, POST_CREATE, POST_UPDATE, POST_DELETE, USER_VIEW]
   └── role          └── permissions from that role
```

Both the role and its permissions are loaded as authorities — you can check either.

### Implementation

#### Permission Enum

```java
public enum Permission {
    POST_CREATE,
    POST_UPDATE,
    POST_DELETE,
    USER_VIEW,
    USER_CREATE,
    USER_DELETE
}
```

#### Role Enum (Updated)

Each role now carries its own set of permissions:

```java
@Getter
@RequiredArgsConstructor
public enum Role {

    USER(Set.of(
        Permission.POST_CREATE,
        Permission.POST_UPDATE
    )),

    MANAGER(Set.of(
        Permission.POST_CREATE,
        Permission.POST_UPDATE,
        Permission.POST_DELETE,
        Permission.USER_VIEW
    )),

    ADMIN(Set.of(
        Permission.POST_CREATE,
        Permission.POST_UPDATE,
        Permission.POST_DELETE,
        Permission.USER_VIEW,
        Permission.USER_CREATE,
        Permission.USER_DELETE
    ));

    private final Set<Permission> permissions;
}
```

> Adding a new permission to a role is a one-line change here. No security rules or config needs to change.

---

#### User Entity — getAuthorities (Updated)

Previously returned only `ROLE_` authorities. Now returns **both** the role and its permissions:

```java
@Override
public Collection<? extends GrantedAuthority> getAuthorities() {
    List<GrantedAuthority> authorities = new ArrayList<>();

    for (Role role : roles) {
        // Add the role itself (e.g., ROLE_ADMIN)
        authorities.add(new SimpleGrantedAuthority("ROLE_" + role.name()));

        // Add all permissions for that role (e.g., POST_CREATE)
        role.getPermissions().forEach(permission ->
            authorities.add(new SimpleGrantedAuthority(permission.name())));
    }

    return authorities;
}
```

**What this produces for a user with `Role.MANAGER`:**

```
Authorities: [ROLE_MANAGER, POST_CREATE, POST_UPDATE, POST_DELETE, USER_VIEW]
```

> Roles get `ROLE_` prefix, permissions don't. This is how Spring distinguishes them — `hasRole("MANAGER")` looks for `ROLE_MANAGER`, `hasAuthority("POST_DELETE")` looks for `POST_DELETE` directly.

---

#### SecurityConfig (Updated)

Now you can mix role-based and permission-based rules:

```java
.authorizeHttpRequests(auth -> auth
    // Public
    .requestMatchers("/api/auth/**").permitAll()

    // Permission-based (fine-grained)
    .requestMatchers(HttpMethod.POST, "/api/posts/**")
        .hasAuthority("POST_CREATE")
    .requestMatchers(HttpMethod.PUT, "/api/posts/**")
        .hasAuthority("POST_UPDATE")
    .requestMatchers(HttpMethod.DELETE, "/api/posts/**")
        .hasAuthority("POST_DELETE")

    // Role-based (broad)
    .requestMatchers("/api/admin/**").hasRole("ADMIN")

    // Catch-all
    .anyRequest().authenticated()
)
```

> **Permission-based rules don't change** when you reorganize roles. If you later decide `USER` should also delete posts, just add `POST_DELETE` to the `USER` enum — no SecurityConfig change needed.

---

### Role-Based vs Permission-Based — When to Use Which

|Use|When|
|---|---|
|`hasRole("ADMIN")`|Entire sections restricted to a role (admin panel, management dashboard)|
|`hasAuthority("POST_DELETE")`|Specific actions that might span multiple roles|
|Both together|Most real apps — broad role checks for sections, fine-grained permission checks for actions|

### Checking Permissions in Code

#### In Controller (Programmatic)

```java
@GetMapping("/admin/users")
public ResponseEntity<List<UserResponse>> getUsers(
        @AuthenticationPrincipal User user) {

    // Check permission programmatically
    boolean canView = user.getAuthorities().stream()
        .anyMatch(a -> a.getAuthority().equals("USER_VIEW"));

    if (!canView) {
        throw new AccessDeniedException("No permission");
    }

    return ResponseEntity.ok(userService.getAllUsers());
}
```

#### With Method Annotations (Cleaner — covered in Page 6.6)

```java
@PreAuthorize("hasAuthority('USER_VIEW')")
@GetMapping("/admin/users")
public ResponseEntity<List<UserResponse>> getUsers() {
    return ResponseEntity.ok(userService.getAllUsers());
}
```

> Method annotations are the preferred way — keeps controllers clean. See Page 6.6 for `@PreAuthorize` and `@Secured`.

### Approaches to Permission Management

|Approach|How It Works|Best For|
|---|---|---|
|**Enum (our approach)**|Roles + permissions defined in code. `@ElementCollection` stores role enums in a join table — no Role/Permission entities needed.|Most apps — permissions are known at design time, simple and compile-safe|
|**DB entities**|`Role`, `Permission`, `UserRole`, `RolePermission` entities with `@ManyToMany` relationships. Managed via API/admin panel.|Apps needing runtime permission changes and custom admin UI|
|**Keycloak / Auth0**|External identity provider handles users, roles, permissions, SSO, admin console. Your app just reads authorities from JWT claims.|Large enterprise, multi-tenant, or when you need dynamic permission management without building it yourself|

> Even with DB-backed permissions, **adding a new permission still requires redeploy** to use it in code (`@PreAuthorize`, SecurityConfig). What DB permissions really provide is dynamic **role → permission mapping** — changing who can access existing features without code changes.

> **Our approach** uses `@ElementCollection` with the `Role` enum — the `user_role` join table is created via migration, and Hibernate manages it through the `User` entity. No `Role` entity, no `UserRole` entity needed — the join table has no model, similar to a pivot table in Laravel. This is the standard for enum-based roles where the values are fixed and defined in code.
