### Method-Level vs URL-Level Security

```
Request
  │
  ▼
URL-Level Security (SecurityConfig - requestMatchers)
  → Who can reach this endpoint?
  │
  ▼
Controller
  │
  ▼
Method-Level Security (@PreAuthorize, @Secured)
  → Who can execute this specific method?
  │
  ▼
Service Layer
```

||URL-Level (Request Matchers)|Method-Level (@PreAuthorize, @Secured)|
|---|---|---|
|Where|SecurityConfig|On any method (controller, service)|
|Granularity|Broad — URL patterns|Fine-grained — per method|
|Checks|Roles, authentication|Roles, permissions, custom logic, method parameters|
|Use for|"Only authenticated users can access `/api/**`"|"Only the wallet owner can view this wallet"|

> Use **request matchers** to secure endpoints broadly (authentication, role-based URL patterns). Use **method annotations** for specific business rules inside controllers or services.

### Enabling Method Security

Method annotations are **disabled by default**. Add this to your config:

```java
@Configuration
@EnableWebSecurity
@EnableMethodSecurity(securedEnabled = true)
public class SecurityConfig {
    // ...
}
```

|Parameter|What It Enables|Default|
|---|---|---|
|`prePostEnabled`|`@PreAuthorize`, `@PostAuthorize`|`true` (enabled by default in `@EnableMethodSecurity`)|
|`securedEnabled`|`@Secured`|`false` (must enable explicitly)|

> `@PreAuthorize` works as soon as you add `@EnableMethodSecurity`. You only need `securedEnabled = true` if you also want to use `@Secured`.

### @Secured

Simple role-based check at method level. Only supports role names — no expressions.

```java
@Secured("ROLE_USER")
@GetMapping("/profile")
public ResponseEntity<UserResponse> getProfile() {
    // only users with ROLE_USER can access
}

@Secured({"ROLE_ADMIN", "ROLE_MANAGER"})
@DeleteMapping("/posts/{id}")
public ResponseEntity<Void> deletePost(@PathVariable Long id) {
    // ADMIN or MANAGER can access
}
```

|Pros|Cons|
|---|---|
|Simple|Roles only — no permissions, no expressions|
|Easy to read|Can't access method parameters|
||Can't use custom logic|

> Must pass the full `ROLE_` prefix — `@Secured("ROLE_ADMIN")`, not `@Secured("ADMIN")`.

### @PreAuthorize

Expression-based check — much more powerful than `@Secured`. Runs **before** the method executes.

#### Check Role

```java
@PreAuthorize("hasRole('ADMIN')")
@GetMapping("/admin/users")
public ResponseEntity<List<UserResponse>> getAllUsers() {
    // hasRole adds ROLE_ prefix automatically
}
```

#### Check Permission (Authority)

```java
@PreAuthorize("hasAuthority('POST_DELETE')")
@DeleteMapping("/posts/{id}")
public ResponseEntity<Void> deletePost(@PathVariable Long id) {
    // checks for POST_DELETE in authorities list
}
```

#### Multiple Conditions

```java
@PreAuthorize("hasRole('ADMIN') or hasRole('MANAGER')")

@PreAuthorize("hasAuthority('POST_CREATE') and hasAuthority('POST_UPDATE')")

@PreAuthorize("hasAnyRole('ADMIN', 'MANAGER')")

@PreAuthorize("hasAnyAuthority('POST_CREATE', 'POST_UPDATE')")
```

#### Access Method Parameters

```java
@PreAuthorize("#userId == authentication.principal.id")
@GetMapping("/users/{userId}")
public ResponseEntity<UserResponse> getUser(@PathVariable Long userId) {
    // user can only access their own profile
}
```

> `#userId` refers to the method parameter. `authentication.principal` is the current logged-in user.

#### Custom Bean Method

For complex business logic, call a method on a Spring bean:

```java
@PreAuthorize("@postSecurity.isOwner(#postId)")
@PutMapping("/posts/{postId}")
public ResponseEntity<PostResponse> updatePost(@PathVariable Long postId,
        @RequestBody PostRequest request) {
    // only the post owner can update
}
```

```java
@Component("postSecurity")
@RequiredArgsConstructor
public class PostSecurity {

    private final PostRepository postRepository;

    public boolean isOwner(Long postId) {
        Authentication auth = SecurityContextHolder.getContext()
            .getAuthentication();
        User user = (User) auth.getPrincipal();

        return postRepository.findById(postId)
            .map(post -> post.getUser().getId().equals(user.getId()))
            .orElse(false);
    }
}
```

> `@postSecurity` refers to the bean name. Spring evaluates `isOwner(#postId)` — if it returns `false`, access is denied with 403.

### @Secured vs @PreAuthorize

|Feature|@Secured|@PreAuthorize|
|---|---|---|
|Role check|✅|✅|
|Permission check|❌|✅|
|SpEL expressions|❌|✅|
|Access method params|❌|✅ (`#paramName`)|
|Custom bean method|❌|✅ (`@beanName.method()`)|
|Multiple conditions (AND/OR)|❌|✅|
|Must include `ROLE_` prefix|Yes|No (`hasRole` adds it)|
|Needs explicit enable|`securedEnabled = true`|Enabled by default|

> **Use `@PreAuthorize` in most cases** — it does everything `@Secured` does and more. `@Secured` exists mainly for legacy compatibility.

### 401 Unauthorized vs 403 Forbidden

||401 Unauthorized|403 Forbidden|
|---|---|---|
|Means|"I don't know who you are"|"I know who you are, but you can't do this"|
|Cause|Missing or invalid token|Valid token but insufficient permissions|
|When|No token, expired token, bad signature|User lacks required role or permission|
|Fix|Login again, refresh token|Need different role/permission|

```
Request with no token
  → 401 Unauthorized (not authenticated)

Request with valid token, user has ROLE_USER
  → GET /api/posts       → 200 OK (authenticated is enough)
  → GET /api/admin/users → 403 Forbidden (needs ROLE_ADMIN)

Request with expired token
  → 401 Unauthorized (token invalid)
```

#### Where They're Thrown

```
Request
  │
  ▼
JwtAuthFilter
  ├── No token → continues (no SecurityContext set)
  ├── Invalid/expired token → 401 Unauthorized
  └── Valid token → sets SecurityContext
  │
  ▼
AuthorizationFilter
  ├── No SecurityContext (unauthenticated) → 401 Unauthorized
  ├── Has SecurityContext but wrong role/permission → 403 Forbidden
  └── Has required role/permission → proceeds to controller
  │
  ▼
@PreAuthorize / @Secured
  ├── Check fails → 403 Forbidden
  └── Check passes → method executes
```

### Layered Security — Putting It All Together

```java
// SecurityConfig — URL level
.authorizeHttpRequests(auth -> auth
    .requestMatchers("/api/auth/**").permitAll()
    .requestMatchers("/api/admin/**").hasRole("ADMIN")
    .anyRequest().authenticated()   // must be logged in
)
```

```java
// Controller — method level
@PreAuthorize("hasAuthority('USER_VIEW')")
@GetMapping("/admin/users")
public ResponseEntity<List<UserResponse>> getAllUsers() {
    return ResponseEntity.ok(userService.getAllUsers());
}
```

```java
// Service — business logic level
@PreAuthorize("@postSecurity.isOwner(#postId)")
public PostResponse updatePost(Long postId, PostRequest request) {
    // ...
}
```

```
Layer 1: URL security    → "Is this user authenticated? Right role for this URL?"
Layer 2: Method security → "Does this user have the specific permission?"
Layer 3: Business logic  → "Is this user the owner of this resource?"
```

> Each layer adds more specific checks. A request must pass **all layers** to succeed.
