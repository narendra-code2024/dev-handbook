### Where Testing Fits — SDLC

```
Requirement Analysis
       │
       ▼
    Defining
       │
       ▼
   Designing
       │
       ▼
    Coding
       │
       ▼
    Testing  ← we are here
       │
       ▼
   Deployment
       │
       ▼
  Maintenance
       │
       └──→ (cycle repeats)
```

Testing isn't a one-time phase — in modern development, you write tests **alongside** code, and tests run automatically before every deployment.

---

### Why Testing Matters

- **Identifies bugs early** — cheaper to fix during development than after deployment
- **Mitigates deployment risks** — tests catch breaking changes before they reach production
- **Increases productivity** — refactor with confidence, tests tell you if something broke
- **Builds trust** — developers and teams trust code that has tests backing it

### Test Writing Approaches

#### TDD — Test-Driven Development

Write tests **before** writing the actual code:

```
1. Write a failing test (Red)
       │
       ▼
2. Write minimum code to pass the test (Green)
       │
       ▼
3. Refactor the code for better design (Refactor)
       │
       └──→ repeat
```

> The "Red-Green-Refactor" cycle. You start with what the code **should do**, then make it work.

#### BDD — Behavior-Driven Development

An extension of TDD — tests are written in human-readable language describing **behavior**:

```
Given: a user with email "john@example.com" exists
When:  I call GET /api/users/1
Then:  I should receive the user's details with status 200
```

> BDD focuses on **what the system does** from the user's perspective, not how it works internally.

#### Other Approaches

|Approach|How It Works|
|---|---|
|**TDD**|Write test first → then code|
|**BDD**|Define behavior first → write scenarios → then code|
|**Test-After Development**|Write code first → then add tests|
|**Simultaneous Development**|Write code and tests together|

> TDD and BDD produce the most reliable tests, but Test-After is the most common in practice — most teams write tests after implementing features.

### Spring Boot Test Example

A quick look at what a test looks like — details covered in upcoming pages:

```java
class EmployeeServiceTest {

    @Mock
    private EmployeeRepository employeeRepository;

    @InjectMocks
    private EmployeeServiceImpl employeeService;

    @Test
    void testGetEmployeeById() {
        Employee employee = new Employee(1L, "John Doe",
            "john.doe@example.com");
        when(employeeRepository.findById(1L))
            .thenReturn(Optional.of(employee));

        EmployeeDto result = employeeService.getEmployeeById(1L);

        assertThat(result.getName()).isEqualTo("John Doe");
        assertThat(result.getEmail()).isEqualTo("john.doe@example.com");
    }
}
```

|Part|What It Does|
|---|---|
|`@Mock`|Creates a fake `EmployeeRepository` (no real DB)|
|`@InjectMocks`|Creates `EmployeeServiceImpl` with the mock injected|
|`when(...).thenReturn(...)`|"When findById is called, return this fake data"|
|`assertThat(...).isEqualTo(...)`|Verify the result is what we expect|

> This is a **unit test** — it tests the service in isolation without a database. We'll cover mocking (Mockito), assertions (AssertJ), and different test types in the next pages.
