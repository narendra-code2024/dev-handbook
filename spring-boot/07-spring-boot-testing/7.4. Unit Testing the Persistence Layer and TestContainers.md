### What We're Testing

Repository methods — custom queries like `findByEmail`, derived queries, native queries. We need a real database for this (not mocks), because we're testing that the **queries actually work**.

---

### Creating Test Classes

In your IDE, go to the repository class → `Cmd+Shift+T` (Mac) or `Ctrl+Shift+T` (Windows) → create test class.

Test files follow the **same package structure** as main:

```
src/main/java/com/app/repository/EmployeeRepository.java
src/test/java/com/app/repository/EmployeeRepositoryTest.java
```

#### Test Naming

Name tests to describe **what they test and what they expect**:

```java
// Pattern: test_[method]_[scenario]_[expectedResult]
void testFindByEmail_WhenEmailIsValid_ThenReturnEmployee()
void testFindByEmail_WhenEmailNotFound_ThenReturnEmpty()
```

> Good test names act as documentation — you can understand what the code does just by reading test names.

---

### @DataJpaTest

Loads **only JPA components** — repositories, EntityManager, and an in-memory database. Does not load controllers, services, or security filters.

```java
@DataJpaTest
class EmployeeRepositoryTest {

    @Autowired
    private EmployeeRepository employeeRepository;

    @Test
    void testFindByEmail_WhenEmailIsValid_ThenReturnEmployee() {
        // Given (Arrange)
        Employee employee = Employee.builder()
            .firstName("John")
            .lastName("Doe")
            .email("john@example.com")
            .build();
        employeeRepository.save(employee);

        // When (Act)
        Optional<Employee> result = employeeRepository
            .findByEmail("john@example.com");

        // Then (Assert)
        assertThat(result).isPresent();
        assertThat(result.get().getFirstName()).isEqualTo("John");
        assertThat(result.get().getEmail()).isEqualTo("john@example.com");
    }

    @Test
    void testFindByEmail_WhenEmailNotFound_ThenReturnEmpty() {
        // When
        Optional<Employee> result = employeeRepository
            .findByEmail("nonexistent@example.com");

        // Then
        assertThat(result).isEmpty();
    }
}
```

#### Test Case Pattern — Given / When / Then

|Step|Also Called|Purpose|
|---|---|---|
|**Given**|Arrange|Set up test data|
|**When**|Act|Execute the method being tested|
|**Then**|Assert|Verify the result|

#### Key Behaviors of @DataJpaTest

- **Transactional by default** — each test runs in a transaction that is **rolled back** after the test. Tests don't affect each other, clean state every time.
- **Auto-configures in-memory DB** — by default uses H2 if it's on the classpath.
- **Only loads JPA layer** — faster than `@SpringBootTest` because it skips web, service, security layers.

---

### Application Context Caching

When you use `@SpringBootTest` or `@DataJpaTest`, Spring **caches** the application context after the first test method loads it. Other test methods in the same class reuse the cached context — they don't reload it. This makes subsequent tests faster.

---

### The Problem with H2

By default, `@DataJpaTest` uses **H2** (in-memory database). H2 is simple and fast, but it doesn't support everything your production database does:

```
Production: PostgreSQL
  → supports JSONB, arrays, window functions, specific SQL syntax

H2 in tests:
  → may not support these features
  → tests pass on H2 but fail on real PostgreSQL
```

> Tests should run against a database as close to production as possible. Creating a dedicated test database is overkill — you don't need a permanent DB just for tests. This is where **TestContainers** comes in.

---

### TestContainers

Spins up a **real database in a Docker container** during tests. Your tests run against actual PostgreSQL (or MySQL, MongoDB, etc.) — same as production. The container starts before tests, runs during tests, and is destroyed after.

```
Without TestContainers:
  Tests → H2 (not like production) → may miss real bugs

With TestContainers:
  Tests → Docker PostgreSQL (same as production) → catches real bugs
```

#### Prerequisites

Docker must be installed and running. Download from https://docker.com

#### Step 1: Add Dependencies

```xml
<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-testcontainers</artifactId>
    <scope>test</scope>
</dependency>
<dependency>
    <groupId>org.testcontainers</groupId>
    <artifactId>postgresql</artifactId>
    <scope>test</scope>
</dependency>
```

#### Step 2: Create Test Configuration

```java
@TestConfiguration(proxyBeanMethods = false)
public class TestContainersConfiguration {

    @Bean
    @ServiceConnection
    PostgreSQLContainer<?> postgresContainer() {
        return new PostgreSQLContainer<>(
            DockerImageName.parse("postgres:latest"));
    }
}
```

> `@ServiceConnection` tells Spring Boot to automatically configure the datasource using this container — no manual URL/username/password setup needed.

#### Step 3: Use in Tests

```java
@DataJpaTest
@Import(TestContainersConfiguration.class)
@AutoConfigureTestDatabase(replace = AutoConfigureTestDatabase.Replace.NONE)
class EmployeeRepositoryTest {

    @Autowired
    private EmployeeRepository employeeRepository;

    @Test
    void testFindByEmail_WhenEmailIsValid_ThenReturnEmployee() {
        // Given
        Employee employee = Employee.builder()
            .firstName("John")
            .lastName("Doe")
            .email("john@example.com")
            .build();
        employeeRepository.save(employee);

        // When
        Optional<Employee> result = employeeRepository
            .findByEmail("john@example.com");

        // Then
        assertThat(result).isPresent();
        assertThat(result.get().getEmail()).isEqualTo("john@example.com");
    }
}
```

#### Why @AutoConfigureTestDatabase(replace = NONE)?

`@DataJpaTest` tries to replace your datasource with H2 by default. Since we want to use the TestContainers PostgreSQL instead, we tell Spring **don't replace** the datasource.

|Value|Behavior|
|---|---|
|`ANY` (default)|Replace the datasource — uses H2|
|`NONE`|Don't replace — use whatever is configured (TestContainers)|
|`AUTO_CONFIGURED`|Only replace if it was auto-configured|

> Either remove H2 from `pom.xml` entirely, or set `replace = NONE`. If both H2 and TestContainers are present without this setting, Spring will use H2 and ignore your container.

---

### H2 vs TestContainers

||H2|TestContainers|
|---|---|---|
|Setup|Zero — just add dependency|Needs Docker installed|
|Speed|Very fast|Slower (container startup)|
|Matches production|❌ Different SQL dialect|✅ Same database as production|
|Catches real bugs|May miss DB-specific issues|✅ Catches them|
|Use when|Quick prototyping, simple queries|Production projects, complex queries|

### Running Tests with Coverage

Right-click on the test class → **Run with Coverage**. Aim for high coverage on repository methods — every custom query should have at least a success and failure test case.
