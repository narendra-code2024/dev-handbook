### What is Mockito?

When unit testing a service, you don't want to use the real repository (that needs a DB) or real external APIs. **Mockito** creates fake (mock) objects that simulate the behavior of real dependencies — so you test **only the service logic** in isolation.

```
Without Mockito:
  ServiceTest → Service → Real Repository → Real DB
  (slow, fragile, not a unit test)

With Mockito:
  ServiceTest → Service → Mock Repository (no DB)
  (fast, isolated, true unit test)
```

---

### Three Core Concepts

|Concept|What It Does|Example|
|---|---|---|
|**Mocking**|Creates fake objects that look like real ones|Fake `EmployeeRepository`|
|**Stubbing**|Defines what the mock returns when a method is called|"When `findById(1L)` is called, return this employee"|
|**Verification**|Checks that certain methods were actually called|"Verify `save()` was called exactly once"|

### Setting Up Mockito

```java
@ExtendWith(MockitoExtension.class)
class EmployeeServiceTest {

    @Mock
    private EmployeeRepository employeeRepository;

    @Spy
    private EmployeeMapper employeeMapper = new EmployeeMapperImpl();

    @InjectMocks
    private EmployeeServiceImpl employeeService;
}
```

|Annotation|What It Does|
|---|---|
|`@ExtendWith(MockitoExtension.class)`|Enables Mockito annotations in the test class|
|`@Mock`|Creates a fake object — all methods return null/default by default|
|`@Spy`|Wraps a **real object** — calls real methods unless explicitly stubbed|
|`@InjectMocks`|Creates the real service and injects `@Mock` and `@Spy` dependencies into it|

#### @Mock vs @Spy

```java
@Mock
private EmployeeRepository repository;
// repository.findById(1L) → returns null (fake, no real logic)

@Spy
private EmployeeMapper mapper = new EmployeeMapperImpl();
// mapper.toDto(employee) → calls REAL mapping logic
// But you CAN override: when(mapper.toDto(any())).thenReturn(...)
```

> Use `@Mock` for dependencies you want to **fully control** (repositories, external services). Use `@Spy` for dependencies whose **real logic you want to keep** (mappers, utility classes).

#### Alternative — Creating Mocks Programmatically

```java
// Instead of @Mock annotation
EmployeeRepository repository = Mockito.mock(EmployeeRepository.class);
```

> Both approaches do the same thing. Annotations are cleaner.

### Stubbing — Defining Mock Behavior

#### `when(...).thenReturn(...)`

"When this method is called, return this value."

```java
Employee employee = new Employee(1L, "John", "john@example.com");

when(employeeRepository.findById(1L))
    .thenReturn(Optional.of(employee));

when(employeeRepository.findByEmail("john@example.com"))
    .thenReturn(Optional.of(employee));

when(employeeRepository.save(any(Employee.class)))
    .thenReturn(employee);
```

#### `when(...).thenThrow(...)`

"When this method is called, throw this exception."

```java
when(employeeRepository.findById(999L))
    .thenThrow(new ResourceNotFoundException("Employee not found"));
```

#### Stubbing Summary

|Method|Purpose|
|---|---|
|`when(mock.method()).thenReturn(value)`|Return a specific value|
|`when(mock.method()).thenThrow(exception)`|Throw an exception|
|`when(mock.method()).thenReturn(first, second)`|Return different values on consecutive calls|

### Verification — Checking Methods Were Called

After executing your service method, verify that the mock's methods were called as expected.

#### `verify(mock).method()`

```java
// Verify save was called exactly once
verify(employeeRepository).save(any(Employee.class));

// Verify findById was called with argument 1L
verify(employeeRepository).findById(1L);
```

#### Verification Modes

```java
// Called exactly 2 times
verify(employeeRepository, times(2)).findById(anyLong());

// Never called
verify(employeeRepository, never()).delete(any());

// Called at least once
verify(employeeRepository, atLeastOnce()).save(any());

// Called at least 3 times
verify(employeeRepository, atLeast(3)).findById(anyLong());

// Called at most 5 times
verify(employeeRepository, atMost(5)).findById(anyLong());

// This was the ONLY method called on the mock
verify(employeeRepository, only()).save(any());
```

|Mode|Checks|
|---|---|
|`times(n)`|Exactly n times|
|`never()`|Zero times|
|`atLeastOnce()`|At least 1 time|
|`atLeast(n)`|At least n times|
|`atMost(n)`|At most n times|
|`only()`|This method and no other method was called|

### Argument Captor — Inspecting Arguments

Sometimes you need to check **what arguments** were passed to a mock method — not just that it was called.

```java
@Captor
private ArgumentCaptor<Employee> employeeCaptor;

@Test
void shouldSaveEmployeeWithCorrectData() {
    EmployeeDto request = new EmployeeDto("John", "john@example.com");

    when(employeeRepository.save(any(Employee.class)))
        .thenReturn(new Employee(1L, "John", "john@example.com"));

    employeeService.createEmployee(request);

    // Capture what was passed to save()
    verify(employeeRepository).save(employeeCaptor.capture());

    Employee captured = employeeCaptor.getValue();
    assertThat(captured.getFirstName()).isEqualTo("John");
    assertThat(captured.getEmail()).isEqualTo("john@example.com");
}
```

> Useful when the service **transforms** data before saving — you want to verify the transformation was correct.

### Argument Matchers

Instead of exact values, use matchers for flexible stubbing and verification:

```java
// any() — matches any value
when(employeeRepository.save(any(Employee.class))).thenReturn(employee);

// anyLong(), anyString() — type-specific
when(employeeRepository.findById(anyLong())).thenReturn(Optional.of(employee));

// eq() — exact match (use when mixing matchers with exact values)
when(service.findByNameAndCity(eq("John"), anyString())).thenReturn(result);
```

> **Important:** If you use a matcher for one argument, you must use matchers for **all** arguments. Can't mix `eq("John")` with a raw `"Delhi"` — use `eq("Delhi")` instead.

### Full Example — Service Test

```java
@ExtendWith(MockitoExtension.class)
class EmployeeServiceTest {

    @Mock
    private EmployeeRepository employeeRepository;

    @Spy
    private EmployeeMapper employeeMapper = new EmployeeMapperImpl();

    @InjectMocks
    private EmployeeServiceImpl employeeService;

    private Employee employee;

    @BeforeEach
    void setUp() {
        employee = Employee.builder()
            .id(1L)
            .firstName("John")
            .lastName("Doe")
            .email("john@example.com")
            .build();
    }

    @Test
    @DisplayName("Should return employee when email is valid")
    void shouldReturnEmployeeByEmail() {
        // Given
        when(employeeRepository.findByEmail("john@example.com"))
            .thenReturn(Optional.of(employee));

        // When
        EmployeeDto result = employeeService
            .getEmployeeByEmail("john@example.com");

        // Then
        assertThat(result).isNotNull();
        assertThat(result.getEmail()).isEqualTo("john@example.com");
        verify(employeeRepository).findByEmail("john@example.com");
    }

    @Test
    @DisplayName("Should throw exception when employee not found")
    void shouldThrowWhenEmployeeNotFound() {
        // Given
        when(employeeRepository.findById(999L))
            .thenReturn(Optional.empty());

        // When / Then
        assertThatThrownBy(() -> employeeService.getEmployeeById(999L))
            .isInstanceOf(ResourceNotFoundException.class)
            .hasMessageContaining("not found");

        verify(employeeRepository).findById(999L);
    }

    @Test
    @DisplayName("Should save employee with valid input")
    void shouldCreateEmployee() {
        // Given
        when(employeeRepository.save(any(Employee.class)))
            .thenReturn(employee);

        EmployeeDto request = new EmployeeDto("John", "Doe",
            "john@example.com");

        // When
        EmployeeDto result = employeeService.createEmployee(request);

        // Then
        assertThat(result).isNotNull();
        assertThat(result.getEmail()).isEqualTo("john@example.com");
        verify(employeeRepository, times(1)).save(any(Employee.class));
    }
}
```

> **Test naming convention:** There's no enforced standard. Common styles:
> 
> |Style|Example|
> |---|---|
> |`should_when`|`shouldReturnEmployee_WhenEmailIsValid`|
> |`test_method_when_then`|`testFindByEmail_WhenValid_ThenReturnEmployee`|
> |`given_when_then`|`givenValidEmail_whenFind_thenReturnEmployee`|
> |`method_scenario_expected`|`findByEmail_validEmail_returnsEmployee`|
> 
> Pick one style and stay consistent within a project.
