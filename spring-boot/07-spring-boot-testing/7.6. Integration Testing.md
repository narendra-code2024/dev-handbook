### Why Integration Testing?

Unit tests verify individual components in isolation. But many bugs only appear when components **interact** — configuration errors, missing data, incorrect queries, broken wiring. Integration tests catch these.

```
Unit test:
  Service (real) → Repository (mocked) → ✅ passes
  
Production:
  Service → Repository → DB → ❌ fails (query was wrong)

Integration test:
  Service → Repository → Real DB → ❌ caught it
```

Integration tests also serve as **regression tests** — when you change code, they ensure existing functionality still works.

---

### What About Controller Unit Tests?

Controller unit testing (`@WebMvcTest`) is **generally not worth it** if your controllers are thin — they just receive requests, call services, and return responses. There's no business logic to test in isolation.

Controllers are better tested through **integration tests** where the full flow is verified — request → controller → service → repository → DB → response.

---

### Test Class Naming

To distinguish unit tests from integration tests, suffix integration test classes with `IT`:

```
src/test/java/com/app/
  ├── repository/EmployeeRepositoryTest.java        ← unit test
  ├── service/EmployeeServiceTest.java              ← unit test
  └── controller/EmployeeControllerIT.java          ← integration test
```

> The `IT` suffix is a common convention. Build tools like Maven Failsafe plugin use it to identify and run integration tests separately from unit tests.

### WebTestClient

`WebTestClient` provides a fluent API for making **real HTTP requests** to your application during tests. It's primarily associated with WebFlux but works with Spring MVC too.

#### Dependency

```xml
<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-webflux</artifactId>
    <scope>test</scope>
</dependency>
```

> Added with `test` scope — only used for testing, not included in production build.

---

### Setup

```java
@SpringBootTest(webEnvironment = SpringBootTest.WebEnvironment.RANDOM_PORT)
@AutoConfigureWebTestClient(timeout = "10000")
@Import(TestContainersConfiguration.class)
class EmployeeControllerIT {

    @Autowired
    private WebTestClient webTestClient;

    @Autowired
    private EmployeeRepository employeeRepository;

    @BeforeEach
    void setUp() {
        employeeRepository.deleteAll();
    }
}
```

|Configuration|Purpose|
|---|---|
|`@SpringBootTest`|Loads full application context (all layers)|
|`webEnvironment = RANDOM_PORT`|Starts the actual server on a random port|
|`@AutoConfigureWebTestClient`|Auto-configures WebTestClient to point to the running server|
|`timeout = "10000"`|10 second timeout for requests|
|`@Import(TestContainersConfiguration.class)`|Uses real PostgreSQL via Docker (from Page 7.4)|
|`@BeforeEach deleteAll()`|Clean state before each test|

### WebTestClient Methods

|Method|Purpose|
|---|---|
|`.get()`, `.post()`, `.put()`, `.delete()`|HTTP method|
|`.uri("/api/employees")`|Endpoint path|
|`.bodyValue(request)`|Request body|
|`.exchange()`|Execute the request|
|`.expectStatus()`|Assert HTTP status|
|`.expectBody()`|Assert response body|
|`.expectHeader()`|Assert response headers|
|`.jsonPath("$.field")`|Access specific field in JSON response|

### Test Examples

#### Create Employee — Success

```java
@Test
@DisplayName("Should create employee and return 201")
void testCreateEmployee_WhenValidInput_ThenReturn201() {
    // Given
    EmployeeDto request = EmployeeDto.builder()
        .firstName("Jane")
        .lastName("Doe")
        .email("jane@example.com")
        .build();

    // When / Then
    webTestClient.post()
        .uri("/api/employees")
        .bodyValue(request)
        .exchange()
        .expectStatus().isCreated()
        .expectBody()
        .jsonPath("$.data.id").isNotEmpty()
        .jsonPath("$.data.firstName").isEqualTo("Jane")
        .jsonPath("$.data.email").isEqualTo("jane@example.com");
}
```

#### Create Employee — Duplicate Email

```java
@Test
@DisplayName("Should return 400 when email already exists")
void testCreateEmployee_WhenDuplicateEmail_ThenReturn400() {
    // Given — save one employee first
    Employee existing = Employee.builder()
        .firstName("Jane")
        .lastName("Doe")
        .email("jane@example.com")
        .build();
    employeeRepository.save(existing);

    EmployeeDto request = EmployeeDto.builder()
        .firstName("John")
        .lastName("Smith")
        .email("jane@example.com") // same email
        .build();

    // When / Then
    webTestClient.post()
        .uri("/api/employees")
        .bodyValue(request)
        .exchange()
        .expectStatus().isBadRequest();
}
```

#### Find Employee by Email — Success

```java
@Test
@DisplayName("Should return employee when email is valid")
void testFindByEmail_WhenEmailExists_ThenReturnEmployee() {
    // Given
    Employee employee = Employee.builder()
        .firstName("Jane")
        .lastName("Doe")
        .email("jane@example.com")
        .build();
    employeeRepository.save(employee);

    // When / Then
    webTestClient.get()
        .uri("/api/employees/email/jane@example.com")
        .exchange()
        .expectStatus().isOk()
        .expectBody()
        .jsonPath("$.data.firstName").isEqualTo("Jane")
        .jsonPath("$.data.email").isEqualTo("jane@example.com");
}
```

#### Find Employee by Email — Not Found

```java
@Test
@DisplayName("Should return 404 when email not found")
void testFindByEmail_WhenEmailNotFound_ThenReturn404() {
    webTestClient.get()
        .uri("/api/employees/email/nonexistent@example.com")
        .exchange()
        .expectStatus().isNotFound();
}
```

### Common JsonPath Assertions

```java
.jsonPath("$.id").isNotEmpty()           // field exists and has value
.jsonPath("$.name").isEqualTo("Jane")    // exact match
.jsonPath("$.email").exists()            // field exists
.jsonPath("$.items").isArray()           // is an array
.jsonPath("$.items.length()").isEqualTo(3)  // array size
.jsonPath("$.items[0].name").isEqualTo("first") // array element
```

### What Each Test Type Covers — Summary

```
Repository Test (@DataJpaTest)
  → "Do my queries work against a real database?"

Service Test (Mockito)
  → "Does my business logic work correctly?"

Integration Test (@SpringBootTest + WebTestClient)
  → "Does the full API work end-to-end?"
  → Request → Security → Controller → Service → Repository → DB → Response
```
