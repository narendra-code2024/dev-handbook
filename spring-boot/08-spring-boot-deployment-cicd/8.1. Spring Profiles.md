Every application has multiple environments, each with different configurations — different databases, different API keys, different log levels, etc.

```
Dev   → local development, localhost DB, debug logging
Test  → automated tests, H2/TestContainers, test data
Stage → mirror of production, final QA and client demo before release
Prod  → live application, real users, real data
```

Spring Profiles let you define **environment-specific configurations** and switch between them without changing code.

> For the short version (config files + precedence + activation), see [1.7 Startup Flow](<../01-introduction-to-spring-boot/1.7. Startup Flow.md>). This file goes deeper on profile-specific files, `@Profile`, and runtime activation.

---

## Profile-Specific Properties Files

Create separate properties files per profile in `src/main/resources`:

```
src/main/resources/
  ├── application.properties          ← common/default config
  ├── application-dev.properties      ← local development
  ├── application-test.properties     ← testing
  ├── application-stage.properties    ← staging / pre-production
  └── application-prod.properties     ← production
```

Spring loads `application.properties` first as base config, then layers `application-{profile}.properties` on top — profile-specific values override the base.

`application.properties` (common):

```properties
spring.application.name=myapp
server.port=8080
```

`application-dev.properties`:

```properties
spring.datasource.url=jdbc:postgresql://localhost:5432/myapp_dev
spring.datasource.username=dev_user
spring.datasource.password=dev_pass
spring.jpa.hibernate.ddl-auto=update
logging.level.root=DEBUG
```

`application-prod.properties`:

```properties
spring.datasource.url=jdbc:postgresql://prod-rds.amazonaws.com:5432/myapp
spring.datasource.username=${DB_USERNAME}
spring.datasource.password=${DB_PASSWORD}
spring.jpa.hibernate.ddl-auto=validate
logging.level.root=WARN
```

> In production, use environment variables (`${DB_USERNAME}`) instead of hardcoding credentials.

## `@Profile` Annotation

Load specific beans only in certain environments:

```java
public interface DataService {
    String getData();
}

@Service
@Profile("dev")
public class DataServiceImplDev implements DataService {
    @Override
    public String getData() {
        return "Data from dev mock";
    }
}

@Service
@Profile("prod")
public class DataServiceImplProd implements DataService {
    @Override
    public String getData() {
        return "Data from production database";
    }
}
```

```
Profile = dev  → DataServiceImplDev bean is created
Profile = prod → DataServiceImplProd bean is created
```

> If no profile is set, the active profile is `"default"`. A bean with no `@Profile` annotation is loaded in **all** profiles.

Other `@Profile` expressions:

- `@Profile("!prod")` — loaded in all profiles EXCEPT prod
- `@Profile({"dev", "test"})` — loaded in dev OR test

## Setting the Active Profile

**In properties (default for local development):**

```properties
# application.properties
spring.profiles.active=dev
```

**Via Maven:**

```bash
./mvnw spring-boot:run -Dspring-boot.run.profiles=prod
```

**Via environment variable:**

```bash
SPRING_PROFILES_ACTIVE=prod ./mvnw spring-boot:run
```

**Via JAR (production deployment):**

```bash
java -Dspring.profiles.active=prod -jar target/myapp.jar
```

> The JAR form is the most common for production — you build one JAR and activate the profile at runtime.

## Important Interview Questions

1. **Why use profiles instead of separate config files per environment you swap at build time?** Profiles let you build one artifact and activate the right config at runtime. Same JAR goes to dev, stage, and prod — less build variance, fewer "works on my machine" surprises.

2. **How does Spring choose between `application.properties` and `application-{profile}.properties`?** Base `application.properties` is always loaded first; the profile-specific file is loaded on top and overrides any overlapping keys. Profile activation for this resolution is covered in [1.7 Startup Flow](<../01-introduction-to-spring-boot/1.7. Startup Flow.md>).

3. **What does `@Profile("!prod")` mean?** The bean is registered in every profile EXCEPT `prod`. Useful for dev-only stubs, in-memory seed data, or mocked external integrations that should never reach production.

4. **What happens if no profile is active?** Spring uses the literal profile name `default`. Beans with no `@Profile` annotation load in all profiles; beans annotated `@Profile("default")` load only when no other profile is active.

5. **Can multiple profiles be active at once?** Yes — `spring.profiles.active=prod,aws` activates both. Properties from `application-prod.properties` and `application-aws.properties` both load; later-listed profiles win on overlapping keys.

## Quick Reference

|Concept|Remember|
|---|---|
| Base config | `application.properties` — always loaded |
| Profile file | `application-{profile}.properties` — overrides base |
| Activate (properties) | `spring.profiles.active=dev` |
| Activate (JAR) | `java -Dspring.profiles.active=prod -jar app.jar` |
| Activate (env var) | `SPRING_PROFILES_ACTIVE=prod` |
| `@Profile("dev")` | Bean only in `dev` |
| `@Profile("!prod")` | Bean in every profile except `prod` |
| No active profile | Profile name is `default` |
