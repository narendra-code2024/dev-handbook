
---

A database migration tool that manages schema changes through **versioned SQL scripts**. You write the SQL, Flyway executes it in order on startup, and tracks what's been applied тАФ the same migrations run in dev, stage, and prod with no manual `psql` sessions or "did you remember to apply that ALTER on staging?" surprises.

---

## Why Not `ddl-auto=update`?

Hibernate's `ddl-auto=update` is convenient for development, but dangerous in production:

**Renaming a column:**

```java
// Before
private String name;

// After (renamed)
private String fullName;
```

```
What you expect: ALTER TABLE users RENAME COLUMN name TO full_name
What Hibernate does: ALTER TABLE users ADD COLUMN full_name (keeps old name column)
тЖТ Old data stuck in name column, new column is empty
```

**Removing a field:**

```java
// Removed phoneNumber from entity
```

```
Hibernate may drop the column тЖТ production data lost
```

**Other risks:**

- No version history тАФ you can't track what changed when
- No rollback тАФ can't undo a bad migration
- No review тАФ schema changes happen automatically, no one approves them
- Different behavior across databases тАФ Hibernate generates different SQL for PostgreSQL vs MySQL

> `ddl-auto=update` is fine for dev. In production, always use `ddl-auto=validate` (checks schema matches entities) or `ddl-auto=none` (does nothing) тАФ and manage schema changes with **Flyway**.

## Setup

**Dependency:**

```xml
<dependency>
    <groupId>org.flywaydb</groupId>
    <artifactId>flyway-core</artifactId>
</dependency>

<!-- For PostgreSQL -->
<dependency>
    <groupId>org.flywaydb</groupId>
    <artifactId>flyway-database-postgresql</artifactId>
</dependency>
```

**Properties:**

```properties
# Production
spring.jpa.hibernate.ddl-auto=validate
spring.flyway.enabled=true
spring.flyway.baseline-on-migrate=true
spring.flyway.locations=classpath:db/migration
```

|Property|Purpose|Default|
|---|---|---|
|`ddl-auto=validate`|Hibernate only checks schema matches entities, never modifies|тАФ|
|`flyway.enabled=true`|Enable Flyway|`true` (when dependency is present)|
|`flyway.baseline-on-migrate=true`|Allows Flyway to work with an existing database that wasn't managed by Flyway before|`false`|
|`flyway.locations`|Where Flyway looks for migration files|`classpath:db/migration`|

> **Dev vs prod.** In development, many teams stick with `ddl-auto=update` (Flyway optional) for fast iteration. In production, always `ddl-auto=validate` + `flyway.enabled=true` so Hibernate only verifies the schema and Flyway owns all changes. Some teams run Flyway in dev too for consistency тАФ either works, just make sure prod always uses Flyway.

## Migration Files

Place SQL files in `src/main/resources/db/migration`:

```
src/main/resources/
  тФФтФАтФА db/
      тФФтФАтФА migration/
          тФЬтФАтФА V1__create_users_table.sql
          тФЬтФАтФА V2__create_user_role_table.sql
          тФЬтФАтФА V3__create_sessions_table.sql
          тФЬтФАтФА V4__create_posts_table.sql
          тФФтФАтФА V5__add_phone_to_users.sql
```

**Naming convention:**

```
V1__create_users_table.sql
тФВ тФВ  тФВ
тФВ тФВ  тФФтФАтФА description (underscores for spaces)
тФВ тФФтФАтФА double underscore separator (required)
тФФтФАтФА version number
```

> Naming is strict: must start with `V`, followed by version number, then `__` (double underscore), then description. Flyway rejects files that don't follow this pattern.

## Migration Examples

**V1 тАФ initial schema:**

```sql
-- V1__initial_schema.sql
CREATE TABLE users (
    id BIGSERIAL PRIMARY KEY,
    first_name VARCHAR(255) NOT NULL,
    last_name VARCHAR(255) NOT NULL,
    email VARCHAR(255) NOT NULL UNIQUE,
    password VARCHAR(255)
);

CREATE TABLE user_role (
    user_id BIGINT NOT NULL,
    role VARCHAR(50) NOT NULL,
    FOREIGN KEY (user_id) REFERENCES users(id),
    PRIMARY KEY (user_id, role)
);

CREATE TABLE sessions (
    id BIGSERIAL PRIMARY KEY,
    refresh_token VARCHAR(500) NOT NULL UNIQUE,
    expires_at TIMESTAMP NOT NULL,
    provider VARCHAR(50) NOT NULL,
    user_id BIGINT NOT NULL,
    FOREIGN KEY (user_id) REFERENCES users(id)
);
```

> Initial setup can group related tables in one file. After that, one change per file.

**V2 тАФ add a column later:**

```sql
-- V2__add_phone_to_users.sql
ALTER TABLE users ADD COLUMN phone VARCHAR(20);
```

> Migrations should focus on **schema changes** only (CREATE, ALTER, DROP). For seeding data, see the section below.

## Seeding Data

Seeding is separate from migrations тАФ different environments need different data. Use `CommandLineRunner` with `@Profile` (see [8.1 Spring Profiles](<8.1. Spring Profiles.md>) тЖТ `@Profile` Annotation):

```java
@Component
@Profile("dev")
@RequiredArgsConstructor
public class DevDataSeeder implements CommandLineRunner {

    private final UserRepository userRepository;
    private final PasswordEncoder passwordEncoder;

    @Override
    public void run(String... args) {
        if (userRepository.count() == 0) {
            User admin = User.builder()
                .firstName("Dev")
                .lastName("Admin")
                .email("admin@dev.com")
                .password(passwordEncoder.encode("password"))
                .roles(Set.of(Role.ADMIN))
                .build();
            userRepository.save(admin);
        }
    }
}

@Component
@Profile("prod")
@RequiredArgsConstructor
public class ProdDataSeeder implements CommandLineRunner {

    // seed production-specific data тАФ admin account, default config
}
```

> `if (userRepository.count() == 0)` prevents duplicate seeding on every restart. Each environment gets its own seeder with its own data.

```
App starts
  тФВ
  тЦ╝
Flyway checks flyway_schema_history table in DB
  тФВ
  тЦ╝

  тФЬтФАтФА V1 тЬЕ applied
  тФЬтФАтФА V2 тЬЕ applied
  тФЬтФАтФА V3 тЬЕ applied
  тФЬтФАтФА V4 тЭМ new
  тФФтФАтФА V5 тЭМ new
  тФВ
  тЦ╝
Runs V4 and V5 in order
  тФВ
  тЦ╝

  тФВ
  тЦ╝

```

> Flyway creates a `flyway_schema_history` table in your database to track which migrations have been applied. It only runs new ones.

## Migration Rules

**One migration per logical change.** Each file should do one thing:

```
тЬЕ Good:
  V1__create_users_table.sql
  V2__create_user_role_table.sql
  V3__add_phone_to_users.sql

тЭМ Bad:
  V1__create_all_tables.sql          (too much in one file)
  V2__march_changes.sql              (not specific)
```

> For initial setup, grouping related tables in one file is fine (e.g., `V1__initial_schema.sql`). After that, one change per file.

**Never edit an already-applied migration.** Flyway checksums each file тАФ if you change a migration that's already been applied, Flyway fails on startup with a checksum mismatch.

```
Need to change something already applied?
  тЖТ Write a NEW migration (V6__fix_whatever.sql)
  тЖТ Never modify V1-V5
```

**Migrations are immutable** тАФ once applied, they're history. New changes always go in new files.

**Roll back forward.** Flyway Community (free) **does not support automatic rollback**. If a migration goes wrong, write a new forward migration to undo it:

```
V4__add_phone_to_users.sql
  тЖТ ALTER TABLE users ADD COLUMN phone VARCHAR(20);

V5__remove_phone_from_users.sql
  тЖТ ALTER TABLE users DROP COLUMN phone;
```

> Flyway Teams (paid) supports `flyway undo` with `U` prefixed files (e.g., `U4__undo_add_phone.sql`). Most teams on the free tier just write a new `V` migration to fix things.

## Important Interview Questions

1. **Why not just use `ddl-auto=update` in production?** It silently produces incorrect DDL (renames become add-column, deletes can drop data), has no version history, no rollback, and no review step. Flyway gives you reviewed, versioned, checksummed SQL that runs identically across environments.

2. **What is `flyway_schema_history`?** A table Flyway creates in the target database to record every applied migration with its version, checksum, and install timestamp. On startup Flyway consults it to decide which `V*` files are new and need to run.

3. **You edited an already-applied migration. What happens?** Flyway recomputes the checksum, finds it doesn't match the stored one, and fails the app on startup with a migration validation error. The fix is to revert your edit and add a new `V` migration for the change.

4. **How do you roll back a bad migration on Flyway Community?** Write a forward migration that reverses it (e.g., `V5__remove_phone_from_users.sql` to undo the column add in V4). Automatic `flyway undo` is a Teams/paid feature.

5. **What does `baseline-on-migrate=true` do?** It lets Flyway adopt an existing, non-empty database by marking the current schema as the baseline version, so it doesn't try to re-run migrations that created tables already present. Useful when introducing Flyway to a legacy project.

6. **Why separate seeding from migrations?** Schema is the same everywhere; seed data differs per environment (dev fixtures vs. prod admin account). Keeping seeding in `@Profile`-gated `CommandLineRunner`s lets each environment seed its own data without polluting shared migration files.
