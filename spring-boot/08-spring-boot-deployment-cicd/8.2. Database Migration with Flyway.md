### The Problem — Why Not `ddl-auto=update` in Production?

Hibernate's `ddl-auto=update` is convenient for development, but dangerous in production:

**Renaming a column:**

```java
// Before
private String name;

// After (renamed)
private String fullName;
```

```
What you expect: ALTER TABLE users RENAME COLUMN name TO full_name
What Hibernate does: ALTER TABLE users ADD COLUMN full_name (keeps old name column)
→ Old data stuck in name column, new column is empty
```

**Removing a field:**

```java
// Removed phoneNumber from entity
```

```
Hibernate may drop the column → production data lost
```

**Other risks:**

- No version history — you can't track what changed when
- No rollback — can't undo a bad migration
- No review — schema changes happen automatically, no one approves them
- Different behavior across databases — Hibernate generates different SQL for PostgreSQL vs MySQL

> `ddl-auto=update` is fine for dev. In production, always use `ddl-auto=validate` (checks schema matches entities) or `ddl-auto=none` (does nothing) — and manage schema changes with **Flyway**.

---

### What is Flyway?

A database migration tool that manages schema changes through **versioned SQL scripts**. You write the SQL, Flyway executes it in order, and tracks what's been applied.

```
Without Flyway:
  Developer runs SQL manually → forgets on staging → production breaks

With Flyway:
  Migration scripts in code → Flyway runs them automatically on startup
  → Same migrations run in dev, stage, and prod
```

### Setup

#### Dependency

```xml
<dependency>
    <groupId>org.flywaydb</groupId>
    <artifactId>flyway-core</artifactId>
</dependency>

<!-- For PostgreSQL -->
<dependency>
    <groupId>org.flywaydb</groupId>
    <artifactId>flyway-database-postgresql</artifactId>
</dependency>
```

#### Properties

```properties
# Production
spring.jpa.hibernate.ddl-auto=validate
spring.flyway.enabled=true
spring.flyway.baseline-on-migrate=true
spring.flyway.locations=classpath:db/migration
```

|Property|Purpose|Default|
|---|---|---|
|`ddl-auto=validate`|Hibernate only checks schema matches entities, never modifies|—|
|`flyway.enabled=true`|Enable Flyway|`true` (when dependency is present)|
|`flyway.baseline-on-migrate=true`|Allows Flyway to work with an existing database that wasn't managed by Flyway before|`false`|
|`flyway.locations`|Where Flyway looks for migration files|`classpath:db/migration`|

### Migration Files

Place SQL files in `src/main/resources/db/migration`:

```
src/main/resources/
  └── db/
      └── migration/
          ├── V1__create_users_table.sql
          ├── V2__create_user_role_table.sql
          ├── V3__create_sessions_table.sql
          ├── V4__create_posts_table.sql
          └── V5__add_phone_to_users.sql
```

#### Naming Convention

```
V1__create_users_table.sql
│ │  │
│ │  └── description (underscores for spaces)
│ └── double underscore separator (required)
└── version number
```

> **Naming is strict:** must start with `V`, followed by version number, then `__` (double underscore), then description. Flyway rejects files that don't follow this pattern.

### Migration Examples

#### V1 — Initial Schema

```sql
-- V1__initial_schema.sql
CREATE TABLE users (
    id BIGSERIAL PRIMARY KEY,
    first_name VARCHAR(255) NOT NULL,
    last_name VARCHAR(255) NOT NULL,
    email VARCHAR(255) NOT NULL UNIQUE,
    password VARCHAR(255)
);

CREATE TABLE user_role (
    user_id BIGINT NOT NULL,
    role VARCHAR(50) NOT NULL,
    FOREIGN KEY (user_id) REFERENCES users(id),
    PRIMARY KEY (user_id, role)
);

CREATE TABLE sessions (
    id BIGSERIAL PRIMARY KEY,
    refresh_token VARCHAR(500) NOT NULL UNIQUE,
    expires_at TIMESTAMP NOT NULL,
    provider VARCHAR(50) NOT NULL,
    user_id BIGINT NOT NULL,
    FOREIGN KEY (user_id) REFERENCES users(id)
);
```

> Initial setup can group related tables in one file. After that, one change per file.

#### V2 — Add a Column Later

```sql
-- V2__add_phone_to_users.sql
ALTER TABLE users ADD COLUMN phone VARCHAR(20);
```

> Migrations should focus on **schema changes** only (CREATE, ALTER, DROP). For seeding data, see the section below.

### Seeding Data

Seeding is separate from migrations — different environments need different data. Use `CommandLineRunner` with `@Profile`:

```java
@Component
@Profile("dev")
@RequiredArgsConstructor
public class DevDataSeeder implements CommandLineRunner {

    private final UserRepository userRepository;
    private final PasswordEncoder passwordEncoder;

    @Override
    public void run(String... args) {
        if (userRepository.count() == 0) {
            User admin = User.builder()
                .firstName("Dev")
                .lastName("Admin")
                .email("admin@dev.com")
                .password(passwordEncoder.encode("password"))
                .roles(Set.of(Role.ADMIN))
                .build();
            userRepository.save(admin);
        }
    }
}

@Component
@Profile("prod")
@RequiredArgsConstructor
public class ProdDataSeeder implements CommandLineRunner {

    // seed production-specific data — admin account, default config
}
```

> `if (userRepository.count() == 0)` prevents duplicate seeding on every restart. Each environment gets its own seeder with its own data.

### How Flyway Works

```
App starts
  │
  ▼
Flyway checks flyway_schema_history table in DB
  │
  ▼
Which migrations have already been applied?
  ├── V1 ✅ applied
  ├── V2 ✅ applied
  ├── V3 ✅ applied
  ├── V4 ❌ new
  └── V5 ❌ new
  │
  ▼
Runs V4 and V5 in order
  │
  ▼
Records them in flyway_schema_history
  │
  ▼
App continues startup
```

> Flyway creates a `flyway_schema_history` table in your database to track which migrations have been applied. It only runs new ones.

### Important Rules

**One migration per logical change.** Each file should do one thing:

```
✅ Good:
  V1__create_users_table.sql
  V2__create_user_role_table.sql
  V3__add_phone_to_users.sql

❌ Bad:
  V1__create_all_tables.sql          (too much in one file)
  V2__march_changes.sql              (not specific)
```

> For initial setup, grouping related tables in one file is fine (e.g., `V1__initial_schema.sql`). After that, one change per file.

**Never edit an already-applied migration.** Flyway checksums each file — if you change a migration that's already been applied, Flyway fails on startup with a checksum mismatch.

```
Need to change something already applied?
  → Write a NEW migration (V6__fix_whatever.sql)
  → Never modify V1-V5
```

**Migrations are immutable** — once applied, they're history. New changes always go in new files.

---

### Rollback

Flyway Community (free) **does not support automatic rollback**. If a migration goes wrong, write a new migration to undo it:

```
V4__add_phone_to_users.sql
  → ALTER TABLE users ADD COLUMN phone VARCHAR(20);

V5__remove_phone_from_users.sql
  → ALTER TABLE users DROP COLUMN phone;
```

> Flyway Teams (paid) supports `flyway undo` with `U` prefixed files (e.g., `U4__undo_add_phone.sql`). Most teams on the free tier just write a new `V` migration to fix things.

### Dev vs Prod Workflow

```
Development:
  spring.jpa.hibernate.ddl-auto=update    ← Hibernate manages schema
  spring.flyway.enabled=false             ← Flyway off (optional)

Production:
  spring.jpa.hibernate.ddl-auto=validate  ← Hibernate only checks
  spring.flyway.enabled=true              ← Flyway manages schema
```

> Some teams use Flyway in dev too for consistency. Others use `ddl-auto=update` locally for speed and Flyway only in production. Either approach works — just make sure production always uses Flyway.
