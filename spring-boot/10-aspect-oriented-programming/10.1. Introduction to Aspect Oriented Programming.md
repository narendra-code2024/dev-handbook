### The Problem — Cross-Cutting Concerns

Imagine you need logging in every service method — who called the API, when, how long it took. With 50 methods, you'd write logging code in all 50:

```java
public EmployeeDto getEmployee(Long id) {
    log.info("getEmployee called with id: {}", id);    // logging
    checkPermission();                                   // security
    // ... actual business logic ...
    log.info("getEmployee completed in {}ms", duration); // logging
}

public EmployeeDto createEmployee(EmployeeDto dto) {
    log.info("createEmployee called");                   // same logging
    checkPermission();                                   // same security
    // ... actual business logic ...
    log.info("createEmployee completed in {}ms", duration); // same logging
}

// ... repeat for 48 more methods
```

The logging and security code is **duplicated everywhere** — it's not part of the business logic but it **cuts across** all layers:

```
Presentation Layer  ──┐
                      ├── Logging, Security, Transactions
Business Layer      ──┤   (same concerns in every layer)
                      │
Data Access Layer   ──┘
                        Cross-Cutting Concerns
```

These are called **cross-cutting concerns** — they span multiple modules and can't be cleanly separated with regular OOP.

---

### What Are Cross-Cutting Concerns?

Code that **doesn't belong to one specific feature** but is needed **across many parts** of your application. You can't put it in one place — it "cuts across" everything.

```
Your app has:
  EmployeeService   → needs logging, security, transactions
  PostService       → needs logging, security, transactions
  PaymentService    → needs logging, security, transactions
                         │
                         ▼
              Same concerns cutting across all services
              = Cross-Cutting Concerns
```

---

### What is AOP?

**Aspect-Oriented Programming** — a paradigm that complements OOP by letting you **separate cross-cutting concerns** from business logic. Instead of writing logging in 50 methods, you write it once in an **Aspect** and Spring applies it automatically.

```
Without AOP:
  50 methods × logging code = logging scattered everywhere

With AOP:
  1 LoggingAspect → Spring applies it to all 50 methods
  Business logic stays clean
```

---

### Benefits

- **Cleaner code** — business methods contain only business logic
- **No duplication** — write the concern once, apply everywhere
- **Enhanced modularity** — each concern is in its own class (LoggingAspect, SecurityAspect, CachingAspect)
- **Loose coupling** — business code doesn't know about logging/security, and vice versa

---

### Key Concepts

**Aspect** — a class where you write the cross-cutting logic. One concern = one aspect.

> "I want to log every service method" → create a `LoggingAspect` class.

**Join Point** — a point in your code where an aspect _can_ run. In Spring, this is always a **method execution**.

> Every method in your app is a potential join point.

**Pointcut** — a filter that selects _which_ join points to target. You write an expression to pick specific methods.

> "Only methods in the service package" or "only methods starting with get".

**Advice** — the actual code that runs at the selected join points. The _what_ and _when_.

> "Before the method runs, log its name." — that's the advice.

**Weaving** — the process of connecting aspects to your code. Spring does this automatically at **runtime** using proxies.

> You write the aspect, Spring figures out where and when to apply it.

#### How They Fit Together

```
Aspect    → "I'm the LoggingAspect class"
Pointcut  → "Target all methods in com.app.service.*"
Advice    → "Before each method, log the method name"
       │
       ▼
Weaving (Spring does this automatically at runtime)
       │
       ▼
Method called → Advice runs first → Method executes
```

---

### Quick Example

```java
@Aspect
@Component
public class LoggingAspect {

    private static final Logger log = LoggerFactory.getLogger(LoggingAspect.class);

    @Before("execution(* com.app.service.*.*(..))")
    public void logBefore(JoinPoint joinPoint) {
        log.info("Calling: {}", joinPoint.getSignature().getName());
    }
}
```

```
Service method called: getEmployeeById(1L)
       │
       ▼
LoggingAspect.logBefore() runs → logs "Calling: getEmployeeById"
       │
       ▼
Actual getEmployeeById() executes
```

> No logging code inside the service — the aspect handles it. Pointcuts and advice types are covered in detail in the next pages.

### Dependency

AOP is included with `spring-boot-starter-data-jpa` — no additional dependency needed. If you're not using JPA, add:

```xml
<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-aop</artifactId>
</dependency>
```

### Common Cross-Cutting Concerns

|Concern|What It Does|
|---|---|
|**Logging**|Log method calls, arguments, execution time|
|**Security**|Check permissions before method execution|
|**Transaction management**|Begin/commit/rollback transactions (`@Transactional` uses AOP)|
|**Caching**|Return cached results instead of executing method (`@Cacheable` uses AOP)|
|**Error handling**|Catch and handle exceptions uniformly|
|**Performance monitoring**|Measure and log execution time|

> You've already been using AOP without knowing it — `@Transactional`, `@Cacheable`, `@PreAuthorize` all work through AOP internally.
