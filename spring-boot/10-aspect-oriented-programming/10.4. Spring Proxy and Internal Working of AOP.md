### What is a Proxy?

A **wrapper object** that Spring creates around your bean. It looks identical to your bean but intercepts method calls to run AOP advice (logging, transactions, security, etc.) before/after the actual method.

```
Without proxy:
  Controller → EmployeeService (direct call)

With proxy:
  Controller → Proxy (looks like EmployeeService)
                  │
                  ├── runs @Before advice
                  ├── runs @Around advice (before proceed)
                  │
                  ▼
               Actual EmployeeService method executes
                  │
                  ├── runs @AfterReturning / @AfterThrowing
                  ├── runs @Around advice (after proceed)
                  │
                  ▼
               Result returned to Controller
```

> When you `@Autowired` a service, you're not getting the actual object — you're getting the proxy. The proxy handles the extra behavior, then delegates to the real object.

---

### When Does Spring Create a Proxy?

Spring only creates proxies when needed — not every bean gets one:

|Condition|Proxy Created?|
|---|---|
|Bean has `@Transactional` methods|✅ Yes|
|Bean has `@Cacheable` methods|✅ Yes|
|Bean has `@Async` methods|✅ Yes|
|Bean matches an AOP pointcut|✅ Yes|
|Bean has `@PreAuthorize` methods|✅ Yes|
|Plain bean with no cross-cutting concerns|❌ No (no overhead)|

---

### How Proxies Are Created

Spring picks the proxy type automatically based on your bean:

```
Does the bean implement an interface?
  │
  ├── Yes → JDK Dynamic Proxy
  │         (creates a new class implementing the same interface)
  │
  └── No  → CGLIB Proxy
            (creates a subclass of your bean at runtime)
```

||JDK Dynamic Proxy|CGLIB Proxy|
|---|---|---|
|When used|Bean implements an interface|Bean has no interface|
|How it works|Implements the same interface|Creates a subclass|
|Can proxy|Interface methods only|All non-final methods|

> In practice, you don't choose — Spring decides automatically. Both work the same way from your perspective.

### How It Works Step by Step

```
1. App starts → Spring scans for aspects and pointcuts
         │
         ▼
2. For each bean, Spring checks:
   "Do any methods match a pointcut?"
         │
         ├── No  → create normal bean (no proxy)
         │
         └── Yes → create proxy that wraps the bean
                        │
                        ▼
3. Proxy stored in ApplicationContext
   (replaces the actual bean)
         │
         ▼
4. @Autowired injects the proxy, not the real bean
         │
         ▼
5. Method called on proxy:
         │
         ├── Matches a pointcut?
         │     ├── Yes → run advice, then call actual method
         │     └── No  → call actual method directly (no overhead)
         │
         ▼
6. Result returned to caller
```

### The Self-Invocation Gotcha

**Calling a method from within the same class bypasses the proxy:**

```java
@Service
public class EmployeeService {

    @Transactional
    public void methodA() {
        this.methodB(); // ❌ calls real object, NOT proxy
    }

    @Transactional
    public void methodB() {
        // @Transactional won't apply when called from methodA
    }
}
```

```
External call (through proxy):
  Controller → Proxy → methodA()  ✅ advice applies

Self call (bypasses proxy):
  methodA() → this.methodB()      ❌ advice skipped
  (this = real object, not the proxy)
```

**Why?** When `methodA` calls `methodB`, it uses `this` — which is the **real object** inside the proxy. The call never goes through the proxy, so no AOP advice is applied.

**How to fix:** Extract `methodB` into a **separate bean**, so the call goes through its own proxy:

```java
@Service
public class EmployeeService {

    private final EmployeeHelper helper;

    @Transactional
    public void methodA() {
        helper.methodB(); // ✅ goes through helper's proxy
    }
}

@Service
public class EmployeeHelper {

    @Transactional
    public void methodB() {
        // @Transactional applies correctly
    }
}
```

> This is the most common AOP pitfall. If `@Transactional`, `@Cacheable`, or any advice seems to not work, check if you're calling the method from within the same class.

### Weaving — When Aspects Are Applied

Weaving is the process of connecting aspects to your code. It can happen at different times:

|Type|When|Used By|
|---|---|---|
|**Runtime weaving**|When method is called|Spring AOP (default) — uses proxies|
|**Compile-time weaving**|During compilation|AspectJ (requires special compiler)|
|**Load-time weaving**|When class is loaded into JVM|AspectJ with Spring integration|

> Spring AOP uses **runtime weaving** through proxies — no special compiler or setup needed. Compile-time and load-time weaving are only needed for advanced scenarios with full AspectJ.

### Summary

```
Spring AOP = proxy-based
  │
  ├── Proxy wraps your bean
  ├── Intercepts external method calls
  ├── Runs advice before/after the method
  ├── Created only when needed (AOP, @Transactional, etc.)
  │
  └── Limitation: self-invocation bypasses the proxy
      → fix by extracting to a separate bean
```
