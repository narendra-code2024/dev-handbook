### Do You Really Need AOP?

Spring already provides built-in annotations for most cross-cutting concerns. Before writing a custom aspect, check if a built-in solution exists:

|Concern|Built-in Solution|Custom AOP Needed?|
|---|---|---|
|Security|`@Secured`, `@PreAuthorize`|❌ Usually not|
|Transactions|`@Transactional`|❌ No|
|Caching|`@Cacheable`, `@CacheEvict`|❌ Usually not|
|Validation|`@Valid`, `@Validated`|❌ No|
|Auditing|Spring Data JPA `@CreatedBy`, `@LastModifiedDate`|❌ Usually not|
|Logging / Monitoring|No built-in|✅ Yes — best use case for AOP|
|Performance profiling|No built-in|✅ Yes|
|Custom exception handling|`@RestControllerAdvice` handles most|⚠️ Sometimes|

> Most companies don't use custom AOP for security, caching, or transactions — the built-in annotations are well-optimized and easier to maintain. **AOP is most commonly used for logging, monitoring, and profiling** that applies across multiple layers.

---

### Where AOP Still Makes Sense

```
Built-in annotations cover your concern?
  ├── Yes → use them, skip AOP
  └── No → custom AOP is the right tool
           ├── Cross-layer logging
           ├── Performance monitoring
           ├── Custom auditing beyond JPA
           └── Retry logic, rate limiting
```

### Use Case Examples

These are **for understanding AOP** — in production, prefer the built-in alternatives mentioned above unless they don't meet your needs.

#### 1. Security Aspect

```java
@Aspect
@Component
public class SecurityAspect {

    @Before("execution(* com.app.service.*.*(..)) " +
            "&& @annotation(RequiresAdmin)")
    public void checkAdminAccess() {
        Authentication auth = SecurityContextHolder.getContext()
            .getAuthentication();
        if (!auth.getAuthorities().contains(
                new SimpleGrantedAuthority("ROLE_ADMIN"))) {
            throw new SecurityException(
                "Only admins can access this resource.");
        }
    }
}
```

> **In practice:** use `@PreAuthorize("hasRole('ADMIN')")` instead — simpler, well-tested, and integrated with Spring Security.

#### 2. Caching Aspect

```java
@Aspect
@Component
public class CachingAspect {

    private Map<String, Object> cache = new HashMap<>();

    @Around("execution(* com.app.service.*.*(..))")
    public Object cacheMethodResults(ProceedingJoinPoint joinPoint)
            throws Throwable {
        String key = joinPoint.getSignature().getName();
        if (cache.containsKey(key)) {
            return cache.get(key);
        }
        Object result = joinPoint.proceed();
        cache.put(key, result);
        return result;
    }
}
```

> **In practice:** use `@Cacheable("cacheName")` — handles cache keys, eviction, TTL, and integrates with Redis, Caffeine, etc.

#### 3. Auditing Aspect

```java
@Aspect
@Component
public class AuditAspect {

    private static final Logger log =
        LoggerFactory.getLogger(AuditAspect.class);

    @AfterReturning("execution(* com.app.service.*.get*(..))")
    public void auditAccess(JoinPoint joinPoint) {
        String user = SecurityContextHolder.getContext()
            .getAuthentication().getName();
        log.info("Data accessed by {} at {} — method: {}",
            user, LocalDateTime.now(),
            joinPoint.getSignature().getName());
    }
}
```

> **In practice:** Spring Data JPA's `@CreatedBy`, `@LastModifiedDate`, `@CreatedDate` handle most auditing. Use AOP only for custom audit logging beyond what JPA provides.

#### 4. Exception Handling Aspect

```java
@Aspect
@Component
@Slf4j
public class ExceptionHandlingAspect {

    @AfterThrowing(
        pointcut = "execution(* com.app.service.*.*(..))",
        throwing = "ex")
    public void handleException(JoinPoint joinPoint, Exception ex) {
        log.error("Exception in {}.{}: {}",
            joinPoint.getTarget().getClass().getSimpleName(),
            joinPoint.getSignature().getName(),
            ex.getMessage());
    }
}
```

> **In practice:** `@RestControllerAdvice` handles most exception scenarios. AOP-based exception handling is useful for logging exceptions in the service layer that don't propagate to controllers.

### What Most Companies Actually Use AOP For

```
Common:
  ✅ Logging — method entry/exit, arguments, execution time
  ✅ Performance monitoring — track slow methods
  ✅ Custom metrics — count method calls, track patterns

Rare (built-in is better):
  ⚠️ Security — @PreAuthorize is simpler
  ⚠️ Caching — @Cacheable is simpler
  ⚠️ Transactions — @Transactional is simpler
  ⚠️ Auditing — Spring Data JPA handles most cases
```

> AOP is a powerful tool but most of its original use cases now have dedicated Spring annotations. Learn it to understand how Spring works internally (`@Transactional`, `@Cacheable`, `@PreAuthorize` all use AOP under the hood), and use it for logging/monitoring where no built-in alternative exists.
