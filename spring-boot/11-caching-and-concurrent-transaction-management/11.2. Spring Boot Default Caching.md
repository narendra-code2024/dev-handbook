### Spring Boot Cache Abstraction

Spring Boot provides a **Cache Abstraction API** — a layer on top of different cache providers. You write the same annotations regardless of whether the underlying cache is in-memory, Redis, or something else.

By default (no extra dependencies), Spring uses a simple **ConcurrentHashMap** as the cache — just a `Map<key, value>` stored in memory.

```
Cache Abstraction API
  │
  ├── Default: ConcurrentHashMap (in-memory, single server)
  ├── Redis (distributed, shared across servers)
  ├── Caffeine (high-performance in-memory)
  └── Others (EhCache, Hazelcast, etc.)
```

> You can switch the underlying provider later without changing your annotations — just swap the dependency and config.

---

### How It Works Internally

```
@Cacheable method called
  │
  ▼
Spring AOP intercepts the call (proxy)
  │
  ▼
CacheManager retrieves the Cache by name (e.g., "employees")
  │
  ▼
Cache checks: does this key exist?
  │
  ├── Hit → return cached value (method never executes)
  │
  └── Miss → execute the actual method
                │
                ▼
           Store result in cache (key → value)
                │
                ▼
           Return result to caller
```

> Under the hood, the default cache is a `ConcurrentHashMap` — `CacheManager` manages multiple named caches, each cache is a `ConcurrentMap<Object, Object>`.

---

### Enabling Caching

Add `@EnableCaching` to any configuration class:

```java
@Configuration
@EnableCaching
public class CacheConfig {
}
```

> This sets up a `CacheManager` and enables all caching annotations. Without this, `@Cacheable` and others do nothing.

### Caching Annotations

#### @Cacheable — Cache the Result

If the result is already in cache, **return it without executing the method**. If not, execute the method and **store the result**.

```java
@Cacheable(cacheNames = "employees", key = "#id")
public EmployeeDto getEmployeeById(Long id) {
    // This only runs on cache miss
    return employeeRepository.findById(id)
        .map(employeeMapper::toDto)
        .orElseThrow();
}
```

```
First call:  getEmployeeById(1L) → cache miss → DB query → store in cache
Second call: getEmployeeById(1L) → cache hit → return from cache (no DB)
Third call:  getEmployeeById(2L) → cache miss → DB query (different key)
```

**Key expressions:**

```java
@Cacheable(cacheNames = "employees", key = "#id")
// cache key = the id argument

@Cacheable(cacheNames = "employees", key = "#email")
// cache key = the email argument

@Cacheable(cacheNames = "employees")
// no key specified = uses all method arguments as the key
```

---

#### @CachePut — Update the Cache

**Always executes the method** and updates the cache with the new result. Use for update operations where you want the cache to reflect the latest data.

```java
@CachePut(cacheNames = "employees", key = "#id")
public EmployeeDto updateEmployee(Long id, EmployeeDto dto) {
    // Always runs — updates DB and cache
    Employee employee = employeeRepository.findById(id).orElseThrow();
    employee.setFirstName(dto.getFirstName());
    return employeeMapper.toDto(employeeRepository.save(employee));
}
```

**@Cacheable vs @CachePut:**

||@Cacheable|@CachePut|
|---|---|---|
|Method executes?|Only on cache miss|Always|
|Purpose|Read — avoid repeated fetches|Write — update cache after changes|
|Use for|GET operations|UPDATE operations|

```
@Cacheable: "Is it in cache? Yes → skip method. No → run method."
@CachePut:  "Always run method, then update cache with new result."
```

---

#### @CacheEvict — Remove from Cache

Removes entries from the cache. Use when data is deleted or when you want to force a fresh fetch next time.

```java
@CacheEvict(cacheNames = "employees", key = "#id")
public void deleteEmployee(Long id) {
    employeeRepository.deleteById(id);
    // Cache entry for this id is removed
}
```

**Remove all entries from a cache:**

```java
@CacheEvict(cacheNames = "employees", allEntries = true)
public void refreshAllEmployees() {
    // Clears entire "employees" cache
}
```

### Which Layer to Cache?

You can put caching annotations on any layer — controller, service, or repository. There's no single right answer:

|Layer|Pros|Cons|
|---|---|---|
|**Service**|Most common — caches business logic results|—|
|**Repository**|Caches raw DB queries|May cache data before business logic transforms it|
|**Controller**|Caches the final response|Tightly couples cache to HTTP layer|

> **Service layer is the most common** — it sits between the controller and repository, so you cache the processed result that's ready to return.

### Annotations Summary

| Annotation       | Level          | Purpose                                                  |
| ---------------- | -------------- | -------------------------------------------------------- |
| `@EnableCaching` | Class (config) | Enable caching in the application                        |
| `@Cacheable`     | Method         | Return from cache if exists, otherwise execute and cache |
| `@CachePut`      | Method         | Always execute and update cache                          |
| `@CacheEvict`    | Method         | Remove entry or all entries from cache                   |
