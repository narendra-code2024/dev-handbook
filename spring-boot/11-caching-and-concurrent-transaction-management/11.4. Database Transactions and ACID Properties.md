### What is a Transaction?

A **logical unit of work** that groups multiple database operations into one. Either **all operations succeed** (commit) or **none of them do** (rollback).

```
Transfer $100 from Alice to Bob:
  Step 1: Debit Alice  -$100
  Step 2: Credit Bob   +$100

Both must succeed or neither should happen.
If Step 2 fails after Step 1 → rollback → Alice gets her $100 back.
```

---

### Commit and Rollback

**Commit** — all steps succeeded, make the changes permanent.

**Rollback** — something failed, undo everything back to the initial state as if the transaction never happened.

```
Transaction starts
  │
  ├── All steps succeed → COMMIT (changes saved permanently)
  │
  └── Any step fails → ROLLBACK (all changes undone)
```

### Seeing It Live — Terminal Example

Using a simple `accounts` table:

```sql
CREATE TABLE accounts (
    id SERIAL PRIMARY KEY,
    name VARCHAR(50),
    amount DECIMAL(10, 2)
);

INSERT INTO accounts (name, amount) VALUES ('Alice', 1000.00);
INSERT INTO accounts (name, amount) VALUES ('Bob', 500.00);
```

#### Successful Transaction

```sql
BEGIN;

UPDATE accounts SET amount = amount - 100 WHERE name = 'Alice';
UPDATE accounts SET amount = amount + 100 WHERE name = 'Bob';

COMMIT;
-- Alice: 900, Bob: 600 — both changes saved
```

#### Failed Transaction — Rollback

```sql
BEGIN;

UPDATE accounts SET amount = amount - 100 WHERE name = 'Alice';

-- Oops, wrong query (typo in table name)
SELECT * FROM accountsssss;
-- ERROR: relation "accountsssss" does not exist
-- current transaction is aborted...

-- Even if you try to commit, it won't work
COMMIT;
-- Transaction is rolled back automatically
-- Alice still has her original amount — nothing changed
```

> Once any statement fails inside a transaction, the entire transaction is marked as aborted. Even `COMMIT` triggers a rollback at that point.

#### Manual Rollback

```sql
BEGIN;

UPDATE accounts SET amount = amount - 100 WHERE name = 'Alice';

-- Changed my mind
ROLLBACK;
-- Alice still has her original amount
```

### ACID Properties

|Property|What It Means|Example|
|---|---|---|
|**Atomicity**|All or nothing — either every operation succeeds or none do|Bank transfer: debit + credit both happen or neither|
|**Consistency**|Database moves from one valid state to another, all rules/constraints maintained|E-commerce: can't place order if stock is 0, even mid-transaction|
|**Isolation**|Concurrent transactions don't interfere with each other|Two people can't book the same airline seat at the same time|
|**Durability**|Once committed, changes are permanent even if system crashes|Confirmed payment is stored permanently, survives server restart|

---

### Why ACID Matters

```
Without ACID:
  Alice: -$100 (debited)
  Server crashes before crediting Bob
  Alice lost $100, Bob got nothing
  $100 disappeared

With ACID:
  Alice: -$100 (debited)
  Server crashes before crediting Bob
  Transaction was not committed → ROLLBACK
  Alice still has her $100
```

---

### Advantages

- **Data integrity** — data stays consistent and free from corruption
- **Reliability** — transactions execute predictably
- **Concurrency control** — multiple users can access data without conflicts
- **Fault tolerance** — data survives system failures

> Critical for banking, reservations, e-commerce — anywhere money or limited resources are involved.

---

### Disadvantages

- **Performance overhead** — extra processing for guarantees
- **Complexity** — harder to implement and maintain
- **Scalability challenges** — strict ACID is difficult in distributed systems
- **Potential for deadlocks** — locking can cause transactions to wait on each other
- **Limited concurrency** — strict isolation can slow down throughput

---

### When ACID Isn't Necessary

Not every system needs strict ACID. Some systems use **eventual consistency** — data will become consistent eventually, but not immediately.

|Needs ACID|Can Use Eventual Consistency|
|---|---|
|Banking / payments|Caching systems|
|Reservation systems|Analytics / reporting|
|E-commerce orders|Social media feeds|
|Inventory management|Large-scale blogging|

> MongoDB uses eventual consistency by default — writes may not be immediately visible to all readers, but they will be eventually. This is a trade-off for better performance and scalability.
