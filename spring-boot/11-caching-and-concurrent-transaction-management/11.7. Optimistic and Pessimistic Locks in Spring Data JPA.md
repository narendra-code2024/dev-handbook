### Why Locks When We Have Isolation Levels?

**Isolation levels** define what a transaction **can see** — they prevent dirty reads, non-repeatable reads, phantom reads. Set at the **database level**, the database enforces them automatically.

**Locks** control **access to specific rows** — they prevent concurrent modifications to the same data. Set at the **application/query level** by you.

```
Isolation levels → "What data can other transactions see?"
  → Prevents reading inconsistent data
  → Database handles it automatically

Locks → "Who can modify this row right now?"
  → Prevents concurrent writes to the same row
  → You choose where to apply them
```

**Same scenario, three approaches — incrementing balance from 1000:**

```
Serializable isolation (no locks):
  T1 reads 1000
  T2 reads 1000
  T2 writes 1010 → commits
  T1 writes 1010 → CONFLICT → exception → T1 aborted
  (correct data, but T1 failed — needs retry at application level)

Pessimistic lock:
  T1 locks row, reads 1000
  T2 tries to read → WAITS
  T1 writes 1010 → commits → lock released
  T2 reads 1010 → writes 1020 → commits
  (both succeed, no exceptions, no retries)

Optimistic lock:
  T1 reads 1000, version=1
  T2 reads 1000, version=1
  T2 writes 1010, version=2 → commits
  T1 writes 1010, version check fails → exception → retry
  T1 reads 1010, version=2 → writes 1020, version=3 → commits
  (correct data, T1 retried once)
```

All three end up with correct data, but handle conflicts differently:

|Approach|How It Handles Conflict|
|---|---|
|**Serializable isolation**|Detects conflict → throws exception → you deal with it|
|**Pessimistic lock**|Prevents conflict → other transaction waits|
|**Optimistic lock**|Detects conflict → throws exception → retry with fresh data|

### The Problem

100 concurrent requests try to increment the same balance:

```bash
seq 100 | xargs -P 10 -I {} curl -X PUT \
  'http://localhost:8080/employees/incrementBalance/1'
```

```
Without locks:
  Start: balance = 1000, 100 increments of ₹10
  Expected: 2000
  Actual: 1050 (lost updates)
  Exception: "Row was updated or deleted by another transaction"
             (StaleObjectStateException)
```

Two approaches to solve this: **Optimistic** and **Pessimistic** locking.

---

### Optimistic Locking

Assumes conflicts are **rare**. Doesn't lock anything — instead, uses a **version number** to detect if someone else changed the data.

#### How It Works

```
1. Add a "version" column to the table
2. Transaction reads the row (version = 1)
3. Transaction updates the row, increments version (version = 2)
4. Database checks: is current version still 1?
   ├── Yes → update succeeds, version becomes 2
   └── No (someone else already changed it) → update fails, retry
```

```
Transaction A                         Transaction B
     │                                     │
     Read(balance=1000, version=1)         │
     │                                     Read(balance=1000, version=1)
     │                                     │
     balance = 1010                        balance = 1010
     UPDATE WHERE version=1                │
     → Success (version now 2)             │
     │                                     UPDATE WHERE version=1
     │                                     → FAIL! version is now 2, not 1
     │                                     → OptimisticLockException
     │                                     → Retry with fresh data
```

#### Implementation

**Entity — add @Version:**

```java
@Entity
public class Employee {

    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    private String name;
    private BigDecimal balance;

    @Version
    private Long version;  // Hibernate manages this automatically
}
```

> That's it. Just `@Version` on a field. Hibernate automatically checks and increments the version on every update. No repository or service changes needed.

**What Hibernate generates:**

```sql
UPDATE employees
SET balance = 1010, version = 2
WHERE id = 1 AND version = 1;
--                ^^^^^^^^^^^^ checks version hasn't changed
```

If another transaction already changed the version, zero rows are updated → Hibernate throws `OptimisticLockException`.

**Handling the retry in service:**

```java
@Service
public class EmployeeService {

    @Transactional
    public void incrementBalance(Long id, BigDecimal amount) {
        try {
            Employee employee = employeeRepository.findById(id)
                .orElseThrow();
            employee.setBalance(employee.getBalance().add(amount));
            employeeRepository.save(employee);
        } catch (OptimisticLockException e) {
            // Retry — read fresh data and try again
            incrementBalance(id, amount);
        }
    }
}
```

### Pessimistic Locking

Assumes conflicts are **likely**. Explicitly **locks the row** in the database — other transactions wait until the lock is released.

#### How It Works

```
Transaction A                         Transaction B
     │                                     │
     SELECT ... FOR UPDATE (locks row)     │
     Read(balance=1000)                    │
     │                                     SELECT ... FOR UPDATE
     │                                     → WAITING (row is locked by A)
     │                                     │
     balance = 1010                        │ (still waiting...)
     COMMIT (lock released)                │
     │                                     → Lock acquired
     │                                     Read(balance=1010) ← fresh value
     │                                     balance = 1020
     │                                     COMMIT
```

> No conflicts, no retries — transactions queue up and wait. But slower because of waiting.

#### Implementation

**Repository — add @Lock:**

```java
public interface EmployeeRepository extends JpaRepository<Employee, Long> {

    @Lock(LockModeType.PESSIMISTIC_WRITE)
    @Query("SELECT e FROM Employee e WHERE e.id = :id")
    Optional<Employee> findByIdWithLock(@Param("id") Long id);
}
```

**What Hibernate generates:**

```sql
SELECT * FROM employees WHERE id = 1 FOR UPDATE;
--                                    ^^^^^^^^^^^ locks the row
```

**Service:**

```java
@Service
public class EmployeeService {

    @Transactional
    public void incrementBalance(Long id, BigDecimal amount) {
        Employee employee = employeeRepository.findByIdWithLock(id)
            .orElseThrow();  // row is locked until this transaction ends
        employee.setBalance(employee.getBalance().add(amount));
        employeeRepository.save(employee);
    }
}
```

#### Lock Modes

|Mode|SQL|What It Does|
|---|---|---|
|`PESSIMISTIC_READ`|`FOR SHARE`|Shared lock — others can read but not write|
|`PESSIMISTIC_WRITE`|`FOR UPDATE`|Exclusive lock — others can't read or write|

> `PESSIMISTIC_WRITE` is more commonly used — if you're locking, you probably want exclusive access.

---

### Optimistic vs Pessimistic

||Optimistic|Pessimistic|
|---|---|---|
|Assumption|Conflicts are rare|Conflicts are likely|
|How it works|Version check on update|Lock the row on read|
|DB lock|No|Yes (`FOR UPDATE`)|
|If conflict|Fails, needs retry|Waits for lock release|
|Performance|Better (no locking overhead)|Slower (transactions wait)|
|Deadlock risk|No|Yes|
|Best for|Read-heavy, low contention|Write-heavy, high contention|
|Example|Google Docs (collaborative editing)|Airline seat booking|

### Transaction Strategies in the Real World

|Strategy|Use Case|Example|
|---|---|---|
|**Read Committed**|E-commerce, online transactions — consistent data with good performance|Amazon|
|**Read Uncommitted**|Analytics, non-critical reporting|Google Analytics|
|**Serializable**|Financial transactions, critical operations|Banking (SBI, PNB), stock trading|
|**Optimistic Locking**|Collaborative editing — conflicts rare but need detection|Google Docs|
|**Pessimistic Locking**|Resource booking — need exclusive access|Booking.com, airline reservations|
|**No Transactions**|Large-scale distributed systems — eventual consistency acceptable|Facebook, Twitter|
