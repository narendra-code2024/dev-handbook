### The Problem — Services Don't Know Each Other

Microservices need to communicate, but they don't know each other's IP and port. With a few services, you could hardcode addresses. But as services scale:

```
3 services → each needs to know 2 others (manageable)

12 services → each needs to know 11 others
  → Add 1 new service → update all 12 existing services
  → IP changes → update everywhere
```

---

### The Solution — Service Registry

A **centralized location** where all services register their address (IP + port). When a service wants to talk to another, it asks the registry instead of hardcoding addresses.

```
Without registry:

  Service A ↔ Service B     Every service must know
  Service A ↔ Service C     every other service's address.
  Service B ↔ Service C     Add one service → update all others.


With registry:

  Service A ──register──→ ┌───────────────────┐
                          │                   │
  Service B ──register──→ │  Service Registry │
                          │  (knows everyone) │
  Service C ──register──→ └───────────────────┘

  Service A wants to call B?
    → asks registry: "Where is Service B?"
    → registry returns: 10.4.5.6:7777
    → Service A calls B directly at that address
```

---

### What is Eureka?

A REST-based **service registry** developed by Netflix, integrated with Spring Cloud.

Your microservices system has two types of apps:

**Eureka Server (Service Registry)** — a standalone Spring Boot app that acts as the central registry. You create one `discovery-service` app whose only job is to keep track of all other services.

**Eureka Client (every other service)** — your actual microservices (order-service, inventory-service, etc.). They register themselves with the Eureka Server on startup so other services can find them.

```
┌───────────────────┐                  ┌─────────────────────────┐
│ Order Service     │                  │ Discovery Service       │
│ (Eureka Client)   │──register──→     │ (Eureka Server)         │
│ 10.1.2.3:8081     │                  │ port: 8761              │
└───────────────────┘                  │                         │
                                       │ Knows:                  │
┌───────────────────┐                  │  order-service →        │
│ Inventory Service │──register──→     │    10.1.2.3:8081        │
│ (Eureka Client)   │                  │  inventory-service →    │
│ 10.4.5.6:8082     │                  │    10.4.5.6:8082        │
└───────────────────┘                  └─────────────────────────┘
```

---

### How Eureka Works

```
Order Service starts up
       │
       ▼
Registers with Eureka Server:
  "I'm order-service, my address is 10.1.2.3:8081"
       │
       ▼
Sends heartbeats to Eureka Server every 30s:
  "I'm still alive"
       │
       ▼
Inventory Service wants to call Order Service:
  → asks Eureka Server: "Where is order-service?"
  → Eureka returns: 10.1.2.3:8081
  → Inventory Service calls that address directly
       │
       ▼
Eureka Server runs health checks periodically
       │
       ▼
If Order Service stops sending heartbeats for 90s:
  → Eureka removes it from registry
  → Other services can no longer discover it
```

### Setting Up Eureka Server

#### Step 1: Create Discovery Service

From [start.spring.io](https://start.spring.io/):

- Group: `com.example.ecommerce`
- Artifact: `discovery-service`
- Dependency: **Eureka Server** (`spring-cloud-starter-netflix-eureka-server`)

#### Step 2: Enable Eureka Server

```java
@SpringBootApplication
@EnableEurekaServer
public class DiscoveryServiceApplication {
    public static void main(String[] args) {
        SpringApplication.run(DiscoveryServiceApplication.class, args);
    }
}
```

#### Step 3: Configure

```properties
spring.application.name=eureka-server
server.port=8761

# Don't register itself as a client
eureka.client.register-with-eureka=false
eureka.client.fetch-registry=false
```

|Property|Why|
|---|---|
|`server.port=8761`|Default Eureka port (standard across Spring Boot projects)|
|`register-with-eureka=false`|The server itself shouldn't register as a client|
|`fetch-registry=false`|The server doesn't need to fetch the registry from itself|

> Visit `http://localhost:8761` to see the Eureka dashboard — shows all registered services.

### Setting Up Eureka Clients

Add to **every microservice** (order-service, inventory-service, etc.):

#### Step 1: Add Dependencies

```xml
<dependency>
    <groupId>org.springframework.cloud</groupId>
    <artifactId>spring-cloud-starter-netflix-eureka-client</artifactId>
</dependency>
```

Also add Spring Cloud dependency management (if not already present):

```xml
<properties>
    <spring-cloud.version>2023.0.3</spring-cloud.version>
</properties>

<dependencyManagement>
    <dependencies>
        <dependency>
            <groupId>org.springframework.cloud</groupId>
            <artifactId>spring-cloud-dependencies</artifactId>
            <version>${spring-cloud.version}</version>
            <type>pom</type>
            <scope>import</scope>
        </dependency>
    </dependencies>
</dependencyManagement>
```

#### Step 2: Configure

```properties
# order-service
spring.application.name=order-service
eureka.client.service-url.defaultZone=http://localhost:8761/eureka
```

> `spring.application.name` is critical — this is the name registered in Eureka. Other services use this name to discover you. The `defaultZone` points to where the Eureka Server lives. `http://localhost:8761/eureka` is also the default value, so even without setting it explicitly, it works.

### Service Discovery — Calling Another Service

Once services are registered, they can discover each other by **name** instead of hardcoded IP:port.

**Order Service — the endpoint being called:**

```java
@RestController
@RequestMapping("/api/v1/orders")
public class OrderController {

    @GetMapping("/helloOrders")
    public String helloOrders() {
        return "Hello from order service";
    }
}
```

**Inventory Service — calling Order Service via discovery:**

```java
@RestController
@RequestMapping("/api/v1/products")
@RequiredArgsConstructor
public class ProductController {

    private final DiscoveryClient discoveryClient;
    private final RestClient restClient;

    @GetMapping("/fetchOrders")
    public String fetchFromOrderService() {
        // 1. Ask Eureka: "Where is order-service?"
        ServiceInstance orderService = discoveryClient
            .getInstances("order-service")
            .getFirst();

        // 2. Get the URI (IP:port)
        URI uri = orderService.getUri();

        // 3. Make HTTP call using the discovered address
        return restClient.get()
            .uri(uri + "/api/v1/orders/helloOrders")
            .retrieve()
            .body(String.class);
    }
}
```

```
Inventory Service: discoveryClient.getInstances("order-service")
       │
       ▼
Eureka returns: order-service is at http://10.1.2.3:8081
       │
       ▼
Inventory Service calls: http://10.1.2.3:8081/api/v1/orders/helloOrders
       │
       ▼
Gets response: "Hello from order service"
```

> No hardcoded URLs — if order-service moves to a different port or server, Eureka handles it automatically.

### Eureka Client Configuration Options

```properties
# Where Eureka Server lives
eureka.client.service-url.defaultZone=http://localhost:8761/eureka

# Register this service (default: true)
eureka.client.register-with-eureka=true

# Fetch list of other services (default: true)
eureka.client.fetch-registry=true

# Register IP instead of hostname
eureka.instance.prefer-ip-address=true

# Heartbeat interval — how often client tells server "I'm alive" (default: 30s)
eureka.instance.lease-renewal-interval-in-seconds=30

# Eviction timeout — server removes client if no heartbeat for this long (default: 90s)
eureka.instance.lease-expiration-duration-in-seconds=90
```

> Default settings are suited for most production apps. Only change them if you have specific requirements.
