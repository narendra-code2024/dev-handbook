### Why API Gateway?

Without a gateway, clients must know the IP and port of every microservice — and those change as services scale or restart.

```
Without API Gateway:

  Client ──→ Order Service (10.1.2.3:8081)
  Client ──→ Inventory Service (10.4.5.6:8082)
  Client ──→ Payment Service (10.7.8.9:8083)
  (client must know every address)


With API Gateway:

  Client ──→ API Gateway (single address, port 8080)
                  │
                  ├──→ Order Service
                  ├──→ Inventory Service
                  └──→ Payment Service
  (client knows only one address)
```

The API Gateway is the **single entry point** for all clients (web, mobile, etc.). It routes requests to the correct microservice behind the scenes.

---

### What Else Can API Gateway Do?

|Capability|Example|
|---|---|
|**Routing**|`/api/v1/orders/**` goes to Order Service|
|**Load balancing**|5 instances of Order Service → distribute traffic evenly|
|**Security**|Verify JWT token before forwarding request|
|**Filters**|Add headers, strip URL prefixes, redirect requests|
|**Observability**|Logging, tracing, metrics aggregation|

### Spring Cloud Gateway — Three Building Blocks

```
Request arrives at API Gateway
       │
       ▼
┌─────────────────────────────┐
│  Route (where to send)      │
│    │                        │
│    ├── Predicate (if match) │
│    │                        │
│    └── Filters (modify)     │
└─────────────────────────────┘
       │
       ▼
Downstream Microservice
```

|Block|What It Does|Example|
|---|---|---|
|**Route**|The destination — which service to send the request to|Order Service, Inventory Service|
|**Predicate**|The condition — only route if this matches|Path matches `/orders/**`, method is GET|
|**Filter**|Modify request/response before or after routing|Add header, strip prefix, redirect|

### Setup

#### Step 1: Create API Gateway Service

From [start.spring.io](https://start.spring.io/):

- Artifact: `api-gateway`
- Dependencies: **Gateway** (`spring-cloud-starter-gateway`), **Eureka Discovery Client**

> Remove `-mvc` from the dependency if Spring Initializr adds `spring-cloud-starter-gateway-mvc`. API Gateway uses reactive (Netty), not MVC.

> API Gateway is also a **Eureka Client** — it uses Eureka to discover where each service lives. No hardcoded service addresses.

#### Step 2: Configure Routes

```properties
spring.application.name=api-gateway
server.port=8080

# Eureka
eureka.client.service-url.defaultZone=http://localhost:8761/eureka

# Order Service route
spring.cloud.gateway.routes[0].id=order-service
spring.cloud.gateway.routes[0].uri=lb://ORDER-SERVICE
spring.cloud.gateway.routes[0].predicates[0]=Path=/api/v1/orders/**
spring.cloud.gateway.routes[0].filters[0]=StripPrefix=2

# Inventory Service route
spring.cloud.gateway.routes[1].id=inventory-service
spring.cloud.gateway.routes[1].uri=lb://INVENTORY-SERVICE
spring.cloud.gateway.routes[1].predicates[0]=Path=/api/v1/inventory/**
spring.cloud.gateway.routes[1].filters[0]=StripPrefix=2
```

Or in YAML (cleaner for gateway config):

```yaml
spring:
  application:
    name: api-gateway
  cloud:
    gateway:
      routes:
        - id: order-service
          uri: lb://ORDER-SERVICE
          predicates:
            - Path=/api/v1/orders/**
          filters:
            - StripPrefix=2

        - id: inventory-service
          uri: lb://INVENTORY-SERVICE
          predicates:
            - Path=/api/v1/inventory/**
          filters:
            - StripPrefix=2

server:
  port: 8080

eureka:
  client:
    service-url:
      defaultZone: http://localhost:8761/eureka
```

### Understanding the Config

#### Route

```yaml
- id: order-service              # unique name for this route
  uri: lb://ORDER-SERVICE        # destination service
```

`lb://ORDER-SERVICE` means:

- `lb://` — use **load balancer** (if multiple instances, distribute traffic)
- `ORDER-SERVICE` — the `spring.application.name` registered in Eureka

> Gateway asks Eureka: "Where is ORDER-SERVICE?" → gets IP:port → routes there.

#### Predicate

```yaml
predicates:
  - Path=/api/v1/orders/**       # route only if URL matches this pattern
```

Common predicates:

|Predicate|What It Matches|
|---|---|
|`Path=/api/v1/orders/**`|URL path matches the pattern|
|`Method=GET`|Only GET requests|
|`Method=GET,POST`|Only GET or POST requests|
|`Header=User-Agent, Mozilla/*`|Request has this header value|

#### Filter

```yaml
filters:
  - StripPrefix=2                # remove first 2 path segments
```

What `StripPrefix=2` does:

```
Client sends:    /api/v1/orders/core/helloOrders
StripPrefix=2:   removes /api/v1
Service receives: /orders/core/helloOrders
```

> The `/api/v1` prefix is for the gateway's routing. The actual service doesn't know about it — StripPrefix removes it before forwarding.

---

### Common Built-in Filters

|Filter|What It Does|Example|
|---|---|---|
|`StripPrefix=N`|Remove first N path segments|`StripPrefix=2` removes `/api/v1`|
|`AddRequestHeader=Key, Value`|Add a header to the request sent to service|`AddRequestHeader=X-Custom-Header, Anuj`|
|`AddResponseHeader=Key, Value`|Add a header to the response sent to client|`AddResponseHeader=X-Response-Id, abcd`|
|`RedirectTo=status, URL`|Redirect the request|`RedirectTo=302, https://example.com`|
|`RemoveRequestHeader=Name`|Remove a header from the request|`RemoveRequestHeader=Cookie`|

#### AddRequestHeader Example

```yaml
- id: inventory-service
  uri: lb://INVENTORY-SERVICE
  predicates:
    - Path=/api/v1/inventory/**
  filters:
    - StripPrefix=2
    - AddRequestHeader=X-Custom-Header, Anuj
```

The downstream service can read this header:

```java
@GetMapping("/fetchOrders")
public String fetchOrders(HttpServletRequest request) {
    String customHeader = request.getHeader("X-Custom-Header");
    log.info("Header value: {}", customHeader); // "Anuj"
    // ...
}
```

> Useful for passing user ID, token, or other info from gateway to services after authentication.

#### RedirectTo Example

```yaml
- id: no-service
  uri: lb://NO-SERVICE
  predicates:
    - Path=/api/v1/no-service/**
  filters:
    - AddRequestHeader=X-Custom-Header, Anuj
    - RedirectTo=302, https://codingshuttle.com
```

> Any request to `/api/v1/no-service/**` gets redirected to codingshuttle.com.

### How Request Flows Through Gateway

```
Client: GET http://localhost:8080/api/v1/orders/core/helloOrders
       │
       ▼
API Gateway (port 8080)
       │
       ▼
Check predicates: Path=/api/v1/orders/** → matches order-service route
       │
       ▼
Apply filters: StripPrefix=2 → /api/v1 removed → /orders/core/helloOrders
       │
       ▼
Resolve URI: lb://ORDER-SERVICE → ask Eureka → 10.1.2.3:8081
       │
       ▼
Forward to: http://10.1.2.3:8081/orders/core/helloOrders
       │
       ▼
Order Service responds → Gateway returns response to client
```

### Startup Order

Start services in this order:

```
1. Discovery Service (Eureka Server) — must be up first
2. Order Service, Inventory Service (Eureka Clients) — register themselves
3. API Gateway — discovers services through Eureka
```

> If gateway starts before services register, you may get "No server available" errors. Wait ~30 seconds for Eureka to sync, then it works.

### Note on `server.servlet.context-path`

Some projects use `context-path` to add a prefix to all endpoints in a service:

```properties
# order-service
server.servlet.context-path=/orders
```

This means every endpoint in that service is automatically prefixed with `/orders` — so `@RequestMapping("/core")` becomes `/orders/core/...`. This is **not mandatory** — you can achieve the same by putting `/orders` directly in your `@RequestMapping`. The API Gateway routes based on the predicate pattern, not the context-path. Either approach works:

```
With context-path:       context-path=/orders + @RequestMapping("/core")
Without context-path:    @RequestMapping("/orders/core")
→ Both produce: /orders/core/...
```
