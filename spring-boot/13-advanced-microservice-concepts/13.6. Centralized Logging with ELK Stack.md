
---

### Why Centralized Logging?

Zipkin handles distributed tracing (timing + request paths) but doesn't offer deep log analysis. In a microservice system, when something goes wrong you'd have to check logs of each service individually — tedious with 10+ services.

ELK Stack solves this by providing:

- Centralized view of all microservice logs at one location
- Deep log analysis and powerful search (e.g., search by Trace ID across all services)
- Long-term log storage
- Real-time monitoring, alerting, and dashboards

---

### What is the ELK Stack?

Three open-source tools from Elastic, used together:

```
Spring Boot Apps → logs → Logstash → Elasticsearch ← Kibana (visualize)
```

#### Elasticsearch

A **NoSQL database** (based on Lucene) that stores and indexes log data. Built in Java so it runs on any platform. Excels at indexing streams of data like logs and performing fast aggregation queries.

> In Elasticsearch, **indexes** are equivalent to tables in SQL.

#### Logstash

A data pipeline tool that **collects, parses, transforms, and ships** log data from microservices to Elasticsearch. Configured via a `.conf` file that defines input (log files) and output (Elasticsearch).

#### Kibana

An **analytics and visualization layer** on top of Elasticsearch. Used to search, view, filter, and build dashboards from the indexed log data — no need to write raw queries.

---

### How It All Works Together

```
Spring Boot Microservices
  │  generate log files (via logback)
  ▼
Logstash (reads log files via logstash.conf)
  │  parses each line as a structured event
  │  ships events in near real-time
  ▼
Elasticsearch (indexes the log events)
  │  logs are now searchable by Trace ID, service name, timestamp, etc.
  ▼
Kibana Dashboard
  │  visualize, search, filter, alert
```

### Setup

#### Step 1 — Download & Extract ELK Stack

Download the ZIP for your OS and **extract/unzip** each one:

|Tool|Download|Guide|
|---|---|---|
|Elasticsearch|https://www.elastic.co/downloads/elasticsearch|[Docs](https://www.elastic.co/docs/deploy-manage/deploy/self-managed/installing-elasticsearch)|
|Logstash|https://www.elastic.co/downloads/logstash|[Docs](https://www.elastic.co/docs/reference/logstash/installing-logstash)|
|Kibana|https://www.elastic.co/downloads/kibana|[Docs](https://www.elastic.co/docs/deploy-manage/deploy/self-managed/install-kibana)|

> Windows + Mac combined guide: https://logz.io/blog/elk-stack-windows/

After extracting, you'll have three folders. Each has the same structure:

```
elasticsearch-<version>/
  ├── bin/              ← executables (start server, reset password, etc.)
  ├── config/           ← configuration files
  │     └── elasticsearch.yml   ← main config (SSL, ports, cluster name, etc.)
  ├── logs/
  └── plugins/

logstash-<version>/
  ├── bin/
  ├── config/
  └── ...               ← place your logstash.conf here

kibana-<version>/
  ├── bin/
  ├── config/
  │     └── kibana.yml  ← base URL, port, SSL settings
  └── ...
```

> **Mac OS only:** Run this command for each extracted folder before starting, to bypass Gatekeeper warnings:
> 
> ```bash
> xattr -d -r com.apple.quarantine <folder-name>
> ```

---

#### Step 2 — Run Elasticsearch

```bash
cd elasticsearch-<version>
bin/elasticsearch
```

Runs on `https://localhost:9200` — note **https**, because `config/elasticsearch.yml` has SSL enabled by default (`xpack.security.enabled: true`).

On first run, Elasticsearch generates:

- **Username:** `elastic`
- **Password:** _(shown once in terminal — copy immediately)_
- **Enrollment token:** _(needed for Kibana setup)_
- **CA certificate fingerprint**

> To reset password later: `bin/elasticsearch-reset-password -u elastic`

Verify it's running: `https://localhost:9200` → login with `elastic` / `<password>`

---

#### Step 3 — Run Kibana

```bash
cd kibana-<version>
bin/kibana
```

On first run, Kibana provides a setup URL in the terminal. Open it, paste the **enrollment token** from Elasticsearch, then log in with `elastic` / `<password>`.

Kibana dashboard: `http://localhost:5601`

#### Step 4 — Configure Logback in Each Microservice

Create `logback-spring.xml` in `src/main/resources` of each microservice:

```xml
<configuration>
  <!-- Console appender -->
  <appender name="STDOUT" class="ch.qos.logback.core.ConsoleAppender">
    <encoder>
      <pattern>%d{yyyy-MM-dd HH:mm:ss} [%thread] [%X{traceId},%X{spanId}] %-5level %logger{36} - %msg%n</pattern>
    </encoder>
  </appender>

  <!-- Rolling file appender — same pattern as STDOUT -->
  <appender name="FILE" class="ch.qos.logback.core.rolling.RollingFileAppender">
    <file>logs/${spring.application.name}/application.log</file>
    <rollingPolicy class="ch.qos.logback.core.rolling.TimeBasedRollingPolicy">
      <!-- New file per day, max 10MB per file -->
      <fileNamePattern>logs/${spring.application.name}/application-%d{yyyy-MM-dd}.%i.log</fileNamePattern>
      <timeBasedFileNamingAndTriggeringPolicy
        class="ch.qos.logback.core.rolling.SizeAndTimeBasedFNATP">
        <maxFileSize>10MB</maxFileSize>
      </timeBasedFileNamingAndTriggeringPolicy>
    </rollingPolicy>
    <encoder>
      <pattern>${same pattern as STDOUT above}</pattern>
    </encoder>
  </appender>

  <root level="INFO">
    <appender-ref ref="STDOUT"/>
    <appender-ref ref="FILE"/>
  </root>
</configuration>
```

> Log pattern includes **Trace ID and Span ID** from Micrometer — enabling cross-service log search by Trace ID in Kibana.  
> Each microservice gets its own log folder: `logs/order-service/`, `logs/inventory-service/`, etc.

---

#### Step 5 — Configure Logstash

Create `logstash.conf` inside the Logstash installation folder (`logstash-<version>/`):

```ruby
input {
  file {
    type  => "log"
    # Absolute path to your project's root logs folder
    # Replace <your-project-root> with the absolute path to your project
    # e.g. /Users/john/documents/ecommerce
    path  => "<your-project-root>/logs/**/*.log"
    start_position => "beginning"
  }
}

output {
  elasticsearch {
    hosts    => ["https://localhost:9200"]
    index    => "spring-boot-logs-%{+YYYY.MM.dd}"
    ssl_certificate_verification => false
    user     => "elastic"
    password => "<your-elasticsearch-password>"
  }
}
```

> To get the absolute path of your project root: right-click the project root folder in IntelliJ → **Copy Path → Absolute Path**.

---

#### Step 6 — Run Logstash

```bash
cd logstash-<version>
bin/logstash -f logstash.conf
```

Logstash reads all log files, converts each line into a structured event, and ships them to Elasticsearch.

---

#### Step 7 — Create Data View in Kibana

1. Go to **Analytics → Discover → Create Data View**
2. Set index pattern: `spring-boot-logs-*`
3. Set timestamp field: `@timestamp`
4. Save → logs are now visible and searchable in Kibana

---

### Searching Logs in Kibana

The real power — search all microservice logs by **Trace ID**:

```
Search: <traceId>   →  shows logs from ALL services for that one request
                        sorted by timestamp
```

Example output:

```
[order-service]     Creating order method called    traceId=abc123
[inventory-service] Reducing stock for product      traceId=abc123
```

> One Trace ID spans across all microservices — so one search gives the full picture of a request across the entire system.

### Tracing vs Logging — Summary

||Zipkin (13.5)|ELK Stack (13.6)|
|---|---|---|
|Purpose|Request flow & timing|Centralized log storage & search|
|Shows|Which services called & how long|What happened inside each service|
|Search by|Trace ID (visual timeline)|Trace ID, service name, log level, time|
|Best for|Finding bottlenecks|Debugging errors & exceptions|
