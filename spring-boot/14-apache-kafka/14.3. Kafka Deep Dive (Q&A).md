### Leader Partitions, Brokers & Replication

**Does Kafka maintain the same data across brokers?**

Yes — through replication. When you create a topic you set a `replication.factor` which determines how many brokers hold a copy of each partition.

**Does every broker have all partitions?**

No — each broker only holds the partitions assigned to it, either as leader or replica. Only when `replication.factor = total number of brokers` does every broker hold all partitions.

```
replication.factor=2, 3 partitions, 3 brokers:
Broker 1 → P1 (Leader), P3 (Replica)     ← no P2 at all
Broker 2 → P2 (Leader), P1 (Replica)     ← no P3 at all
Broker 3 → P3 (Leader), P2 (Replica)     ← no P1 at all

replication.factor=3, 3 partitions, 3 brokers:
Broker 1 → P1 (Leader), P2 (Replica), P3 (Replica)  ← has all 3
Broker 2 → P2 (Leader), P1 (Replica), P3 (Replica)  ← has all 3
Broker 3 → P3 (Leader), P1 (Replica), P2 (Replica)  ← has all 3
```

Replicas don't receive from the producer directly — they **pull from the leader** continuously.

|Replication Factor|Fault Tolerance|
|---|---|
|1|❌ No fault tolerance|
|2|✅ Can tolerate 1 failure|
|3|✅ Can tolerate 1 failure safely (2 only in ideal conditions where all replicas are in ISR)|

> Kafka can only guarantee safe failover using **ISR (in-sync replicas)**. If replicas fall out of sync, fewer safe failovers are possible.

#### Unclean Leader Election

If no in-sync replica is available when a leader goes down, Kafka can optionally elect an **out-of-sync replica** as the new leader:

- ✅ Avoids downtime
- ❌ May cause **data loss** (out-of-sync replica may be missing recent messages)

Controlled by: `unclean.leader.election.enable` (default: `false`)

**How many leader partitions does one topic have?**

One topic can have **multiple leader partitions**, spread across different brokers — one leader per partition.

- **One topic** → many partitions
- **One partition** → exactly one leader (on one broker)
- **One topic** → as many leaders as partitions, distributed across brokers

```
Topic: "orders" (4 partitions, replication.factor=2)

Broker 1 → P1 (Leader), P2 (Leader), P3 (Replica), P4 (Replica)
Broker 2 → P3 (Leader), P4 (Leader), P1 (Replica), P2 (Replica)
```

This is why Kafka scales well — multiple producers can write to the same topic simultaneously, each hitting a different broker's leader partition.

**Does the same partition number appearing on multiple brokers mean duplicate data?**

No — it means **replication**. The key distinction:

- **Same partition number across brokers** = replication (same data, different roles — leader vs replica)
- **Different partition numbers across brokers** = distribution (different data)

```
Broker 1: P1 Leader  ┐ same partition,
Broker 2: P1 Replica ┘ different role → replication

Broker 1: P1 Leader  ┐ different partitions,
Broker 2: P3 Leader  ┘ different data  → distribution
```

---

No. Leadership is at the **partition level**, not broker level. Every broker is a leader for some partitions and a replica for others.

```
Broker 1 → Partition 1 (Leader)  + Partition 3 (Replica)
Broker 2 → Partition 2 (Leader)  + Partition 1 (Replica)
Broker 3 → Partition 3 (Leader)  + Partition 2 (Replica)
```

Kafka intentionally distributes partition leaders across brokers for load balancing, fault tolerance, and parallel throughput — this also naturally distributes consumer reads across brokers.

> If all leaders were on one broker → all writes go there → **hotspot**. Replicas on other brokers would sit idle, defeating the purpose of multiple brokers.

**Do consumers read from the leader broker or any broker?**

There is no such thing as a "leader broker" — leadership is per partition. By default, consumers read from the **leader partition** (followers are not used for reads in standard Kafka setups).

> Advanced: follower fetch is possible since Kafka 2.4 but not the default behavior.

```
Consumer A reads Partition 1 → goes to Broker 1 (P1 leader is there)
Consumer B reads Partition 2 → goes to Broker 2 (P2 leader is there)
Consumer C reads Partition 3 → goes to Broker 3 (P3 leader is there)
```

This is why Kafka is fast — consumers naturally spread across brokers because partition leaders are distributed.

**How does a consumer know which broker holds the leader partition?**

Kafka handles this automatically — the consumer doesn't need to know upfront:

1. Consumer connects to **any** broker initially — called a **bootstrap broker**
2. That broker returns **cluster metadata** — which broker holds which partition leader
3. Consumer then connects directly to the correct broker
4. If a leader changes (broker goes down, rebalancing), Kafka updates the metadata and consumer retries automatically

The consumer just says "I want to read from this topic/partition" — Kafka handles the routing.

### Offsets & Messages

**Are the numbers shown in partitions (in Softwaremill visualizer) offsets?**

Those numbers are individual **messages**, where each number is the message's **offset value** within that partition.

```
Partition 1 → messages at offsets: 21, 20, 8, 6
Partition 2 → messages at offsets: 18, 10, 7, 4
Partition 3 → messages at offsets: 17, 16, 15, 14...
```

The `A` markers in the visualizer indicate the **currently active/consumed offset** for the consumer in each partition.

**Are offsets unique across all partitions?**

No — offsets are **local to a partition**. The same offset number can exist in multiple partitions and refers to completely different messages.

```
Partition 1 → offset 5 = "Order #100 created"
Partition 2 → offset 5 = "Payment #200 processed"   ← different message, same offset number
```

To uniquely identify a message: `Topic + Partition + Offset`

### Consumer Offset Tracking

**Do consumers need to remember which offset they read up to?**

Yes — consumers track offsets logically, but **committed offsets are stored in Kafka** itself in an internal topic called `__consumer_offsets`. The app controls when to commit, Kafka persists it.

```
__consumer_offsets topic
┌─────────────────┬─────────────┬────────┐
│   Group ID      │  Partition  │ Offset │
├─────────────────┼─────────────┼────────┤
│ consumer-group1 │      1      │   20   │
│ consumer-group1 │      2      │   18   │
│ consumer-group2 │      1      │   15   │  ← different group, independent offset
└─────────────────┴─────────────┴────────┘
```

Each consumer group maintains its own offset per partition — groups don't interfere with each other.

**What gets stored exactly?**

`Group ID + Partition + Offset` — because multiple consumer groups can read the same topic independently, each at their own pace.

### Consumer Groups & Partition Assignment

**Do consumers from the same group always read from different partitions?**

Yes — this is a core Kafka rule. One partition is assigned to **exactly one consumer** within a consumer group at a time.

```
Consumer Group A (4 consumers, 4 partitions):
├── Consumer 1 → is assigned Partition 1
├── Consumer 2 → is assigned Partition 2
├── Consumer 3 → is assigned Partition 3
└── Consumer 4 → is assigned Partition 4
```

Two consumers from the same group can never read the same partition simultaneously — this prevents duplicate processing.

|Consumers|Partitions|Result|
|---|---|---|
|4|4|Perfect — 1 each|
|2|4|Each consumer reads 2 partitions|
|6|4|4 active, 2 sit idle|

> Different consumer groups CAN read the same partition independently — e.g., Inventory Service group and Notification Service group both consume the same `orders` topic without interfering.

**If all partitions are at the same offset, which does the consumer read first?**

Kafka guarantees ordering **only within a partition** — there is no ordering guarantee across partitions. The consumer fetches data using a **poll loop** (`poll()`), which retrieves **batches of messages** from assigned partitions in round-robin fashion. You should never rely on cross-partition ordering.

### Why Multiple Partitions?

**Why have multiple partitions at all?**

1. **Parallelism** — multiple consumers in a group can read simultaneously, one per partition. More partitions = higher throughput.
    
2. **Storage distribution** — partitions are spread across brokers, so no single broker holds all the data.
    
3. **Fault tolerance** — each partition is replicated across brokers. If one broker goes down, another already has the replica ready to become leader.
    

```
Single partition:    Consumer A reads → others wait       ❌ bottleneck
Multiple partitions: Consumer A → P1, Consumer B → P2    ✅ parallel
                     Consumer C → P3, Consumer D → P4
```

**Are partitions split across brokers for the same topic?**

Yes. When you create a topic, Kafka automatically distributes its partitions across available brokers. Partition IDs are **independent per topic** — `orders P3` and `notifications P3` are completely different partitions that just share the same number.

```
Topic: "orders"        → Broker 1: P1, P2  |  Broker 2: P3, P4
Topic: "payments"      → Broker 1: P1, P3  |  Broker 2: P2, P4
Topic: "notifications" → Broker 1: P2, P4  |  Broker 2: P1, P3
```

Replication works independently per topic. With `replication.factor=2`, each partition also has a copy on another broker:

```
Topic: "orders" (replication.factor=2)
  P1 → Leader: Broker 1, Replica: Broker 2
  P2 → Leader: Broker 1, Replica: Broker 2
  P3 → Leader: Broker 2, Replica: Broker 1
  P4 → Leader: Broker 2, Replica: Broker 1
```

> The partition distribution table above shows which broker is the **leader** for each partition — each partition also has replicas on other brokers depending on the replication factor.

**How does Cluster → Broker → Node (machine) fit together?**

```
Kafka Cluster
├── Node 1 (Machine/Server)
│     └── Broker 1 (Kafka process running on it)
│           ├── Partition 1 → Leader
│           └── Partition 2 → Leader
│
├── Node 2 (Machine/Server)
│     └── Broker 2 (Kafka process running on it)
│           ├── Partition 1 → Replica
│           └── Partition 3 → Leader
│
└── Node 3 (Machine/Server)
      └── Broker 3 (Kafka process running on it)
            ├── Partition 2 → Replica
            └── Partition 3 → Replica
```

Node = the machine. Broker = Kafka running on it. Typically 1 node = 1 broker. A cluster is just a group of these brokers coordinated by KRaft (or ZooKeeper in older versions).

### Full Nested Structure

```
Kafka Cluster
└── Brokers (Nodes/Machines)
      └── Topics (distributed across brokers)
            └── Partitions (leader on one broker, replicas on others)
                  └── Messages (each with an offset — local to partition)

Consumer Groups
└── Consumers (one per partition within a group)
      └── Offset tracked per: Group ID + Partition
            └── Stored in: __consumer_offsets (internal Kafka topic)

Producer
└── Sends to: Topic → specific Partition (via key hash or round-robin)
      └── Always writes to: Leader partition
```
