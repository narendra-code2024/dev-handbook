### The Problem

In 14.6, we duplicated the `UserCreatedEvent` class in both services with matching package names. This doesn't scale:

- 10 microservices = copy the class 10 times
- Change one field → update every service manually
- No central tracking of what events look like or their history

---

### What is Schema Registry?

A **centralized server** that stores and manages event schemas. Producers and consumers reference schemas from this registry instead of maintaining their own copies.

- Enforces schema compliance — producers can't send malformed messages
- Saves downstream consumers from handling unexpected data formats
- Critical component for **data governance** in large microservice systems

#### How It Works

<p align="center"> <img src="https://cs-prod-assets-bucket.s3.ap-south-1.amazonaws.com/image_8_33e2dbaa15.png" width="600"> </p>

1. Producer registers schema with registry → gets a **schema ID**
2. Schema ID is **embedded in the message payload** (not the header)
3. Producer serializes message using that schema → publishes to Kafka topic
4. Consumer reads message from topic
5. Consumer reads schema ID from payload → fetches schema from registry (**cached after first use** — registry is not called for every message)
6. Consumer deserializes message using that schema

### Setup Steps

#### Step 0 — Run Confluent Schema Registry via Docker

Use the official Confluent KRaft docker-compose file: [cp-all-in-one-kraft/docker-compose.yml](https://github.com/confluentinc/cp-all-in-one/blob/7.7.1-post/cp-all-in-one-kraft/docker-compose.yml)

Copy the raw file content into `docker-compose.yml` at the root of your project. Keep only what you need:

```
Required:
  - broker          ← Kafka broker
  - schema-registry ← Schema Registry (port 8081)

Optional:
  - control-center  ← Confluent UI (port 9021), similar to Kafbat
  - connect         ← only if using Kafka Connect

Remove:
  - ksqlDB, REST proxy, CLI  ← not needed
```

```bash
# Run services
docker-compose up -d

# Check running containers
docker-compose ps
```

> Stop your local Kafka before running Docker — both use port 9092.  
> Replace the default cluster ID in the docker-compose file with your own generated ID.

> **If Schema Registry is unavailable** — producers cannot register new schemas and may fail to send messages. Consumers may still work if schemas are already cached locally.

#### Step 1 — Add Avro Schema File

Create `src/main/resources/avro/user-created-event.avsc` in **both** user-service and notification-service:

```json
{
  "name": "UserCreatedEvent",
  "namespace": "com.learn.kafka.event",
  "type": "record",
  "fields": [
    { "name": "id",       "type": "long",   "default": 0  },
    { "name": "name",     "type": "string", "default": "" },
    { "name": "email",    "type": "string", "default": "" }
  ]
}
```

> **`namespace`** — determines the Java package where the generated class will be created.  
> **Default values** are required when **adding new fields** to maintain backward compatibility — without them, the new schema may be rejected by the registry.  
> Install the **Avro Schema** plugin in IntelliJ for syntax highlighting.

---

#### Step 2 — Add Dependencies (`pom.xml` in both services)

```xml
<!-- Confluent Kafka Avro Serializer -->
<dependency>
    <groupId>io.confluent</groupId>
    <artifactId>kafka-avro-serializer</artifactId>
    <version>7.7.1</version>
</dependency>

<!-- Confluent Schema Registry Client -->
<dependency>
    <groupId>io.confluent</groupId>
    <artifactId>kafka-schema-registry-client</artifactId>
    <version>7.7.1</version>
</dependency>

<!-- Avro -->
<dependency>
    <groupId>org.apache.avro</groupId>
    <artifactId>avro</artifactId>
    <version>1.11.3</version>
</dependency>

<!-- Confluent repository (not on Maven Central) -->
<repositories>
    <repository>
        <id>confluent</id>
        <url>https://packages.confluent.io/maven/</url>
    </repository>
</repositories>
```

Also add the Avro Maven plugin to auto-generate Java classes from `.avsc` files:

```xml
<plugin>
    <groupId>org.apache.avro</groupId>
    <artifactId>avro-maven-plugin</artifactId>
    <version>1.11.3</version>
    <executions>
        <execution>
            <id>schemas</id>
            <phase>generate-sources</phase>
            <goals><goal>schema</goal></goals>
            <configuration>
                <sourceDirectory>${project.basedir}/src/main/resources/</sourceDirectory>
                <outputDirectory>${project.basedir}/src/main/java/</outputDirectory>
            </configuration>
        </execution>
    </executions>
</plugin>
```

---

#### Step 3 — Build to Generate Java Classes

Run Maven install to trigger the Avro plugin and auto-generate the Java class.

**Option 1 — Terminal:**

```bash
./mvnw install -pl user-service
```

**Option 2 — IntelliJ:**

```
Maven panel (right side) → user-service → Lifecycle → double-click install
```

This generates `UserCreatedEvent.java` in `src/main/java/com/learn/kafka/event/` — with all fields, getters, setters, and Avro-specific methods. Repeat for notification-service.

> **Why do we have `.avsc` in both services?**  
> In this demo, we copy the `.avsc` file to both services manually so the Maven plugin can generate the Java class at build time. This is a **build-time** step — separate from Schema Registry which is a **runtime** concern.
> 
> ```
> Schema Registry   = runtime  → schema validation + versioning
> Maven Avro plugin = build-time → Java class generation from .avsc
> ```
> 
> In production, `.avsc` files are kept in a **shared Git repo** — services pull the latest schema file from there instead of copying manually.

> The generated class replaces the manually created `UserCreatedEvent.java` — delete the old one.

#### Step 4 — Update Serializer & Deserializer

**user-service `application.yml`** — switch to Avro serializer:

```yaml
spring:
  kafka:
    producer:
      key-serializer: org.apache.kafka.common.serialization.LongSerializer
      value-serializer: io.confluent.kafka.serializers.KafkaAvroSerializer
      properties:
        schema.registry.url: http://localhost:8081
```

**notification-service `application.yml`** — switch to Avro deserializer:

```yaml
spring:
  kafka:
    consumer:
      key-deserializer: org.apache.kafka.common.serialization.LongDeserializer
      value-deserializer: io.confluent.kafka.serializers.KafkaAvroDeserializer
      properties:
        schema.registry.url: http://localhost:8081
        specific.avro.reader: true   # deserialize into specific generated class, not GenericRecord
```

> `specific.avro.reader: true` — tells the deserializer to map the incoming Avro bytes to the specific generated `UserCreatedEvent` class.

> **Keys** are usually simple types (`Long`, `String`) but can also use Avro serialization if needed.

> **No more `spring.json.trusted.packages`** — that was needed for JSON deserialization. Avro + Schema Registry handles type safety through the schema ID embedded in each message payload.

### Schema Versioning & Compatibility

The Schema Registry tracks versions automatically. Each time you update the `.avsc` file and rebuild, a new schema version is registered:

```
Version 1: { id, name, email }           ← original
Version 2: { id, fullName, email }       ← after renaming name → fullName
```

- View and compare schema versions in Control Center → Topics → Schema tab
- Query schemas via REST API: `GET http://localhost:8081/subjects`

#### Compatibility Modes

Schema Registry enforces compatibility between versions. Supported modes:

|Mode|Description|
|---|---|
|`BACKWARD` (default)|New consumers can read data written by old producers|
|`FORWARD`|Old consumers can read data written by new producers|
|`FULL`|Both backward and forward compatible|
|`NONE`|No compatibility validation|

> **BACKWARD is the default** — this is why default values are required when adding new fields. Without them, the new schema is rejected as incompatible.

#### Subject Naming

Schema Registry groups schemas under **subjects**. By default the subject name is `<topic-name>-value`:

```
Topic: user-created-topic  →  Subject: user-created-topic-value
```

### JSON vs Avro — When to Use Which

||JSON Serializer (14.6)|Avro + Schema Registry (14.7)|
|---|---|---|
|Setup|Simple|Requires Schema Registry server|
|Schema management|Manual (duplicate classes)|Centralized `.avsc` files|
|Schema validation|❌ None|✅ Registry enforces schema|
|Schema versioning|❌ None|✅ Full version history|
|Performance|Larger payload (verbose JSON)|Smaller payload (binary format)|
|Best for|Small projects, quick setup|Production, multiple teams/services|
